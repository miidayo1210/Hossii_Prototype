## 1. Hossiiとは何か

`[思想・原則]`

**Hossii は、目的や縁によって集う人たちのために、関係・空間・対話・活動・記録・成長のあり方を、ビジュアルに設計できるコミュニケーション基盤である。**

Hossii は単一の SNS でも、単一のチャットツールでも、単一のログツールでもない。人が集まる「場」を、目的に応じて視覚的に立ち上げ、その中で交わされる対話・活動・記録を、関係と時間を保ったまま残し、次の対話や行動につなげるための基盤である。

この定義は、次のいずれの規模・関係性も含められる抽象度を保つ。

- 二人（ペア）
- 仲良しのメンバー
- 部活動
- 学校・授業
- 企業・組織
- イベント
- 地域
- 場所を起点に集う人
- 継続的なコミュニティ
- 一人の振り返り

つまり Hossii は「特定の一つの使い方」に閉じない。関係の単位（二人〜組織〜地域）と、目的（対話・学習・活動・記録・振り返り）の組み合わせを、視覚的な場として設計できることが本質である。

## 2. 解決したい課題

`[思想・原則]`

既存のツールは、次のいずれかに寄っている。

- **チャット型:** 対話は流れるが、蓄積・可視化・振り返りが弱い。時間が経つと文脈が失われる。
- **ログ／記録型:** 蓄積はできるが、対話や関係性、場の空気が乗らない。無味乾燥になりやすい。
- **SNS 型:** 個人の発信は強いが、特定の目的やコミュニティに閉じた「安心して集まれる場」を作りにくい。

いずれも、「個人」と「集団」と「場所」と「時間」の関係を保ったまま、対話も記録も同じ景色の中で扱う、という体験を提供しにくい。

Hossii が解決したいのは次である。

- 目的に応じた「場」を、専門知識なしに視覚的に立ち上げたい。
- そこでの対話・活動・記録を、関係・空間・時間を保ったまま残したい。
- 記録を単なるアーカイブに終わらせず、次の対話や行動につなげたい。
- 個人の営みと集団の営みを、どちらも従属させずに両立させたい。

## 3. Hossiiの中心価値

`[思想・原則]`

Hossii の中心価値は、次の複数の柱からなる。単一の主目的ではない。

1. **目的に応じた場を、視覚的に設計できること。**
   関係の単位と目的に合わせて、構成・機能・世界観を組み合わせた「場」を立ち上げられる。
2. **対話・活動・記録を、関係と時間を保って残せること。**
   誰が・いつ・どの場で・どんな関係の中で残したのかを失わない。
3. **記録が次の対話や行動を生むこと。**
   記録は過去の保存で終わらず、可視化・編集・共有・伝達を通じて、次を促す。
4. **個人と集団の記憶・記録を両立させること。**（→ 10 章）
   これは中心価値の一つであり、特に「なぜビジュアルなのか」（→ 11 章）を説明する思想でもある。

## 4. 対象となる利用シーン

`[思想・原則]`

Hossii が想定する利用シーンは、規模と関係性で幅広い。すべて同一の抽象度で扱える。

- **二人:** ペアでの対話・記録・振り返り（例：コーチと本人、パートナー）。
- **仲良しのメンバー:** 少人数の継続的な集まり。
- **部活動:** 継続的な活動・記録・世代を超えた蓄積。
- **学校・授業:** クラス・授業単位の対話と学習記録。
- **企業・組織:** チーム・部署・全社の対話と活動ログ。
- **イベント:** 期間限定で集まる場（ワークショップ・上映会・説明会など）。
- **地域:** 地域コミュニティ・場所を核にした集まり。
- **場所を起点に集う人:** 特定の場所（店・拠点）に縁のある人たち。
- **継続的コミュニティ:** 目的を持って続くコミュニティ全体。
- **一人の振り返り:** 個人が自分の記録を見返し、次につなげる内省。

## 5. 共通システム構造

`[思想・原則]` `[直近方針]`

Hossii は、規模や目的が違っても、共通のシステム構造の上に成り立つ。

- **アカウント（account）:** 端末を越えて本人を保持する単位。
- **所属（membership）:** 個人がコミュニティ・スペースにどう関わるかの記録。
- **コミュニティ（community）:** 所属・目的・継続性・運営の単位。
- **スペース（space）:** 対話・活動・記録が営まれる視覚的な場。
- **投稿（post / Hossii）:** 場に置かれる対話・活動・記録の最小単位。
- **公開範囲（visibility）:** 誰に見えるか。
- **アーカイブ / アウトプット（archive / output）:** 蓄積と、その再編集・持ち帰り。

これらは下位文書（`core/core-concepts.md`・仕様書 109）で具体的なデータ構造・RLS として定義される。本書ではデータ設計を確定しない。

## 6. アカウントと所属

`[思想・原則]` `[直近方針]`

**アカウント**は、メール／ID によって端末を越えて本人を保持する単位である。ゲスト参加（ログインなしの URL / QR 参加）は public スペースで維持しつつ、本人としての継続的な所有権・所属はアカウントに紐づく。

**所属（membership）**は、個人がコミュニティやスペースにどう関わっているかの記録である。

- 所属は「関わりの記録」であり、それ自体が管理者権限の正本ではない。
- 管理者判定は従来どおりの仕組みに従い、membership の role を権限の正本として扱わない。

Hossii 全体としての個人領域（→ 8 章）と、コミュニティ内の実際のスペースは、別の概念として区別する。

## 7. コミュニティとHossii環境

`[思想・原則]` `[直近方針]`

コミュニティ・スペース・テンプレートは、単純な上下関係ではなく、次の役割として整理する。

- **コミュニティ**
  = 所属・目的・継続性・運営の単位。人が「属する」対象。
- **スペース**
  = 対話・活動・記録が営まれる視覚的な場。人が「入って営む」対象。
- **テンプレート**
  = 目的に応じた構成・機能・世界観の組み合わせ。場を立ち上げるための設計パターン。

### 「スペースを売る」の再配置

`space-product-vision.md` が掲げる「スペースを売る」という思想は否定しない。本書ではそれを次のように再配置する。

> Hossii は、コミュニティや目的に合わせて設計した、**スペースの組み合わせ・体験パッケージ**を提供する。

単体のスペースを売るのではなく、コミュニティの目的に沿って複数のスペースやテンプレートを組み合わせた「体験パッケージ」を届ける、という理解である。

## 8. スペースと人の関係

`[思想・原則]` `[設計案・未実装]`

スペースを一つの enum に押し込めず、次の **独立した分類軸** で整理する。これらは混ぜない。DB 設計はここでは確定しない。

### 8.1 関係・利用主体

`shared` / `personal` / `pair` / `team`

- `shared` — 複数人が共有する場。
- `personal` — 本人中心の場。
- `pair` — 二人の場。
- `team` — チームの場。

### 8.2 用途・テンプレート

`event` / `workshop` / `class` / `coaching` / `community` / `reflection`

- 目的に応じた構成・機能・世界観のパターン。

### 8.3 公開範囲

`public` / `members_only` / `invite_only` / `owner_only`

- `public` — URL を知っていれば参加できる（ゲスト体験）。
- `members_only` — 所属メンバーに限定。
- `invite_only` — 個人単位の招待制。
- `owner_only` — 本人だけに見える。

### 8.4 利用期間

`temporary` / `ongoing`

- `temporary` — 期間限定。
- `ongoing` — 継続的。

> これらの軸は、組み合わせて一つのスペースを性格づけるためのものであり、実装上の単一 enum ではない。DB 設計は下位文書で検討する。

## 9. 投稿・対話・活動ログ

`[思想・原則]` `[実装済み・専用branch検証]`

スペースに置かれる投稿（Hossii）は、対話・活動・記録の最小単位である。投稿は次を保持する。

- 誰が残したか（本人性は authorship を正本とする）。
- いつ残したか（時間）。
- どの場・どの関係の中で残したか（スペース・所属）。
- どこまで見えるか（公開範囲 `visibility`）。

投稿時の名前（投稿時名）と、投稿者の現在の表示名（現在名）を区別して扱う。名前が変わっても過去の投稿の「投稿時名」は変えず、現在名は最新の表示名を反映する（→ 12 章）。

## 10. 個人と集団の記憶・記録

`[思想・原則]`

個人と集団の記憶・記録の両立は、Hossii の**唯一の主目的ではなく、中心価値の一つ**である。同時に、これは特に「なぜビジュアルなのか」（→ 11 章）を説明する思想でもある。

Hossii は次の考え方を土台にする。

- 個人の想いや行動が集団をつくる。
- 集団や場所での経験が個人をつくる。
- 個人があって集団がある。
- 集団があることで個人の経験も生まれる。
- 個人を集団の目的に従属させない。
- 集団を個人投稿の単なる集合として扱わない。
- 個人・集団・場所・時間の関係を保って記録する。
- 記録を促進・可視化・編集・共有・伝達する。
- アーカイブであると同時に、次の対話や行動を生む。

この思想は、システム構造の次の要素に接続する。

- **account / membership** — 個人が、どの集団に、どう関わっているか。
- **community / space** — 集団と、その営みの場。
- **post** — 個人の営みが場に置かれ、集団の記録になる。
- **visibility** — 個人の領域と集団の領域の境界。
- **archive / output** — 記録の蓄積と、次を生む再編集・共有。

## 11. ビジュアルである理由

`[思想・原則]`

Hossii がビジュアルであるのは、装飾のためではなく、10 章の思想を成立させるためである。

- 記録が空間に置かれる（時間の羅列ではなく、場の中に位置づく）。
- 関係する記録が近くに見える。
- 個人と集団の記録が同じ景色に重なる。
- 時間の経過が感じられる。
- 場の文化や世界観が現れる。
- 熱量や密度を感じられる。
- レポート・スクリーン・ZINE などへ再編集できる。
- 過去の保存だけでなく、次の対話を生む。

テキストの時系列だけでは、個人・集団・場所・時間の関係は表現しきれない。ビジュアルは、これらの関係を「同じ景色」として同時に見せるための手段である。

## 12. 個人とコミュニティの関係

`[思想・原則]` `[直近方針]`

### 12.1 Hossii 全体の個人領域

Hossii 全体としての個人領域は、必ずしもスペースではない。次を含む。

- マイページ
- 横断マイログ（複数スペース横断で自分の投稿を見る）
- 所属コミュニティ
- 参加スペース
- 共通プロフィール

これらは「本人がどこに属し、何を残してきたか」を横断的に見るための領域であり、特定の一つの space ではない。

### 12.2 コミュニティ内個人スペース

一方、**コミュニティ内個人スペース**は、特定コミュニティの目的に沿った実際の space である。共有範囲の例：

- 本人のみ
- 本人とコーチ
- 本人と管理者
- 本人が明示的に共有した相手

### 12.3 用語を混同しない

次を同義語として扱わない。

- **My Hossii**（本人の表現・アバター的な存在）
- **MyLogs**（横断的に自分の投稿を見る領域）
- **マイページ**（個人領域の入り口）
- **個人スペース**（コミュニティ内の実際の space）

これらは異なる概念であり、下位文書でも区別する。

## 13. 直近のプロダクト方針

`[直近方針]`

- 汎用 SNS 化ではなく、**コミュニティや目的に特化した環境**を優先する。
- 本人性（authorship）・所属（membership）・アクセス制御を、UI 表示だけでなく DB / RLS でも安全に成立させる。
- 個人領域（マイページ・横断マイログ・参加スペース）を、まず横断的に見られる状態にする。
- 個人が自分の投稿・表示名・参加スペースを、安全に管理できるようにする。
- 大規模な作り直しを避け、既存構造・既存 UI パターンを優先して段階的に積み上げる。

## 14. 現在の実装状態

現在の実装状態は、**`feat/community-platform-phase2-6` branch と development での検証**に基づく。すべて **main / production には未反映**である。検証の範囲は **unit test / lint / build と、実 authenticated session を用いた DB・API・RLS レベルの受入**である。UI の実ブラウザ受入は Playwright 未整備のため **browser 未確認**（該当項目を「ブラウザ検証済み」とは書かない）。

### 14.1 専用 branch / development で実装・検証済み `[実装済み・専用branch検証]`

**Phase 1（Identity）・space membership 周辺（従来）**
- authorship（`hossii_authorships` を本人性の正本とする）
- メールログイン導線
- space membership（`space_memberships`）と自動登録
- 投稿時名＋現在名、本人投稿編集 / `owner_only` / soft delete、mutation 権限堅牢化
- 参加スペース一覧、スペースニックネーム本人変更

**Phase 2〜5（`feat/community-platform-phase2-6`）**
- `community_memberships` と管理者メンバー追加 RPC
- RLS 基盤（`can_access_space`）
- コミュニティ内個人スペース（`ensure_my_personal_space`）
- 個人スペーステンプレート（community 設定・作成時適用）
- shared space `access_mode`（public / invite_only）と space member 管理 RPC

**Phase 6（同 branch）**
- community HOME・複数コミュニティ切替（localStorage は UX のみ）
- 招待（token hash 保存・リンクコピー UI。**メール自動送信未接続**）
- community membership suspend / reactivate / remove
- 管理者発行参加 ID の membership 明示連携（オプトイン）
- suspended / removed 時の community スコープアクセス遮断

### 14.2 未実装 `[設計案・未実装]` / `[将来構想]`

- メール招待の自動送信（SMTP / Edge 未接続）
- account merge（発行 ID ↔ メールアカウント自動統合）→ 110 参照
- pair / team スペース（Phase 6 後）
- アカウント横断 HOME の完成
- 101（管理者発行参加者）の deprecated 化

## 16. Hossiiがやらないこと

`[思想・原則]`

- 汎用的な公開 SNS を目指さない。
- 個人を集団の目的に従属させない。
- 集団を、個人投稿の単なる集合として扱わない。
- セキュリティを UI 表示だけに依存させない（DB / RLS で強制する）。

| コミュニティ | 所属・目的・継続性・運営の単位 |
| スペース | 対話・活動・記録が営まれる視覚的な場 |
| テンプレート | 目的に応じた構成・機能・世界観の組み合わせ |
| 投稿（Hossii） | 場に置かれる対話・活動・記録の最小単位 |
| authorship | 投稿の本人性の正本（`hossii_authorships`） |
| 投稿時名 | 投稿した時点の表示名。後から変えない |
| 現在名 | 投稿者の現在の表示名。最新を反映 |
| 公開範囲（visibility） | 誰に見えるか（`public` / `members_only` / `invite_only` / `owner_only` 等） |
| 個人領域 | マイページ・横断マイログ・所属コミュニティ・参加スペース・共通プロフィール。必ずしも space ではない |
| コミュニティ内個人スペース | 特定コミュニティの目的に沿った、本人中心の実際の space |
| アーカイブ / アウトプット | 記録の蓄積と、その再編集・持ち帰り |
