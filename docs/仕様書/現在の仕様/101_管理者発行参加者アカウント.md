## 0. 要約（変更差分の見積もり）

**結論: 差分は中〜小規模で実現可能。**

既存の Supabase Auth・`LoginScreen`・`GuestEntryScreen`・`user_profiles`・`space_nicknames` を活かし、**新規テーブル 1 本 + Edge Function 1 本 + 管理 UI 1 タブ + 参加者ログイン導線の有効化 + `authorId` を Auth UID に寄せる 1 箇所** が中心になる。

| レイヤ | 新規 | 既存活用 | 備考 |
|--------|------|----------|------|
| DB | `space_participant_accounts` | `user_profiles`, `space_nicknames`, `auth.users` | 発行スロット管理のみ追加 |
| 認証 | Edge Function（発行時のみ service role） | `AuthContext.login()` / セッション永続化 | 参加者ログインは既存 API |
| 管理 UI | SpaceSettings に「参加者アカウント」タブ | 既存タブ・保存 UX | 最大 **50** スロット・一括発行（§8.5） |
| 参加 UI | 初回ニックネームモーダル（簡易版） | `GuestEntryScreen` / `LoginScreen` | Coming soon 解除 |
| 本人識別 | ログイン時 `profile.id = auth.uid()` | 投稿 `authorId` パイプライン | **端末横断の要** |

**スコープ外（本フェーズ）:** 一般向け自己登録、OAuth、ゲスト投稿のアカウント引き継ぎ、パスワード変更 UI。

## 1. 背景・目的

### 1.1 現状

| 参加方法 | 状態 | 本人識別 |
|----------|------|----------|
| ゲスト（ニックネームのみ） | ✅ 利用可 | 端末ローカル `hossii.profile.id` |
| 管理者発行アカウントでログイン | ✅ 利用可 | `auth.uid()` |
| 新規会員登録（自己登録） | UI は **Coming soon** | [41](./41_個人ユーザー登録.md) で将来想定 |

ゲストは同じブラウザなら再入室が楽だが、**端末を変えると別人** になる（[02](./02_匿名利用機能.md)）。

### 1.2 目標

ワークショップ・内省スペースなど **限定人数** の場で、管理者が事前に **参加 ID とパスワードを発行** し、参加者がそれでログインして記録を残せるようにする。

> **初版（MVP）:** 最大 **20** スロット。現行は §8.5（最大 **50** スロット・一括発行）。

| 要件 | 内容 |
|------|------|
| 自己登録 | **まだ不可**（LoginScreen の新規登録タブ・GuestEntryScreen の登録導線は非表示のまま） |
| 管理者発行 | スペースごとに最大 **50** スロットの ID / パスワードを発行（初版は 20） |
| 初回ログイン | **ニックネーム登録のみ**（メール・生年月日等は不要） |
| 同一ブラウザ | Supabase セッションにより **そのまま再ログイン**（GuestEntryScreen をスキップ） |
| 別端末 | 「アカウントでログイン」→ 同じ ID / パスワード → **同一人物** として扱う |
| 記録 | ログイン後の投稿は `authorId = auth.uid()` でサーバーに紐づく。「私のログ」が端末横断で一致 |

## 2. ユーザー種別の整理

本仕様で追加する第 4 種別（既存 3 種別は [27_User別.md](./27_User別.md) 参照）。

| 種別 | 識別 | 登録経路 | ログイン |
|------|------|----------|----------|
| スーパー管理者 | `isSuperAdmin` | 運営手動 | `/admin/login` |
| コミュニティ管理者 | `isAdmin` | コミュニティ申請 | `/admin/login` |
| **発行参加者**（本仕様） | `auth.uid()` + `space_participant_accounts` | **管理者がスペース設定から発行** | スペース参加画面 |
| ゲスト | 端末ローカル ID | ニックネームのみ | 不要 |
| 一般ユーザー（将来） | `auth.uid()` | 自己登録 [41] | LoginScreen |

> 発行参加者は `app_metadata.role` を付与しない（管理者権限なし）。`isAdmin: false` の一般参加者として扱う。

## 3. 管理者側フロー

### 3.1 入口

**スペース設定画面（`SpaceSettingsScreen`）** に新タブ **「参加者アカウント」** を追加する。

- 表示条件: コミュニティ管理者（`isAdmin: true`）のみ
- 対象: **アクティブな 1 スペース** につき最大 **50** スロット（初版 UI モックは 20 件表記）

> **共有URL参加 UI:** 参加者側ログイン導線の詳細は [123](./docs/123_URLからスペース参加UI.md) を正とする。

### 3.2 発行 UI

> **初版 UI モック（20 件表記）。** 現行は §8.5（最大 50 件・一括発行 UI）。

```
┌─ 参加者アカウント ─────────────────────────────┐
│ このスペース専用のログイン ID を最大 20 件まで     │
│ 発行できます。参加者に ID とパスワードを共有       │
│ してください。                                   │
│                                                  │
│ 発行済み: 3 / 20                                 │
│                                                  │
│ ┌────┬──────────┬────────────┬────────────────┐ │
│ │ #  │ 参加 ID   │ 状態       │ 操作           │ │
│ ├────┼──────────┼────────────┼────────────────┤ │
│ │ 1  │ ws-01    │ 発行済み   │ [再表示][無効化]│ │
│ │ 2  │ ws-02    │ 初回未使用 │ [再表示][無効化]│ │
│ │ 3  │ —        │ 未発行     │ [発行]          │ │
│ │ …  │          │            │                │ │
│ └────┴──────────┴────────────┴────────────────┘ │
└──────────────────────────────────────────────────┘
```

| 操作 | 動作 |
|------|------|
| **発行** | 空きスロットに参加 ID・初期パスワードを生成。モーダルで **一度だけ** 平文パスワードを表示（コピーボタン付き） |
| **再表示** | 参加 ID のみ表示。**パスワードは再表示不可**（セキュリティ）。必要なら **パスワード再発行** |
| **パスワード再発行** | 新パスワードを生成し、旧パスワードは無効化。初回ログイン済みかどうかは維持 |
| **無効化** | スロットを「未発行」に戻す。紐づく Auth ユーザーは `banned` または論理削除。既存投稿は保持 |

### 3.3 参加 ID・パスワードの形式

| 項目 | 仕様 |
|------|------|
| 参加 ID | スペース内一意。例: `{spaceSlug の短縮}-01` 〜 `-20`、またはランダム 6 桁英数字。管理者向け UI では **人間が読みやすい連番** をデフォルトとする |
| 初期パスワード | ランダム 8 文字（英大小 + 数字）。発行時のみ平文表示 |
| 上限 | **1 スペースあたり 50 スロット**（初版は 20 固定。migration `20260721130000` で拡張） |
| Auth 内部メール | 参加者には見せない。`{space_uuid}.{login_id}@participants.internal` 形式で Supabase Auth に登録 |

### 3.4 発行 API（Edge Function）

クライアントから service role で Auth ユーザーを作れないため、**Edge Function `issue-participant-account`** を用意する。

```
POST /functions/v1/issue-participant-account
Authorization: Bearer {管理者の access token}

Body: { spaceId, slotNumber?, regeneratePassword?, revoke? }
```

| チェック | 内容 |
|----------|------|
| 呼び出し元 | JWT の `app_metadata.role === 'admin'` かつ対象 `spaceId` が自コミュニティ配下 |
| 副作用 | `auth.admin.createUser` → `space_participant_accounts` INSERT/UPDATE |
| 返却 | `{ loginId, password }`（再発行・新規発行時のみ password を含む） |

## 4. 参加者側フロー

### 4.1 全体像

```
/s/[slug] にアクセス
  │
  ├─ Supabase セッションあり（発行参加者）
  │     │
  │     ├─ ニックネーム未設定 → 初回ニックネームモーダル → SpaceScreen
  │     └─ ニックネーム済み → GuestEntryScreen スキップ → SpaceScreen
  │
  └─ 未ログイン
        │
        ↓
  GuestEntryScreen
        │
        ├─「ゲストとして参加」→ 従来どおり（本仕様で変更なし）
        │
        └─「アカウントでログイン」→ ParticipantLoginScreen
              │
              ├─ 参加 ID + パスワード入力
              ├─ 認証成功
              ├─ 初回のみ → ニックネーム入力（1 項目のみ）
              └─ SpaceScreen へ入室

ゲスト入室後（スペース内）
  │
  └─ アカウントタブ →「アカウントでログイン」→ ParticipantLoginScreen
        （新規会員登録ボタンは Coming soon のまま無効）
```

### 4.2 GuestEntryScreen の変更

| 項目 | 変更 |
|------|------|
| 「アカウントでログイン」 | **有効化**（Coming soon バッジ削除） |
| 「新規登録の方はこちら」 | **非表示のまま**（自己登録は未開放） |
| ログイン済み参加者 | セッション検知時は GuestEntryScreen を **スキップ**（[16](./16_ゲスト入室フロー.md) のゲスト自動入室と同様の分岐を Auth 側に追加） |

### 4.2.1 App.tsx のルーティング優先順位

`pendingParticipantSpaceId` がセットされたとき **ParticipantLoginScreen を GuestEntryScreen より先に表示** する。
GuestEntryScreen の表示条件と pendingParticipantSpaceId が同時に真になると、ログインボタンを押しても画面が切り替わらない不具合が起きるため、本順序を仕様として固定する。

| 順序 | 条件 | 表示 |
|------|------|------|
| 1 | 未ログイン かつ pendingParticipantSpaceId あり | ParticipantLoginScreen |
| 2 | 未ログイン かつ guestSpaceId あり かつ ゲスト入室前 | GuestEntryScreen |
| 3 | 未ログイン かつ pendingLoginSlug あり | LoginScreen（汎用・自己登録用。本機能では未使用） |

ゲストモード中（`isGuestMode === true`）でも、pendingParticipantSpaceId がセットされれば ParticipantLoginScreen を全画面表示する。

### 4.3 参加者ログイン画面

**ParticipantLoginScreen** を新設済み。汎用 LoginScreen は使わない。

**入力項目（ログインのみ）:**

| 項目 | 必須 |
|------|------|
| 参加 ID | ✅ |
| パスワード | ✅ |

- メールアドレス欄は **出さない**
- Google / Facebook は **出さない**
- 新規登録タブは **出さない**
- スペース文脈（`spaceId`）は URL / 参加フローから自動で渡す（参加者が space を選ばない）

**認証処理:**

```
loginParticipant(spaceId, loginId, password)
  │
  ├─ space_participant_accounts から auth 用メールアドレスを解決
  ├─ supabase.auth.signInWithPassword({ email, password })
  └─ resolveAppUser() → isAdmin: false, uid 確定
```

### 4.4 初回ニックネーム登録

| 項目 | 仕様 |
|------|------|
| 表示条件 | ログイン成功後、`space_nicknames` に当該 `spaceId` のレコードがない |
| 入力 | ニックネーム **1 項目のみ**（1〜30 文字） |
| 保存先 | `space_nicknames`（`profile_id = auth.uid()`）+ 必要に応じ `user_profiles.username` |
| 再表示 | **初回のみ**。2 回目以降はスキップ（ニックネーム変更はアカウント画面から、将来対応可） |

ゲスト参加時の Hossii キャラ・吹き出し UI は [16](./16_ゲスト入室フロー.md) / `GuestEntryScreen` のニックネームステップを **再利用** する。

### 4.5 2 回目以降の入室

| 状況 | ふるまい |
|------|----------|
| **同じブラウザ** | Supabase がセッションを保持 → アプリ起動 / `/s/[slug]` アクセス時に自動ログイン → GuestEntryScreen スキップ → SpaceScreen |
| **別端末・別ブラウザ** | GuestEntryScreen →「アカウントでログイン」→ 同じ参加 ID / パスワード → 同一 `auth.uid()` → サーバー上のニックネーム・投稿履歴を復元 |
| **ゲスト入室中** | アカウントタブ →「アカウントでログイン」→ ParticipantLoginScreen → 認証後も当該スペースに留まる |
| **ログアウト後** | 再度 ID / パスワード入力が必要（セッション削除） |

## 5. 本人識別と投稿の紐づけ

### 5.1 問題（現状）

投稿の `authorId` は **端末ローカル** の `hossii.profile.id` を使用している（[02](./02_匿名利用機能.md)）。Auth でログインしても `authorId` が Auth UID と一致しない。

### 5.2 変更方針（最小差分）

| ユーザー状態 | `profile.id` / `authorId` |
|--------------|---------------------------|
| ゲスト | 従来どおり端末ローカル ID（変更なし） |
| **発行参加者（ログイン済み）** | **`auth.uid()` を使用** |
| ログアウト後ゲスト継続 | ローカル ID にフォールバック（ゲスト投稿と混在しないよう、ログイン中はローカル ID を上書き） |

**変更箇所（想定）:** `HossiiStoreProvider` で `currentUser` が存在し `isAdmin === false` のとき、初回プロフィール生成・投稿時の `authorId` に `currentUser.uid` を使う。

### 5.3 「私のログ」

[83](./83_ログ一覧の私のログ切替.md) の `authorId === profile.id` フィルタが、発行参加者では **端末横断で一致** するようになる。

## 6. データベース設計

### 6.1 新規テーブル `space_participant_accounts`

| カラム | 型 | 説明 |
|--------|-----|------|
| `id` | uuid | PK |
| `space_id` | uuid | FK → `spaces.id` ON DELETE CASCADE |
| `slot_number` | int | 1〜**50**（初版は 1〜20）。`(space_id, slot_number)` UNIQUE |
| `login_id` | text | スペース内表示用 ID。`(space_id, login_id)` UNIQUE |
| `auth_user_id` | uuid | FK → `auth.users.id`。発行時に作成 |
| `auth_email` | text | Auth 用内部メール（UNIQUE） |
| `status` | text | `'active'` / `'revoked'` |
| `first_login_at` | timestamptz | 初回ログイン日時（NULL = 未使用） |
| `issued_at` | timestamptz | 発行日時 |
| `issued_by` | uuid | 発行した管理者の `auth.uid()` |

**制約:**

```sql
CHECK (slot_number BETWEEN 1 AND 50)   -- migration 20260721130000（初版は 1 AND 20）
```

**revoked slot:** DB 行が残るため **occupied** 扱い。再利用はしない（空き slot 計算は active ではなく行の有無）。

### 6.2 既存テーブルの利用

| テーブル | 用途 |
|----------|------|
| `auth.users` | 参加者の認証主体。`app_metadata.participant = true`（任意） |
| `user_profiles` | 任意。初回ニックネームを `username` にもミラー可 |
| `space_nicknames` | スペース別表示名（`profile_id = auth.uid()`） |
| `hossiis.author_id` | 投稿本人識別（`auth.uid()` が入る） |

### 6.3 RLS

| テーブル | ポリシー |
|----------|----------|
| `space_participant_accounts` | **管理者**: 自コミュニティ配下スペースのみ SELECT。INSERT/UPDATE は Edge Function（service role）経由のみ |
| `space_participant_accounts` | **参加者**: SELECT 不可（ログイン ID 解決は Edge Function または security definer RPC） |
| `space_nicknames` / `hossiis` | 既存ポリシーを踏襲 |

**ログイン ID 解決:** 参加者が `login_id` だけ入力するため、**スペース ID + login_id → auth_email** の lookup は RPC（`resolve_participant_login`）または Edge Function で行い、クライアントに他スロットの情報を漏らさない。

## 11. セキュリティ・運用上の注意

- 初期パスワードは **HTTPS 前提** で管理者から参加者へ安全に共有する（口頭・対面推奨）
- 平文パスワードは **発行/再発行時のみ** UI 表示。DB には Auth 側のハッシュのみ
- ブルートフォース対策: Supabase Auth の rate limit + 必要なら Edge Function 側でも IP 制限
- 参加 ID はスペース URL を知っている人にのみログイン画面が見えるが、**URL 共有＝入室可能** である点はゲストと同様
