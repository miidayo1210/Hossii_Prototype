## 1. 目的

スペース上の投稿を、**投稿の新しさ（時系列）** に応じて空間の奥へ蓄積されていくように見せる。

| 表現 | 意図 |
|------|------|
| 新しい投稿 | 手前に見える（既存 z-index）＋ 現行サイズ（1.00）を維持 |
| 古い投稿 | 奥に見える（既存 z-index）＋ 段階的に小さく |
| 時間経過 | 投稿が空間の奥へ沈んでいくメタファー |

**本機能が解決しないこと:** 投稿の「重要度」「いいね数」「管理者評価」と新しさの混同。新しさは **表示上の奥行き** のみに使う。

**初版の範囲（確定）:** 奥行き表現は **外形サイズ（星ドットのみ）** と **既存 z-index** に限定する。opacity・影・blur・新規アニメーションは追加しない。

**有効化（確定）:** 時系列奥行きは **全スペースへ自動適用しない**。スペース管理者がスペース設定で **ON にしたスペースのみ** 許可される。デフォルトは **OFF**。ただし設定 ON でも **表示への適用はメイン／デフォルト Pane の星モードに限定**する（§2.1）。追加 Pane では設定 ON でも奥行き表示は **適用しない**。OFF または設定値未定義の場合は **現行表示を完全に維持**（depth scale = 1.00 相当）。

**DB 設定の意味（確定）:** `timelineDepthEnabled` は「このスペースでメイン画面の時系列奥行き表示を **使ってよいか**」のスペース単位許可値。Pane 別の ON/OFF 列は **初版では持たない**（§14.1）。

**現行との関係:** [87](./87_スペース表示パフォーマンス最適化.md) の **index ベース z-index** と **animation tier**（0–9: full, 10–29: light, 30+: none）は **初版では変更しない**。サイズだけを `timelineDepthScaleFromIndex` で追加する（設定 ON 時のみ）。

## 2. 初版の対象モード（確定）

### 2.1 適用 Pane（確定）

時系列奥行きが **実際に表示へ反映される** のは、次の **すべて** を満たす場合のみ。

| # | 条件 |
|---|------|
| 1 | `timelineDepthEnabled === true`（DB 正本・§14） |
| 2 | **現在表示中の Pane がメイン／デフォルト Pane**（§2.1.1） |
| 3 | 星モード（`presentationMode === 'stars'` / モバイル縦 `useStarView`） |
| 4 | 描画対象が `starDot`（§2 対象表） |

いずれかを満たさない場合は **従来表示を維持**（depth scale = 1.00 相当）。

| Pane 種別 | 設定 ON 時の奥行き |
|-----------|-------------------|
| **メイン／デフォルト Pane** | ✅ 星モード `starDot` に適用（上記 3・4 も満たす場合） |
| **追加 Pane**（`is_default = false`） | ❌ **常に対象外** |
| Pane 読み込み中・判定不能 | ❌ 安全側で OFF |

**Pane 別 boolean・`timelineDepthScope` enum・DB 列追加は初版対象外。** 将来 Pane 別適用が必要になった段階で **別仕様** として検討する。

#### 2.1.1 メイン／デフォルト Pane の判定（確定）

URL に Pane 指定（slug）が **ない** ことだけで判定 **しない**。既存の Pane 解決処理に従う。

| 情報源 | 用途 |
|--------|------|
| `space_panes.is_default` | DB 上のデフォルト Pane フラグ |
| `SpacePaneProvider.defaultPane` | `pickDefaultPane` による解決結果（`is_default` 行、なければ heal / フォールバック） |
| `SpacePaneProvider.activePane` | URL slug + `resolveActivePane` 後の表示中 Pane |
| `PaneContext`（`hossiiPaneMembership`） | `activePaneId` と `defaultPaneId` の比較 |

**採用する判定（Phase 2 実装案）:**

```ts
// paneContext が揃っているとき（SpaceScreen 標準経路）
isMainPane = isDefaultPane(paneContext)
// 実装: activePaneId === defaultPaneId

// または同等
isMainPane = activePane != null && defaultPane != null && activePane.id === defaultPane.id
```

**採用しない判定:**

- 「配列の先頭だからメイン」
- 「URL に pane slug が無いからメイン」（追加 Pane の slug URL でもメインと誤判定し得るため slug 単体は不可）
- `activePane.isDefault` **単独**（`pickDefaultPane` が `panes[0]` フォールバックした場合、`isDefault` フラグが false の行を defaultPane として扱うことがあるため。**ID 一致**を正とする）

#### 2.1.2 フォールバック（既存コード整合）

| 状況 | 扱い |
|------|------|
| **Pane 機能前の既存スペース**（pane 0 件） | `ensureDefaultSpacePane` / `createDemoDefaultPane` で **1 つの default Pane を heal**。表示はメインとして扱う |
| **明示的 `is_default` 行あり** | その Pane のみメイン。追加 Pane は `is_default = false` で対象外 |
| **追加 Pane を URL で表示中** | `activePaneId !== defaultPaneId` → 奥行き **OFF** |
| **Pane 読み込み中**（`panesLoading` / `activePane === null`） | 奥行き **OFF**（フラッシュ防止） |
| **Pane 取得失敗**（Supabase 設定時 `activePane === null`） | 奥行き **OFF** |

**現行実装参照:** `SpacePaneProvider`（`pickDefaultPane`, `resolveActivePane`）、`src/core/utils/hossiiPaneMembership.ts`（`isDefaultPane`）、`src/core/utils/resolveActivePane.ts`。

### 前提：スペース設定が ON のときのみ

| 項目 | 仕様 |
|------|------|
| **設定キー** | `timelineDepthEnabled`（アプリ型・camelCase） |
| **デフォルト** | **OFF**（新規・既存スペースとも。未定義も OFF） |
| **操作** | スペース管理者のみ（§14） |
| **ON 時** | 下記「対象」かつ **メイン／デフォルト Pane** に depth scale を適用（§2.1） |
| **OFF 時** | 現行表示を完全維持。`timelineDepthScaleFromIndex` は使わず常に 1.00 相当 |

### 対象（設定 ON かつ メイン Pane 表示時のみ）

| モード | 初版 | サイズ変更対象 |
|--------|------|----------------|
| **星モード**（PC `presentationMode === 'stars'` / モバイル縦 `useStarView`） | ✅ | `starDot` の視覚的外形のみ |

星モードでも **モバイル横持ち** は `renderAsStar` だがプレビュー非表示。`starDot` への depth scale 適用は星モード共通とする（**設定 ON かつ メイン Pane 時のみ**）。

### 初版対象外（設定 ON でも適用しない）

| モード / 要素 | 初版 | 備考 |
|---------------|------|------|
| **追加 Pane**（メイン以外の `space_pane`） | ❌ | §2.1。DB 設定 ON でも表示は従来サイズ |
| 星プレビュー吹き出し（`previewBubble`） | ❌ | ホバー領域修正との競合回避。Phase 3 で検討 |
| `HossiiFullTextPopover` / `PostDetailModal` | ❌ | 常に通常サイズ |
| Bubble 表示（`presentationMode === 'custom'` 等） | ❌ | Phase 3 以降 |
| **カスタムモード** | ❌ | `hossii.scale` を常に優先（§13） |
| **ordered**（投稿順格子） | ❌ | 将来案あり（§13） |
| **byAuthor** | ❌ | 将来検討 |
| **slideshow** | ❌ | 基本対象外 |

## 3. 新旧判定の基準（確定）

| 項目 | 仕様 |
|------|------|
| **主キー** | `filteredHossiis` の **描画 index**（0 = 最新） |
| **ソート** | `createdAt` 降順（`runDisplayPipeline` と同じ） |
| **表示上限** | `displayLimit` スライス後の配列に対する index（DB 全体の順位ではない） |
| **タグ絞り込み後** | 絞り込み後配列で index を **再計算**（相対的新しさ） |
| **新規投稿** | 先頭に追加され、既存投稿の index が +1 されることは **許容**（段階的に奥へ） |
| **ピン留め** | ソート順は変えない。サイズは通常の時系列を適用（§12） |

### ページング・表示件数（確定・現行実装に合わせた記述）

| 操作 | 既存投稿の index |
|------|------------------|
| `displayLimit` を **増減** | 表示中の投稿は **基本的に変わらない**（より古い投稿が末尾に追加 or 末尾から消える） |
| unlimited **追加読み込み** | 末尾 append のみ。既存 index **不変** |
| **新規投稿**のリアルタイム追加 | 先頭挿入により既存がシフト → **許容** |
| **タグ絞り込み**変更 | 絞り込み後で index 再計算 → tier が変わる可能性あり **許容** |

### 将来案

- `createdAt` からの **経過日数** ベースの段階
- **全投稿中の絶対順位**（fetch 完了前提）

~~`runDisplayPipeline` と fetch 側のソート不一致~~ → **Phase 0 で `sortHossiisNewestFirst` に統一済み**（コミット `8677b25`）。

## 4. サイズ段階（確定・初期実装値）

初版は **4 段階**。最新投稿は現行サイズ **1.00 を維持**し、古い投稿だけ段階的に小さくする。**1.00 を超える拡大は行わない**（既存配置の衝突を増やさない）。

| 描画 index | 段階 | depth scale（初期実装値） |
|----------|------|---------------------------|
| **−1 以下**（防御） | — | **1.00** |
| 0–9 | L | **1.00** |
| 10–29 | M | **0.94** |
| 30–59 | S | **0.88** |
| 60+ | XS | **0.82** |

> 上記数値は **初期実装値**。Phase 2 の実機確認後に調整可能。

### 純関数（確定）

z-index・animation・opacity には触れず、**サイズ係数だけ**を返す。

```ts
// 配置案: src/core/utils/timelineDepthScale.ts（または animationLevel.ts 隣接）
timelineDepthScaleFromIndex(index: number): number
```

`resolvePostDepth()` のような一括統合関数は **初版では採用しない**。

## 14. 管理画面 ON/OFF（機能仕様：確定）

### 14.1 基本方針

| 項目 | 仕様 |
|------|------|
| **設定単位** | スペース単位の **許可値**（Pane 別 boolean は **しない**） |
| **表示適用** | **メイン／デフォルト Pane のみ**（§2.1）。追加 Pane は対象外 |
| **操作できる人** | コミュニティ管理者（`currentUser.isAdmin === true`。[92](./92_管理スペース設定.md) §2.2） |
| **デフォルト** | **OFF** |
| **未定義** | **OFF** として扱う（既存スペース含む） |
| **ON 時** | メイン Pane の星モード `starDot` に時系列奥行きを適用（§2.1 の全条件を満たす場合） |
| **OFF 時** | 現行表示を **完全に維持**（depth scale の見た目上の影響なし） |
| **将来** | 対象モード・**対象 Pane** を増やしても、初版は同一 `timelineDepthEnabled` を起点に別仕様で拡張検討 |

### 14.2 設定キー・判定（確定）

| 層 | キー |
|----|------|
| **TypeScript（`SpaceSettings`）** | `timelineDepthEnabled?: boolean` |
| **DB 列（採用予定・正本）** | `timeline_depth_enabled boolean NOT NULL DEFAULT false` |
| **判定関数** | `resolveTimelineDepthEnabled(settings) => settings?.timelineDepthEnabled === true`（Pane 判定は含まない） |
| **表示適用判定（Phase 2 案）** | `resolveTimelineDepthActive({ enabled, isMainPane, isStarMode })` 等の純関数で合成。`isMainPane` は §2.1.1 |

> 厳密に `=== true` とし、`undefined` / `false` はいずれも OFF。安全側に倒す。

**localStorage のみの試作は採用しない。** 端末間・管理者間で共有されず、DB 権限も検証できないため。正式実装は **DB 列 + migration**（Phase 1.5B。migration は Phase 1.5A 権限調査後）。

### 14.3 DB と localStorage の優先順位（確定）

| ルール | 内容 |
|--------|------|
| **正本** | `space_settings.timeline_depth_enabled`（DB） |
| **キャッシュ** | `localStorage` `space_settings_{spaceId}` — DB 取得後のミラー。正本ではない |
| **DB が `false`** | localStorage に `true` が残っていても **OFF** |
| **DB に行・列がない** | **OFF**（migration 前・新規スペース含む） |
| **localStorage にキーなし** | **OFF** |
| **DB 取得前の初期表示** | **OFF**（フラッシュで ON に見せない） |
| **DB 保存失敗** | UI を ON のまま確定させない。直前値へロールバックしエラー表示 |
| **保存成功時** | DB 成功後に localStorage と画面 state を確定（既存 `AppearanceTab` パターン。失敗時は必ずロールバック） |
| **別スペース** | `spaceId` 単位のキー／行のみ参照。他スペースのキャッシュを使わない |

`fetchSpaceSettings` 完了後は **DB 値で localStorage を上書き**し、古いキャッシュが勝たないようにする（`random_recall_enabled` 等と同様、本項目も DB 優先マージを Phase 1.5B で明示実装）。

### 14.4 管理者権限（多層防御・確定）

UI の `isAdmin` 非表示だけを権限保証と **しない**。クライアントの `isAdmin` は **DB 側では信頼しない**。

| 層 | 要件 |
|----|------|
| **UI** | 非管理者には設定項目を **表示しない**（`AppearanceTab` + `isAdmin`） |
| **保存 API（クライアント）** | **§14.4.1** のとおり。早期エラー表示用。**最終的な権限保証は DB** |
| **DB** | **§14.4.2** の列限定トリガーで、当該列の管理者以外の変更を拒否 |
| **読み取り** | 非管理者が設定値を **読み取ってよい**（スペース画面で ON/OFF を反映）。**更新は不可** |

**Phase 1.5A 結論:** `space_settings` 全体の RLS 差し替えは行わず、**`timeline_depth_enabled` 列限定トリガー** を Phase 1.5B で追加する。

#### 14.4.1 クライアント側ガード（Phase 1.5B — 確定）

**`upsertSpaceSettings` 全体に `isAdmin` 必須ガードを追加しない。** 他設定（`likesEnabled`・`randomRecallEnabled` 等）の保存経路と共有しており、関数全体を管理者限定にすると既存機能へ影響する。

次の **いずれか**（実装時に選択。108 では両方許容）:

| 方式 | 内容 |
|------|------|
| **A. 専用保存関数（推奨）** | 例: `upsertTimelineDepthEnabled(settings, enabled)` — 本列の DB 更新のみ。入口で `isAdmin` を確認し、非管理者は即エラー |
| **B. 汎用保存内の条件ガード** | `upsertSpaceSettings` 内で、**`timelineDepthEnabled` が変更される場合だけ** 管理者確認。他列のみの更新は従来どおり |

いずれもクライアント側 `isAdmin` は **UX 用の早期拒否** とし、DB トリガーが最終防衛線。

#### 14.4.2 DB 列限定トリガー（Phase 1.5B — 確定）

既存の他設定更新を妨げない。**`timeline_depth_enabled` が関わるときだけ** 管理者を要求する。

**INSERT 時**

| 条件 | 挙動 |
|------|------|
| `NEW.timeline_depth_enabled = true` | 管理者確認 **必須**。非管理者の upsert 経由での `true` 新規保存は **拒否** |
| `false` またはデフォルト（`false`）での INSERT | 既存互換のため **許可**（管理者確認なし） |

**UPDATE 時**

次のとき **のみ** 管理者確認（`false → true` と `true → false` の **両方** を管理者限定）:

```sql
NEW.timeline_depth_enabled IS DISTINCT FROM OLD.timeline_depth_enabled
```

`timeline_depth_enabled` が変わらず、他列だけ更新する場合は **本トリガーで拒否しない**。

**管理者判定（DB）**

- 対象行の `space_id` → `spaces.community_id` → `communities.admin_id = auth.uid()`
- 既存 my_hossii 保護（`enforce_my_hossii_space_settings_admin`）と同様、`auth.jwt() -> 'app_metadata' ->> 'role' = 'super_admin'` も許可
- クライアントから渡された `isAdmin` は **使用しない**

**トリガー実装注意**

- `SECURITY DEFINER` にする場合は既存 migration と同様 `SET search_path = public` を固定し、意図しないオブジェクト参照を避ける

### 14.5 既存セキュリティ課題（別タスク — Phase 1.5B に含めない）

以下は重要だが、108 / Phase 1.5B のスコープ **外**。別途対応する。

| 課題 | 備考 |
|------|------|
| `space_settings` 全体の public INSERT / UPDATE / DELETE | 初期スキーマのデモ用ポリシー。本項目の列トリガーでは他列は未保護のまま |
| `#settings` ルートの管理者ガード | `App.tsx` に `isAdmin` チェックなし。UI 非表示だけでは不十分 |
| 既存設定全体の DB 権限整理 | `likesEnabled` 等を含む横断的な RLS / RPC 設計 |

### 14.6 管理画面 UI（確定）

**配置:** [92](./92_管理スペース設定.md) の **「投稿の見た目」**（`AppearanceTab`）。星マーカー形状の近く。

**表示（管理者のみ）:**

```text
時系列による奥行き表示　[ OFF / ON ]

新しい投稿を手前に、古い投稿を少し小さく表示し、
メイン画面に時間の奥行きをつくります。

※ 現在は、メイン画面の星モードにのみ適用されます。
```

**Pane 別設定 UI:** しない。DB boolean はスペース全体の許可値のみ（§14.1）。

### 14.7 保存フロー（Phase 1.5C 目標）

1. 管理者がトグル変更 → 明示的保存
2. クライアントで管理者確認（§14.4.1 — 早期エラー）
3. 専用保存関数または条件付き `upsertSpaceSettings` で **当該列のみ** DB 更新 — **成功を待ってから** localStorage / state 確定
4. 失敗時: トースト等でエラー、draft を直前値にロールバック
5. `SpaceScreen` の `useSpaceSettings` がフォーカス時に `loadSpaceSettings`（DB フェッチ後のキャッシュを反映）

## 18. 懸念点

1. **index の不安定性（タグ絞り込み）** — 許容済み。タグ変更で tier が変わる
2. **読みづらさ** — 古い星が小さすぎないか実機確認（Phase 2）
3. **操作領域** — 見た目は小さくても 44px ヒットボックス必須
4. **recent highlight との二重強調** — §9 未決
5. ~~**ソートタイブレーク不一致**~~ — Phase 0 で解消
6. **新規投稿による index シフト** — 許容済み
7. **重要性との混同** — ヘルプ等で「新しいほど手前」と明示（管理画面説明文で補足）
8. **全件読み込み後** — 最古が XS（0.82）で止まり、それ以上の「奥」は表現できない
9. **ピン留めの古い投稿** — 初版は小さく見える。Phase 4 で昇格検討
10. **設定 OFF 忘れ** — デフォルト OFF のため既存スペースへの影響なし
11. **`space_settings` 全体の public UPDATE** — §14.5 別タスク。本項目は列限定トリガーのみ保護
12. **設定画面への直接遷移** — §14.5 別タスク
13. **DB 取得前のフラッシュ** — 初期 OFF・フェッチ後に DB 優先マージが必要
14. **Pane 切替時のフラッシュ** — メイン ↔ 追加の切替で一瞬だけ誤った scale が出ないこと（Pane 判定は ID 一致で行う）
