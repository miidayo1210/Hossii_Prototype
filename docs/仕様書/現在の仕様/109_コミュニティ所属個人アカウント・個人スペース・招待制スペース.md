## 1. 背景と目的

### 1.1 背景

現行 Hossii は、次の状態にある。

- ログインなしで URL / QR から参加できる**ゲスト体験**が中核。
- コミュニティは `communities.admin_id` による**単一管理者オーナーモデル**。個人の「所属」概念は存在しない。
- スペースは URL（slug）を知っていれば公開スペースに参加でき、`is_private` はアプリ層のログインゲートのみ。
- 「参加権限」を表す公式テーブルは存在せず、`space_nicknames`（表示名）と `space_participant_accounts`（管理者発行アカウント・101）が近似機能。
- ログイン本人性の正本が曖昧（後述の環境ドリフトあり）。

### 1.2 目的

一人の利用者が、複数コミュニティ・複数スペースを横断しながら、次を継続して持てる基盤を整備する。

- 個人プロフィール（既存 `user_profiles`）
- コミュニティ所属（新規 `community_memberships`）
- 自分専用の空間＝**個人スペース**（既存 `spaces` を拡張）
- 招待された共有スペースへの参加（新規 `space_memberships` + `access_mode`）
- 本人としての投稿所有権（`hossii_authorships` を正式採用）

ゲスト体験は public スペースで維持しつつ、個人スペースと invite_only スペースには**アプリと RLS の両方で強制されるアクセス制御**を導入する。

### 1.3 非目的（初版）

- メール引き継ぎ・アカウント統合
- 本人承認フロー（招待の accept）
- 複数コミュニティ切替 UI
- モデレーション新機能（非表示は将来別権限）
- `author_id` の全面整理・一括変更

## 2. LINE にたとえた全体イメージ

| Hossii の概念 | LINE のたとえ | 説明 |
|---------------|---------------|------|
| 個人スペース | 個人チャット（自分専用） | その人専用の空間。本人と所属コミュニティ管理者だけが見られる |
| 共有スペース（invite_only） | グループチャット | 招待された個人だけが参加できる空間 |
| 共有スペース（public） | オープンチャット / 公開リンク | URL を知っていれば（ゲストでも）参加できる現行体験 |
| コミュニティ所属 | 会社・団体のメンバー登録 | どのコミュニティに属しているか。所属していても全グループには入れない |
| 個人アカウント | LINE アカウント | メール/ID で端末を越えて本人を保持 |

**重要な原則:** コミュニティに所属しているだけでは、すべての共有スペースには参加できない。invite_only 共有スペースは**個人単位の招待制**とする。

## 6. Identity の正本（案 A を正式採用）

### 6.1 正式決定

```text
hossiis.author_id
  = 投稿上の表示主体・既存互換の投稿者識別子（当面維持）

hossii_authorships.auth_user_id
  = ログイン本人としての所有権・本人性・編集権限の正本
```

### 6.2 authorship 作成ルール

| 投稿者 | authorship 作成 |
|--------|-----------------|
| ゲスト | **作成しない** |
| 一般ログインユーザー | 作成する |
| 発行参加者 | 作成する |
| コミュニティ管理者 | **作成する** |
| スーパー管理者 | 作成する（本人として投稿する場合） |

作成契機は「ログインセッション付きの新規 `hossiis` INSERT」時の DB trigger（`auth.uid()` のみを根拠とし、クライアント payload は使わない）。

### 6.3 本人判定

- 投稿の**編集・削除・本人ログ一覧（MyLogs）は authorship を優先**して判定する。
- authorship がある投稿は authorship を正本とする。
- authorship がない過去投稿は自動的に本人扱いしない（§14）。
- **ログイン中の本人判定に `author_id` を使ってはならない**（`author_id` フォールバック禁止）。ログイン本人性は `hossii_authorships.auth_user_id`（＝取得済み `myAuthorshipIds`）のみを根拠とする。
- **ゲストのみ**、従来どおり端末側 `profiles.id` と `hossiis.author_id` の一致で本人判定する。
- authorship の取得状態が確定（`myAuthorshipIdsStatus === 'ready'`）していないときは本人扱いを広げず安全側（編集不可）とする。実装確定仕様は **付録 B**。

### 6.4 やらないこと（初版）

- 既存投稿の `author_id` を一括変更しない。
- `author_id` を auth UID へ置き換えない。
- `author_id` の整理は**別 Phase・別 migration**として将来検討する。

### 6.5 スペース所有者と投稿本人性は別軸で常に併存する

個人スペースの所有者判定と、個々の投稿の本人性判定は、**代替関係でも未決事項でもない**。常に併存する 2 つの独立した軸として扱う。

```text
個人スペースへのアクセス・所有者判定
  = spaces.owner_user_id

個々の投稿の作成者・編集削除権限判定
  = hossii_authorships.auth_user_id
```

- owner だから当該スペース内の全投稿が本人、という推論はしない。
- authorship があるから当該スペースの owner、という推論もしない。
- 「本人の個人スペースか」は `owner_user_id`、「本人の投稿か」は authorship で、それぞれ独立に判定する。

#### 具体例（田中さんの個人スペースで管理者が投稿した場合）

| 項目 | 値 |
|------|-----|
| space owner | 田中さん（`owner_user_id` = 田中さんの auth UID） |
| 投稿者 | 管理者 |
| authorship | 管理者の auth UID |
| 田中さんはこの管理者投稿の所有者か | **いいえ**（authorship は管理者） |
| 管理者は田中さん本人の投稿を編集・削除できるか | **いいえ**（本人投稿の authorship は田中さん） |

管理者は個人スペースの閲覧・管理者自身としての投稿はできるが、owner（田中さん）本人の投稿を直接編集・削除することはできない（§10.3 / §15）。

## 7. 認証・プロフィール・所属・スペース参加の分離

4 概念を明確に分離する。

```text
1. 認証アカウント        auth.users            … メール/パスワード/OAuth/synthetic email
2. 個人プロフィール      user_profiles         … 共通ニックネーム・アバター・通知設定
3. コミュニティ所属      community_memberships … どのコミュニティに所属しているか（role/status）
4. スペース参加          space_memberships     … どの共有スペースに参加できるか（role/status）
```

### 7.1 現行 Identity 表（main 実装ベース・調査結果）

| 利用者 | 認証ID | profile 保存先 | 投稿 author_id | space_nickname |
|--------|--------|----------------|----------------|----------------|
| ゲスト | なし | `profiles`（端末） | 端末 UUID | 端末 UUID |
| 一般ログイン | `auth.users.id` | `user_profiles`（signup 時作成） | auth UID | auth UID |
| 発行参加者 | `auth.users.id` | `user_profiles` あり（ログイン時 ensure） | auth UID | auth UID |
| 管理者 | `auth.users.id` | `user_profiles` **なしの可能性大** | **端末 UUID**（要是正） | auth UID |

### 7.2 目標 Identity 表（本機能適用後）

| 利用者 | 認証ID | profile | 所属 | スペース参加 | authorship |
|--------|--------|---------|------|--------------|------------|
| ゲスト | なし | `profiles` | なし | public のみ（membership なし） | なし |
| 一般ログイン | auth UID | `user_profiles` | `community_memberships` | public + 招待された invite_only | あり |
| 発行参加者 | auth UID | `user_profiles` | `community_memberships` | 同上 | あり |
| 管理者 | auth UID | `user_profiles`（**補完**） | 所属 + admin 権限 | 管理範囲 | あり |

Phase 1 で管理者の `user_profiles` 補完と authorship 作成を行い、7.1 → 7.2 のギャップを埋める。

## 8. コミュニティメンバーシップ

### 8.1 新規テーブル（DB 案）

```text
community_memberships
  id            uuid PK
  community_id  uuid  → communities(id)
  user_id       uuid  → auth.users(id)
  role          text  ('admin' | 'member')          -- 初版は主に 'member'
  status        text  ('invited' | 'active' | 'suspended' | 'removed')
  invited_by    uuid  → auth.users(id)  (nullable)
  invited_at    timestamptz (nullable)
  accepted_at   timestamptz (nullable)
  created_at    timestamptz DEFAULT now()
  updated_at    timestamptz DEFAULT now()
  UNIQUE (community_id, user_id)
```

### 8.2 方針

- **N:N 構造**：1 人が複数コミュニティに所属可能な DB 設計とする。
- **初版の運用・UI は原則 1 コミュニティ所属を中心**に表示する（複数所属切替 UI は将来）。
- `communities.admin_id` は**移行期間中、既存管理者権限の正本として維持**する。
- 初版から membership の `role='admin'` へ完全移行しない（権限判定は当面 `admin_id`）。
- 既存管理者は Phase 2 で `community_memberships(role='admin', status='active')` として backfill する（権限源は当面 `admin_id`、membership は所属の記録）。

### 8.3 状態遷移（community_memberships.status）

```text
invited ──(本人ログイン/管理者有効化)──▶ active
active ──(管理者停止)──▶ suspended ──(管理者復帰)──▶ active
active/suspended ──(管理者解除)──▶ removed
removed ──(再招待)──▶ invited/active   ※復旧方法は将来仕様
```

初版では「管理者が発行・追加した時点で active」とし、本人承認（invited→active の accept）は不要とする（§9）。`invited` 状態は将来のメール招待で使用する余地として定義に残す。

## 9. 個人アカウントの発行・招待

### 9.1 仕様上の統一表現

> コミュニティ管理者が、個人のログインアカウントを発行または招待し、コミュニティ所属を付与する。

### 9.2 初版の方式（101 発行参加者を下敷き）

初版は既存の発行参加者機能（101）を参考に、次を候補とする。

- ID
- 初期パスワード
- synthetic email（`...@participants.internal` 形式）
- 初回ログイン（ニックネーム登録のみ）
- 将来的な本人メールアドレスへの引き継ぎ（**初版対象外**）

### 9.3 101 との関係（初版）

101 でログイン参加者の `author_id` へ auth UID を保存している現行実装は、**案 A と共存可能**である。

- ただし、`author_id` が auth UID であること自体を、本人性・編集権限の正本にはしない。
- ログイン本人性の正本は `hossii_authorships` とする。
- **Phase 1 で、101 を含むすべてのログイン投稿の本人判定を authorship 優先へ移行する。**

補足:
- 101 の `space_participant_accounts` はスペース単位の発行。本機能はコミュニティ単位の所属付与を追加する。
- 初版では両者を**共存**させ、統合／deprecated は Phase 6 で方針確定（§20）。
- Edge Function（service role）で発行する 101 のパターンを、コミュニティ所属発行にも流用できるか Phase 2 で評価する。

### 9.4 初版対象外

- メール引き継ぎ・アカウント統合。
- 実メール送信による招待（Phase 6）。

## 10. 個人スペース

### 10.1 方針

別システムを作らず、既存 `spaces` を拡張して実現する。

```text
spaces（拡張案）
  + space_type     text  ('shared' | 'personal')  DEFAULT 'shared'
  + owner_user_id  uuid  → auth.users(id)  (personal のときのみ NOT NULL 相当)
  + status         text  ('active' | 'suspended' | 'archived')  DEFAULT 'active'
  + community_id   uuid  （既存）
  UNIQUE (community_id, owner_user_id) WHERE space_type = 'personal'
```

- owner は必ず `auth.users.id`。ゲストは owner になれない。
- **1 人につき 1 コミュニティ内で個人スペース 1 つ**（UNIQUE 制約）。
- 同じ人が別コミュニティに所属する場合、**コミュニティごとに別の個人スペース**を持てる。
- owner 判定と投稿 authorship は別概念（§6.5）。

### 10.2 アクセス制御

**アクセスできる:**
- owner 本人
- 所属コミュニティの管理者
- super_admin

**アクセスできない:**
- 他の一般コミュニティメンバー
- 他コミュニティの管理者
- ゲスト
- URL を知っているだけのユーザー

この制御は**アプリの画面制御だけでなく RLS でも同じ条件を強制**する（§17）。RLS 未適用のまま個人スペースを本番公開しない。

### 10.3 管理者権限（初版）

**可能:**
- 個人スペースの閲覧
- 管理者自身としての投稿（authorship 付き）
- 個人スペース設定の変更
- 個人スペースの停止・再開

**不可:**
- 本人投稿の直接編集
- 本人投稿の直接削除

問題投稿への対応は、将来「非表示・モデレーション」を別権限として追加する。

### 10.4 プライバシー表示

本人に対し、次の趣旨を明示する。

```text
このスペースは、あなたとコミュニティ管理者が閲覧できます。
```

- 利用開始時と、個人スペース内の分かりやすい場所に表示する。
- 初版では、本人が管理者閲覧を OFF にする機能は追加しない。

### 10.5 作成タイミング（自動作成）

> `community_membership` が active になった時点で、個人スペースを自動作成する。

#### 完成条件（整合性保証）

```text
community_membership が active であるにもかかわらず、
対応する個人スペースが存在しない状態を恒常的に残さない。
```

上記を満たすために、**原子的処理または補償処理により最終的な整合性を保証する**。実装方式は次のいずれも許容する。

- 同一トランザクション（membership active 化と個人スペース作成を 1 トランザクションで実行）
- 冪等処理・再試行・欠落検知・repair による補償処理（active だが個人スペースが無い状態を検知して後追い作成）

要件:
- サーバー側処理または DB 関数で**冪等**に作成する。
- 同一コミュニティ・同一 owner に**重複作成しない**（UNIQUE + 冪等処理）。
- **再実行可能**。
- slug 衝突を避ける。

#### slug の確定事項

- 表示名・username・メールアドレスを slug の正本にしない。
- 安定した ID から自動生成する。
- コミュニティ内で一意。
- 改名しても slug は変わらない。
- 初版では本人による slug 変更は不可。

#### slug の候補と未決

```text
候補: personal-{membership_id の短縮値}
```

上記は候補であり、**具体的な形式（短縮アルゴリズム・桁数・接頭辞）は Phase 3 で確定する未決事項**とする（§22）。表示名は slug と分離し、別途変更可能とする。

### 10.6 停止・アーカイブ

- membership が suspended → 個人スペースも `status='suspended'`（アクセス不可・データ保持）。
- membership が removed → 個人スペースはアクセス不可・データ保持（`suspended` 相当、hard delete しない）。
- `archived` は将来の長期保管用途として定義に残す。
- **hard delete は初版対象外。** 投稿データ（`hossiis` / `space_panes`）は保持する。

## 14. ゲスト参加との共存

| 局面 | ゲストの扱い |
|------|--------------|
| public 共有スペース | 現行どおりゲスト参加を維持（URL / QR、ニックネームのみ） |
| invite_only 共有スペース | ゲスト参加不可 |
| 個人スペース | ゲストは owner になれず、アクセス不可 |
| 投稿 Identity | ゲスト投稿は端末 UUID の `author_id`、authorship なし |

### 14.1 authorship がない既存投稿

- authorship が存在する場合は authorship を本人性の正本とする。
- authorship がない過去投稿は、**自動的に本人扱いしない**。
- 安全に本人と確認できる場合のみ、限定的な backfill を検討する。
- 単純な `author_id === auth.uid()` だけで全過去投稿を backfill しない。
- 不確実な投稿は既存表示を維持し、本人編集不可でも**データを壊さないことを優先**する。
- backfill 条件は Phase 1 で実 DB を確認して確定する（§22 未決）。

## 17. RLS・セキュリティ要件

RLS は後付けではなく、個人スペース・invite_only の**完成条件**である。

### 17.1 必須要件

- personal space は owner・所属コミュニティ管理者・super_admin のみ SELECT 可能。
- invite_only は active member・管理者・super_admin のみ SELECT 可能。
- `hossiis` は所属スペースのアクセス条件を継承する。
- `space_nicknames` も同様の条件を考慮する。
- URL や Supabase API 直叩きで回避できない。
- anon による `spaces` 設定 UPDATE を禁止する。
- anon による他人投稿の UPDATE / DELETE を禁止する。
- public space のゲスト投稿体験は必要な範囲だけ維持する。
- RLS 適用前に個人スペースを本番公開しない。
- RLS 適用前に invite_only を本番利用しない。

### 17.2 Critical な既存課題（明記）

現行、次の初期 schema ポリシーが残存しており、匿名クライアントが書き込み可能。

```text
spaces          : public read + public insert/update/delete が残存（Critical）
hossiis         : 全公開 CRUD
space_nicknames : 全公開 CRUD
```

これらを整理しない限り、個人スペース・invite_only の RLS 制御は成立しない。

### 17.3 段階移行戦略（Phase 2.5 として独立）

```text
Step 1. 現状ポリシー棚卸し（spaces / hossiis / space_nicknames の全ポリシー列挙）
Step 2. scope 判定の共通 SQL helper を用意
        例: can_access_space(space_id) — personal/invite_only/public を一元判定
Step 3. dev で先行適用し、public space のゲスト投稿体験を回帰確認
Step 4. anon の spaces UPDATE / 他人 hossiis UPDATE・DELETE を禁止
Step 5. 個人スペース・invite_only の SELECT を helper 条件に束ねる
Step 6. production 適用（feature flag / 段階リリース）
```

- 個人スペース（Phase 3）と invite_only（Phase 5）は、この共通 helper を前提に「RLS 込みで完成」とする。
- demo / 本番の互換性のため、public CRUD 撤去は feature flag と段階適用で行う。

## 21. 受け入れ条件

### 21.1 Phase 1（Identity）

- ゲスト投稿は authorship 行が作成されない。
- ログイン投稿（一般 / 発行参加者 / 管理者）は authorship 行が作成される。
- 公開 `hossiis` REST レスポンス・Realtime payload に Auth UID が現れない。
- 別端末で同一スペース再入室後、本人 authorship 投稿が MyLogs に表示される。
- authorship がある投稿は本人が編集可能（`canEditBubble` が authorship ベース）。
- authorship 欠損の過去投稿は本人編集不可でもデータが壊れない。
- main / development / production で `hossii_authorships` が一致する。
- 管理者に `user_profiles` 行が存在する。

### 21.2 Phase 2（community_memberships）

- 既存管理者が `community_memberships(admin, active)` として存在する。
- 管理者が個人を発行/招待し、`community_memberships` が作られる。
- 最小メンバー一覧が表示される。
- 個人スペースはまだ作られない。

### 21.3 Phase 2.5（RLS 基盤）

- anon が `spaces` 設定を UPDATE できない。
- anon が他人投稿を UPDATE / DELETE できない。
- public space のゲスト参加・投稿が従来どおり動作する。

### 21.4 Phase 3（個人スペース）

- membership active で個人スペースが冪等に自動作成される（重複なし）。

### B.8 race 対策（正式）

- `uid + spaceId` 単位で取得する。
- seq 世代 guard で古い response を破棄する。
- logout / guest / space 切替 / user 切替時に即・空 Set。
- error 時は空 Set ＋ `error`。
- 無限 retry はしない。
- API 結果を Store 格納するときに 1 度だけ Set 化する。

### B.9 StrictMode 対応（正式）

React 18 StrictMode の dev 環境では `setup → cleanup → setup` が発生する。旧実装では cleanup で dispose した controller を再利用し、2 回目以降の `sync` / `refresh` が no-op になっていた（`status=idle` / request 0 件）。

正式なライフサイクル方式:

```text
createAuthorshipControllerHost(factory)

getOrCreate():
  生存 controller を取得、無ければ新規生成

peek():
  現在の controller だけを返す（新規生成しない）

disposeIfCurrent(controller):
  対象を dispose し、現在値と一致する場合のみ null 化
```

- controller の `dispose()` は不可逆のまま維持し、再生成責務は Provider 側の host に分離する（controller へ `isDisposed` / `revive` / `reset` は追加しない）。
- StrictMode の再 setup では新 controller を生成する。
- genuine unmount 後の投稿 refresh では controller を再生成しない（`peek()` 経由）。
- 古い cleanup が新 instance を破棄しない（`disposeIfCurrent` により現在値一致時のみ null 化）。C1 の遅延 response は C2 を上書きしない。
