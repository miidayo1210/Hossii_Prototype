## 1. 背景と目的

### 1.1 背景

現行の `AccountScreen` は、挨拶・アカウント情報・所属コミュニティ・参加スペース・マイスペース・マイHossii・ニックネーム設定を **1つの縦長ページ** に並べている。機能追加に伴いセクションが増え、次の体験上の課題が顕在化している。

- 目的の異なる設定が同一スクロール上に混在し、初見で「何をどこで変えればよいか」が分かりにくい
- ホーム相当の挨拶と、詳細フォームが同じ画面にあり、情報の優先度が平坦化している
- 説明文が長く、見出し・状態・操作だけで伝えられる箇所にも文章が重なる
- カード型セクションの積み重ねにより、視覚的な密度が高い

### 1.2 目的

アカウント画面を次の **4区分** に整理し、利用者が迷わず目的の操作へ到達できる構造にする。

1. **ホーム** — 現在の状態の要約と、各区分への入口
2. **プロフィール** — 表示名・アカウント認証に関する操作
3. **参加先** — コミュニティ・スペース・マイスペースの所属と参加
4. **マイHossii** — Hossii の登録とスペース登場設定

本書は UI・ナビゲーション・情報配置の仕様であり、**実装は行わない**。実装 Todo と段階的な着手順を残し、未確定事項は別章で管理する。

### 1.3 非目的（本仕様）

- アカウントマージ（110）
- membership 自動整合（115）
- グローバルナビ（`TopRightMenu` / `BottomNavBar`）の `#account` 以外の変更
- 新規認証方式・新規 DB テーブルの追加

## 2. 現在の問題

### 2.1 情報構造

| 問題 | 現状 | 影響 |
|------|------|------|
| 役割の混在 | 状態確認・所属管理・キャラ設定・ニックネーム編集が同一ページ | 目的別の操作に到達するまでの認知負荷が高い |
| 縦長スクロール | セクション数 7 以上を順に並べるのみ | 下位セクション（ニックネーム等）の発見性が低い |
| 重複表示 | 表示名・ログイン状態・コミュニティが複数セクションで再提示されうる | 同じ情報を何度も読ませる |
| 説明過多 | 各セクションに `sectionDesc` が常設 | 見出しと状態で足りる箇所まで文章が長い |
| 視覚密度 | セクションカードの連続 | 1画面に情報を詰め込みすぎた印象 |

### 2.2 現行セクション一覧（`AccountScreen.tsx`）

| 順序 | セクション名 | 主な内容 |
|------|-------------|----------|
| — | identityBanner | 挨拶・ログイン状態ラベル |
| 1 | アカウント情報 | 表示名・ログアウト / ゲスト時ログイン・新規登録 |
| 2 | 所属コミュニティ | `CommunitySwitcher`（ログイン時のみ） |
| 3 | 参加しているスペース | `JoinedSpacesSection` |
| 4 | コミュニティ内のマイスペース | `CommunityPersonalSpacesSection` |
| 5 | マイHossii | `MyHossiiSettingsSection` |
| 6 | このスペースでのニックネーム | スペース別ニックネーム入力 |
| 7 | デフォルトニックネーム | 共通ニックネーム入力 |

区分再編後は、上記を **1画面1目的** の配置へ **移管** する（**削除ではない**）。

> **確定:** 現行セクションで提供している操作・API・データ取得は、いずれかの区分から引き続き実行できること。UI 再配置のみを行い、機能の縮小や廃止は本仕様の範囲に含めない。

## 3. デザイン方針

| 方針 | 適用内容 |
|------|----------|
| スッキリしていて迷わない | 4区分の役割を固定し、グローバルナビから入った直後はホームを表示する |
| 1画面に詰め込みすぎない | 詳細フォームはプロフィール・参加先・マイHossii に限定。ホームは要約のみ |
| ぷくっとした丸みのある UI | 区分カード・入口ボタン・入力欄に大きめの `border-radius` を適用（Hossii 既存トーンに合わせる） |
| 紫〜ピンクはアクセント | 主操作・選択中区分・アクセントラインに限定。背景・本文はニュートラル |
| 説明文より見出し・状態・操作 | ホームの要約行は「ラベル + 値 + 矢印」で構成。長文 `sectionDesc` は区分内の必要箇所のみ |
| 子どもっぽくしすぎない | 挨拶は短く温かみを残すが、装飾過多・絵文字乱用・過度な擬音は避ける |
| カードの中にカードを重ねすぎない | 区分内はフラットなリストまたは1段のカード。入れ子カードは原則禁止 |

## 4. 情報構造

### 4.1 全体ツリー

```text
#account（アカウント画面）
├── home（ホーム）          … 既定表示
├── profile（プロフィール）
├── spaces（参加先）
└── my-hossii（マイHossii）
```

### 4.2 URL / 状態の持ち方

**確定（Phase 1）:** 区分の表示切替は `AccountScreen` 内で行う。`useRouter` は **拡張しない**（現行の `parseHash` が `#account/{区分}` を既に `screen` + `screenParam` として解釈できるため）。

| 区分 | ハッシュ例 | `screenParam` | 備考 |
|------|-----------|---------------|------|
| ホーム | `#account` | 未指定 | 既定表示 |
| プロフィール | `#account/profile` | `profile` | |
| 参加先 | `#account/spaces` | `spaces` | |
| マイHossii | `#account/my-hossii` | `my-hossii` | |

- グローバルナビの「アカウント」押下時は **常に `#account`（ホーム）** を開く（直前の区分は記憶しない）
- 区分切替は `navigate('account', param)` で hash を更新する（**暫定:** Phase 1 はこれで十分）

**将来（Phase 2 以降・任意）:** ブラウザ戻る/進むでの区分履歴、`#account/home` の明示 URL 正規化。Phase 1 の完了条件には含めない。

### 4.3 ホームの情報重複禁止

ホームに表示するのは **要約行のみ** とする。詳細フォーム・一覧・編集 UI は各区分にのみ置き、ホームでは同項目の編集や全文一覧を **表示しない**。

| ホーム要約（画面上部・1か所のみ） | 詳細の所在 |
|--------------------------------|-----------|
| 表示名（1行） | プロフィール |
| ログイン状態（1行） | プロフィール |
| 選択中コミュニティ名（1行、未所属時は状態のみ） | 参加先 |

入口カードには **同じ値を再掲しない**。区分名と操作を示す短いラベルのみ（例: 「表示名・ニックネームを編集」「参加先を見る」「Hossiiを設定」）。件数・登録状態などの数値・状態は、Phase 2（T8）で入口カードに付与してよいが、画面上部要約と **同一行の重複は禁止**。

## 5. 4区分の役割

### 5.1 ホーム

**役割:** 利用者の現在状態を短く把握し、他区分へ進む入口を提供する。

**含めるもの:**

| ブロック | 内容 |
|----------|------|
| 挨拶 | `resolveAccountIdentity` の `greeting`（既存ロジック維持） |
| 状態要約 | 表示名・ログイン状態ラベル（`statusLabel`）を1ブロックに集約 |
| コミュニティ要約 | 選択中コミュニティ名。未ログイン・未所属時は「未選択」等の状態表示 |
| 区分入口 | プロフィール / 参加先 / マイHossii へのナビゲーションカード（各1行要約 + 遷移） |

**含めないもの:**

- ニックネーム入力フォーム
- 参加スペース一覧・マイスペース一覧
- マイHossii プリセット選択・画像アップロード
- ログアウトボタン（プロフィールへ誘導）
- コミュニティ切替 UI 本体（参加先へ誘導）

**入口カード（Phase 1 — 暫定）:**

| 入口 | カードに含めるもの | 含めないもの |
|------|-------------------|-------------|
| プロフィール | ラベル「プロフィール」+ 補助文「表示名・ニックネーム」 | 表示名の再掲、入力フォーム |
| 参加先 | ラベル「参加先」+ 補助文「コミュニティとスペース」 | スペース一覧、切替 UI |
| マイHossii | ラベル「マイHossii」+ 補助文「Hossiiの登録と登場」 | プリセット選択、画像アップロード |

**入口カード（Phase 2 — 暫定）:** 上記に加え、件数・登録状態など **1語の補助バッジ** を付与してよい（画面上部要約との同一情報の重複は禁止）。

### 5.2 プロフィール

**役割:** 利用者としての表示名と、アカウント認証に関する操作をまとめる。

**含めるもの:**

| ブロック | 現行セクション / コンポーネント |
|----------|-------------------------------|
| アカウント情報 | 現行「アカウント情報」— 表示名・アバター・ログイン種別（**現行:** メールアドレスは画面に表示しない） |
| デフォルトニックネーム | 現行「デフォルトニックネーム」— `setDefaultNickname` |
| スペース別ニックネーム | 現行「このスペースでのニックネーム」— `setSpaceNickname`（アクティブスペース文脈） |
| 認証操作 | ログイン・新規登録（ゲスト時）・ログアウト（ログイン時） |

**補足:**

- ゲスト時のログイン / 新規登録導線は本区分に集約（43 のフロー維持）
- `ACCOUNT_AUTH_COMING_SOON` 時の Coming soon 表示も本区分で維持
- ログアウト確認ダイアログ（`window.confirm`）は現行どおり

### 5.3 参加先

**役割:** コミュニティ所属とスペース参加の管理。

**含めるもの:**

| ブロック | 現行セクション / コンポーネント |
|----------|-------------------------------|
| 所属・参加コミュニティ | 現行「所属コミュニティ」— `CommunitySwitcher` |
| 参加しているスペース | 現行「参加しているスペース」— `JoinedSpacesSection` |
| コミュニティ内のマイスペース | 現行「コミュニティ内のマイスペース」— `CommunityPersonalSpacesSection` |

**補足:**

- 未ログイン時はログイン案内を表示（`JoinedSpacesSection` / `CommunityPersonalSpacesSection` の現行挙動維持）
- スペースを開く導線は正規 URL（`/c/{community}/s/{slug}#screen`）を維持
- `JoinedSpacesSection` 内のニックネームインライン編集は参加先に残す（スペース文脈の一覧操作のため）。プロフィールの「アクティブスペース別ニックネーム」と役割が重なる場合は **未確定事項** で整理する

### 5.4 マイHossii

**役割:** ログインアカウントに紐づく Hossii の登録と、スペースごとの登場設定。

**含めるもの:**

| ブロック | 現行セクション / コンポーネント |
|----------|-------------------------------|
| Hossii登録 | `MyHossiiSettingsSection` の登録フロー |
| プリセット・アップロード・カスタム | 同セクション内の preset / upload / custom 選択 |
| スペースごとの登場設定 | 同セクション内のスペース別表示 ON/OFF・参加資格表示 |

**補足:**

- `activeSpaceId` / `spaceMyHossiiEnabled` 等の既存 Props 連携を維持
- 103 の Phase 1〜5 実装済み範囲を壊さない
- 対象スペースの選択 UI（アクティブスペース固定か切替可能か）は **未確定事項**

## 7. PCとスマホのナビゲーション方針

### 7.1 共通原則

- グローバルナビ（06 準拠）の「アカウント」は `#account`（ホーム）へ遷移する点は変更しない
- アカウント画面内に **区分ナビ** を設け、4区分を切り替える
- 区分ナビの最終 UI 形式（左ナビ / 上部タブ / モバイル専用ナビ）は **未確定**（15章参照）
- ブレークポイントは既存どおり **769px** を境界とする

### 7.2 PC（769px 以上）— 候補

| 候補 | 概要 | メリット |
|------|------|----------|
| A. 左サイドナビ | 画面左に縦並び4項目、右に区分コンテンツ | 区分が常時見え、情報量の多い参加先にも向く |
| B. 上部セグメントタブ | ヘッダー下に横並び4タブ | 実装が軽く、1画面の縦幅を抑えやすい |

**暫定推奨:** 実装 Phase 1 では **B（上部セグメントタブ）** を先行し、参加先の情報量を見て A へ切替検討可能とする。最終決定は未確定。

### 7.3 スマホ（768px 以下）— 候補

| 候補 | 概要 | メリット |
|------|------|----------|
| A. 上部セグメントタブ（横スクロール可） | PC と同型をコンパクト化 | 実装共通化しやすい |
| B. 画面内リスト入口（ホームのみ）+ 下部サブナビ | ホームから各区分へ。区分表示中は下部に4タブ | グローバル BottomNav と役割分担しやすい |
| C. ホーム入口カード + 区分内戻る | ホームの3入口カードから遷移。区分内は戻るでホーム | 最もシンプル。区分間横断はやや手数 |

**暫定推奨:** Phase 1 では **A** を先行（PC との共通化）。BottomNav との干渉（safe area）は 06 / 56 の余白規則に従う。

### 7.4 区分ナビとグローバルナビの関係

```text
[グローバル] スペース | 投稿 | ログ | アカウント
                              ↓
[アカウント内] ホーム | プロフィール | 参加先 | マイHossii
                              ↓
                    区分コンテンツ（1区分のみ表示）
```

## 8. 表示状態

### 8.1 ログイン状態別

| 状態 | ホーム | プロフィール | 参加先 | マイHossii |
|------|--------|-------------|--------|------------|
| ゲスト | 挨拶・ゲスト状態・入口3件 | ログイン・新規登録導線。**現行どおり** デフォルト/スペース別ニックネーム編集も表示 | ログイン案内（一覧は非表示） | ログイン案内 |

| ログイン（参加ID） | 状態ラベル「参加IDでログイン中」 | 同上 | 同上 | 同上 |

### 8.2 データ状態別（区分内）

| 条件 | 表示 |
|------|------|
| コミュニティ未所属 | 参加先: 所属コミュニティなしの状態表示。ホーム要約は「未所属」 |
| 参加スペース 0 件 | 参加先: 空状態 UI。ホーム要約は「0件」 |
| マイHossii 未登録 | マイHossii: 登録 UI。ホーム要約は「未登録」 |
| マイHossii 登録済み・スペース非表示 | マイHossii: 登場設定 UI。ホーム要約は「登録済み（非表示）」等 |
| 読み込み中 | 各 Section の loading 状態を区分内に閉じる。ホーム要約はスケルトンまたは「読み込み中」 |
| 取得エラー | 区分内にエラーと再試行。ホーム要約は「取得できませんでした」 |

### 8.3 アクティブ区分の視覚

- 選択中区分は紫〜ピンク系のアクセント（下線・背景・ラベル色のいずれか1つ）
- 非選択区分はニュートラル
- ホームは区分ナビ上で「ホーム」として明示的に選択可能とする
