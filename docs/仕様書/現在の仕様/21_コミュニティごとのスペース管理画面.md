## 概要

コミュニティ管理者は自分のコミュニティのスペース管理画面のみを操作できる。  
Hossii スーパー管理者（運営）は全コミュニティを一覧表示し、各コミュニティのスペース管理画面へ遷移できる。

## 修正 A: コミュニティ名をスペース管理画面に表示

> **実装ステータス:** ✅ 実装済み（2026-03-06）

### 概要

`SpacesScreen` のヘッダーに「`{コミュニティ名}` のスペース管理」と表示し、ログイン中のコミュニティ管理者が自分のコミュニティのスペースを管理していることを明示する。

### UI 変更

```
【変更前】
✨ Hossii   スペース管理              👤 アカウント

【変更後】
✨ Hossii   {communityName} のスペース管理   👤 アカウント
```

- `currentUser.communityName` を使用する
- `communityName` が取得できない場合は「スペース管理」にフォールバック
- スーパー管理者が他コミュニティを管理中の場合は `overrideCommunityName` を優先表示

### 実装

```ts
// SpacesScreen.tsx
const pageTitle = overrideCommunityName
  ? `${overrideCommunityName} のスペース管理`
  : currentUser?.communityName
    ? `${currentUser.communityName} のスペース管理`
    : 'スペース管理';
```

## 修正 B: スーパー管理者向けコミュニティ一覧画面

> **実装ステータス:** ✅ 実装済み（2026-03-06）

### 概要

Hossii 運営（スーパー管理者）がログインすると、全コミュニティの一覧が表示される。コミュニティをクリックすると、そのコミュニティのスペース管理画面（`SpacesScreen`）に遷移する。

### スーパー管理者の定義

Supabase の `auth.users` テーブルで `app_metadata.role = "super_admin"` のユーザー。  
`AppUser` の `isSuperAdmin` フラグで判定する。

**設定方法（Supabase Dashboard > SQL Editor）:**

```sql
UPDATE auth.users
SET raw_app_meta_data = raw_app_meta_data || '{"role": "super_admin"}'::jsonb
WHERE email IN ('運営メールアドレス@example.com');
```

### ナビゲーションフロー

```
スーパー管理者ログイン
  → CommunitiesScreen（コミュニティ一覧）
      ├── コミュニティカード一覧（全コミュニティ）
      │     ├── コミュニティ名
      │     ├── ステータスバッジ（pending / approved / rejected）
      │     ├── 登録日
      │     └── [スペースを管理] ボタン
      └── コミュニティをクリック
            → SpacesScreen（選択したコミュニティの communityId でフィルタリング）
                  └── 戻るボタン → CommunitiesScreen へ戻る

コミュニティ管理者ログイン
  → SpacesScreen（自分の communityId でフィルタリング、従来通り）
```

### 実装ルート

| ルート名 | 画面 | 説明 |
|---|---|---|
| `communities` | `CommunitiesScreen` | スーパー管理者専用のコミュニティ一覧 |

### CommunitiesScreen の UI

| 要素 | 説明 |
|---|---|
| ヘッダー | 「✨ Hossii  コミュニティ管理」 + アカウントドロップダウン |
| コミュニティカード | コミュニティ名・ステータス・登録日時 |
| ステータスバッジ | `pending`: 🟡 審査中 / `approved`: 🟢 承認済み / `rejected`: 🔴 却下 |
| スペースを管理ボタン | クリックで当該コミュニティの `SpacesScreen` へ遷移 |
| 空の状態 | 「登録済みコミュニティはありません」 |

### overrideCommunityId の受け渡し

`AdminNavigationContext`（`src/core/contexts/AdminNavigationContext.tsx`）を使って `overrideCommunityId` と `overrideCommunityName` を管理する。  
`HossiiProvider` がこの Context を参照し、スペース取得の `communityId` として使用する。

### `SpacesScreen` の「戻るボタン」制御

| ユーザー種別 | 「戻る」の遷移先 |
|---|---|
| コミュニティ管理者 | なし（`SpacesScreen` がトップ） |
| スーパー管理者 | `CommunitiesScreen` へ戻る |

### コミュニティ承認の現状運用

現状、コミュニティの承認は Supabase Dashboard の SQL Editor から手動で行う：

```sql
UPDATE communities SET status = 'approved' WHERE name = 'コミュニティ名';
```

→ 承認 UI とメール通知は **修正 C** として別途実装予定。

## バグ修正: 新コミュニティで全スペースが表示される問題

> **修正日:** 2026-03-06

### 問題

新しくコミュニティを登録してログインすると、そのコミュニティのスペースではなく、他コミュニティのスペースが表示されていた。

### 原因

`useHossiiStore.tsx` の `fetchSpaces` 呼び出し箇所に以下の条件があった：

```tsx
// 修正前（バグあり）
if (supabaseSpaces.length > 0) {
  dispatch({ type: 'SET_SPACES', payload: supabaseSpaces });
}
```

新コミュニティはスペースが 0 件のため、この条件が false になり `dispatch` がスキップされた。  
結果として `state.spaces` が更新されず、ブラウザの localStorage に残っていた別コミュニティのスペースがそのまま表示されていた。

### 修正

```tsx
// 修正後
dispatch({ type: 'SET_SPACES', payload: supabaseSpaces });
```

Supabase に接続している場合は、空配列でも必ず DB の結果で上書きするようにした。

## 修正 C: コミュニティ申請通知・承認 UI（未実装・仕様メモ）

> **実装ステータス:** ⬜ 未実装
> **優先度:** 中（現状は Supabase Dashboard の SQL Editor で手動承認）

### 背景・現状の課題

コミュニティ管理者がサインアップすると `communities.status = 'pending'` で登録される。  
現状、Hossii 運営（スーパー管理者）への通知手段がなく、承認は Supabase Dashboard から手動で SQL を実行する必要がある。

```sql
-- 現状の手動承認方法（Supabase Dashboard > SQL Editor）
UPDATE communities SET status = 'approved' WHERE name = 'コミュニティ名';
```

### C-1: 申請時の運営への通知

新規コミュニティ申請（`adminSignUp`）が完了したタイミングで、Hossii 運営にメール通知する。

| 項目 | 内容 |
|---|---|
| トリガー | `adminSignUp` 成功時（`communities` テーブルへの INSERT 後） |
| 通知先 | Hossii 運営の固定メールアドレス（環境変数で管理） |
| 通知内容 | コミュニティ名・申請者メールアドレス・申請日時 |
| 実装方式候補 | Supabase Edge Function（DB trigger または API 呼び出し）+ Resend などのメール送信 API |

### C-2: `CommunitiesScreen` での承認・却下 UI

スーパー管理者が `CommunitiesScreen` 上で直接ステータスを変更できるようにする。

| 要素 | 説明 |
|---|---|
| 承認ボタン | `pending` のコミュニティカードに表示。クリックで `status = 'approved'` に更新 |
| 却下ボタン | `pending` のコミュニティカードに表示。クリックで `status = 'rejected'` に更新 |
| 確認ダイアログ | 誤操作防止のため `window.confirm` または専用モーダル |
