## 概要

スペース画面に表示される吹き出し（`Bubble` コンポーネント）の以下 3 点の仕様を定義する。

| # | 課題 | ゴール |
|---|---|---|
| L | タグ・いいねがコンテンツを圧迫する | コメント・画像を最大化し、タグ・いいねは邪魔にならない表示にする |
| L2 | フッター帯の区切り線とコンテンツが被る | 区切り線の下に十分な余白を確保する |
| I | 吹き出し内の画像をタップしても何も起きない | クリックで全画面ライトボックス表示・ホバーでポップアニメーション |
| P | リロード・編集後に座標がリセットされる | 一度確定した座標はあらゆる状況で維持する |
| R | コーナーハンドルの拡大縮小方向が逆になる | 外側ドラッグ → 拡大、内側ドラッグ → 縮小に統一する |

---

## L: タグ・いいねのレイアウト

### 現状の問題

`Bubble` 内の `bubbleContent` エリアに、テキスト・画像・ハッシュタグ・いいねボタンがすべて縦積みで並んでいる。ハッシュタグが多い場合やいいね数が多い場合に吹き出しが縦に伸び、テキストや画像の表示領域が相対的に小さくなる。

### 設計方針

- **メインエリア**（テキスト・画像）を最優先で表示する。
- **タグ・いいね**は「フッター帯」として吹き出しの最下部に固定する。文字サイズを小さくし、高さを最小化する。
- フッター帯は最大 1 行に収める。はみ出す場合は省略（横スクロール or `…`）。

### UI 仕様

#### L01: フッター帯レイアウト

```
┌─────────────────────────────┐
│ 😊 投稿者名 · 5分前          │  ← メタ行（変更なし）
│                             │
│ テキスト（最大2行）          │  ← メインエリア（優先）
│ [画像サムネイル]             │
│                             │
├─────────────────────────────┤
│ #タグ1 #タグ2  ❤️3          │  ← フッター帯（新規）
└─────────────────────────────┘
```

- タグ列といいねボタンを同一行に横並びで配置する（flexbox `justify-content: space-between`）。
- タグ列: `font-size: 10px`、`opacity: 0.7`、最大 `3` 件まで表示。超過分は `+N` で省略。
- いいねボタン: `font-size: 11px`。`likes_enabled` が OFF の場合は非表示（変更なし）。
- フッター帯全体の高さ上限: `22px`。

#### L02: フッター帯の非表示条件

- `viewMode === 'bubble'`（コンパクト表示モード）の場合はフッター帯を非表示にする。
- タグが 0 件かついいねが無効な場合もフッター帯を非表示にする。

#### L03: CSS モジュール変更箇所

| クラス名 | 変更内容 |
|---|---|
| `.bubbleHashtags` | `margin-top` を削除し、フッター帯に移動。`flex-wrap: nowrap; overflow: hidden` を追加 |
| `.bubbleHashtag` | `font-size: 10px` に縮小 |
| `.likeButton` | `margin-top` を削除し、フッター帯に移動 |
| `.bubbleFooter`（新規追加） | フッター帯のコンテナ。`display: flex; align-items: center; justify-content: space-between; height: 22px; padding: 0 4px; border-top: 1px solid rgba(0,0,0,0.08)` |

## L2: フッター帯の余白修正

### 現状の問題

`bubbleFooter` の `border-top` 区切り線とタグ・いいねボタンのテキストが縦方向に近接しすぎており視覚的に被って見える。

### 仕様

#### L2-01: `bubbleFooter` の padding-top を追加

`.bubbleFooter` の `padding` を `0 4px` から `4px 4px 0` に変更し、区切り線からコンテンツまでの上余白を確保する。

```css
.bubbleFooter {
  padding: 4px 4px 0;  /* 上: 4px, 左右: 4px, 下: 0 */
  height: auto;        /* 固定 22px を廃止し、padding に委ねる */
  min-height: 22px;
}
```

#### L2-02: 高さ制約の緩和

固定 `height: 22px` を `min-height: 22px` に変更することで、フォントサイズや padding の増減に対して柔軟に対応できるようにする。

## I: 吹き出し内画像のインタラクション

### 現状の問題

吹き出し（`Bubble`）内に投稿画像（`<img className={styles.bubbleImage}>`）が表示されるが、クリックしても何も起きない。画像は `max-height: 150px` で縮小表示されているため、内容を確認しにくい。

### 設計方針

- **ライトボックス（全画面表示）**: 画像クリックでオーバーレイを表示し、元のサイズで画像を見られるようにする。
- **ホバーアニメーション**: マウスオーバーで画像がポコッと膨らむ（scale up）エフェクトを付ける。
- ライトボックスはバブルのドラッグ操作と競合しないよう `stopPropagation` で伝播を止める。

### UI 仕様

#### I01: ホバーエフェクト

```css
.bubbleImage {
  cursor: zoom-in;
  transition: transform 0.18s ease, box-shadow 0.18s ease;
}

.bubbleImage:hover {
  transform: scale(1.06);
  box-shadow: 0 6px 20px rgba(0, 0, 0, 0.18);
  z-index: 1;
}
```

- スケール倍率: `1.06`（ボコっと感を出しつつ周囲を圧迫しない程度）
- トランジション: `0.18s ease`（素早くポップ感を出す）
- カーソル: `zoom-in`（拡大できると伝える）

#### I02: ライトボックス

クリック時にフルスクリーンオーバーレイを表示する。実装は **インラインステート** で行い、外部ライブラリは使わない。

```
┌─────────────────────────────────────────────────────┐
│  （黒半透明オーバーレイ  rgba(0,0,0,0.85)）          │
│                                                     │
│              ┌─────────────┐                        │
│              │   [画像]    │  max-width: 90vw       │
│              │             │  max-height: 85vh      │
│              │             │  object-fit: contain   │
│              └─────────────┘                        │
│                                                     │
│  ✕  （右上 or オーバーレイクリックで閉じる）         │
└─────────────────────────────────────────────────────┘
```

- オーバーレイ: `position: fixed; inset: 0; z-index: 1000; background: rgba(0,0,0,0.85)`
- 画像: `max-width: 90vw; max-height: 85vh; object-fit: contain; border-radius: 8px`
- 閉じる操作: オーバーレイ背景クリック、または Escape キー

#### I03: 実装箇所

`Tree.tsx` の `Bubble` コンポーネント内にライトボックス用のローカル state を追加する。

```tsx
const [lightboxSrc, setLightboxSrc] = useState<string | null>(null);

// 画像要素
<img
  src={hossii.imageUrl}
  alt="投稿画像"
  className={styles.bubbleImage}
  loading="lazy"
  onPointerDown={(e) => e.stopPropagation()}
  onClick={(e) => {
    e.stopPropagation();
    setLightboxSrc(hossii.imageUrl!);
  }}
/>

// ライトボックス（バブル div の外 → Portal or 末尾に追加）
{lightboxSrc && (
  <div
    className={styles.lightboxOverlay}
    onClick={() => setLightboxSrc(null)}
  >
    <img
      src={lightboxSrc}
      alt="拡大表示"
      className={styles.lightboxImage}
      onClick={(e) => e.stopPropagation()}
    />
    <button className={styles.lightboxClose} onClick={() => setLightboxSrc(null)}>✕</button>
  </div>
)}
```

> ライトボックスは `position: fixed` で `z-index: 1000` のため、バブルの `position: absolute` より前面に出る。ただし Bubble の `div` 内に置くと `overflow: hidden` や `transform` の影響を受ける可能性があるため、**`document.body` への Portal（`createPortal`）** で描画することを推奨する。

#### I04: CSS 追加箇所（`SpaceScreen.module.css`）

| クラス名 | 内容 |
|---|---|
| `.lightboxOverlay` | `position: fixed; inset: 0; z-index: 1000; display: flex; align-items: center; justify-content: center; background: rgba(0,0,0,0.85)` |
| `.lightboxImage` | `max-width: 90vw; max-height: 85vh; object-fit: contain; border-radius: 8px; box-shadow: 0 8px 40px rgba(0,0,0,0.5)` |
| `.lightboxClose` | 右上固定の閉じるボタン。`position: absolute; top: 16px; right: 16px; font-size: 24px; color: white; background: none; border: none; cursor: pointer` |
| `.bubbleImage`（更新） | `cursor: zoom-in; transition: transform 0.18s ease, box-shadow 0.18s ease` を追加 |
| `.bubbleImage:hover`（新規） | `transform: scale(1.06); box-shadow: 0 6px 20px rgba(0,0,0,0.18)` |

## P: 座標の永続化

### 現状の問題

1. **新規投稿時**: `ADD_HOSSII` アクションでは `isPositionFixed` が設定されないため、初期位置は `index` から算出したシード座標になる。`displayHossiis` の並び順が変わる（新規投稿が追加される等）と `index` がずれ、他の吹き出しの計算上の座標も変わる。
2. **Supabase リロード時**: `SET_HOSSIIS_FROM_SUPABASE` で全件上書きされる。Supabase への `updateHossiiPosition` 書き込みが失敗した場合、ローカルで確定した座標がリセットされる。
3. **編集後リロード**: Supabase が正常に更新されていても、ブラウザリロード直後に Supabase からの fetch が完了する前に `createBubblePosition(index)` で暫定位置を描画し、fetch 完了後に正しい位置に再描画する（瞬間ズレ）。

### 設計方針

「**座標を一度でも保存したら絶対に動かさない**」を原則とする。

- 投稿直後から座標を確定（`isPositionFixed: true`）にする。
- index ベースの計算座標は投稿時の初期値にのみ使い、以降は使わない。
- Supabase への書き込みが失敗した場合は再試行し、ローカル state を正とする。

### 仕様

#### P01: 投稿時に初期座標を確定する

新規投稿（`ADD_HOSSII` / `ADD_HOSSII_FULL`）時に、`isPositionFixed: true` と初期座標（`positionX`, `positionY`）を同時に設定する。

- 初期座標の算出: 現在の `createBubblePosition(newIndex)` ロジックを流用する。  
  `newIndex` = 新規投稿後の `hossiis` 配列の末尾インデックス。
- Supabase が有効な場合、INSERT 時に `position_x`, `position_y`, `is_position_fixed: true` も書き込む。

```ts
// useHossiiStore.tsx の ADD_HOSSII ケースに追加
const newIndex = state.hossiis.length;
const initialPos = createBubblePosition(newIndex);
const newHossii: Hossii = {
  // ... 既存フィールド ...
  positionX: initialPos.x,
  positionY: initialPos.y,
  isPositionFixed: true,
};
```

> `createBubblePosition` は現在 `SpaceScreen.tsx` に定義されているため、`src/core/utils/bubblePosition.ts` に切り出してストア・コンポーネント双方から参照できるようにする。

#### P02: Supabase 書き込みの失敗リトライ

`updateHossiiPosition` は現在 fire-and-forget で失敗を握りつぶしている。エラー時は 1 回だけリトライし、それでも失敗した場合はコンソールに警告を出す（ユーザー通知は不要）。

```ts
// hossiisApi.ts
export async function updateHossiiPosition(id, x, y) {
  const update = () =>
    supabase.from('hossiis')
      .update({ position_x: x, position_y: y, is_position_fixed: true })
      .eq('id', id);

  const { error } = await update();
  if (error) {
    const { error: retryError } = await update();
    if (retryError) {
      console.warn('[hossiisApi] updateHossiiPosition retry failed:', retryError.message);
    }
  }
}
```

#### P03: SpaceScreen の位置フォールバックを撤廃

P01 が実装されると、すべての Hossii が `isPositionFixed: true` を持つため、`bubblePositions[index]` へのフォールバックは不要になる。実装完了後に削除する。

```ts
// 現在（削除対象）
const pos = hossii.isPositionFixed && hossii.positionX != null && hossii.positionY != null
  ? { x: hossii.positionX, y: hossii.positionY }
  : bubblePositions[index];

// P01 完了後（フォールバックなし）
const pos = { x: hossii.positionX ?? 50, y: hossii.positionY ?? 50 };
```

#### P04: 移行対応（既存データ）

`isPositionFixed: false` の既存 Hossii はページロード時に `bubblePositions[index]` で暫定表示される。表示後、ユーザーがドラッグ・リサイズした時点で座標が確定される（従来どおり）。  
ただし自動マイグレーションは行わない（操作なしのデータはドラッグまで動く可能性がある）。

## R: リサイズハンドルの方向修正

### 現状の問題

現在のリサイズロジック（`Tree.tsx`）は、どのコーナーから操作しても同じデルタ計算を使用している。

```ts
// 現状（問題あり）
const dX = e.clientX - startPX;        // 右 → 正
const dY = startPY - e.clientY;        // 上 → 正
const delta = (dX + dY) / 2;
```

右下（BR）ハンドルを右下に引っ張った場合: `dX` は正（拡大方向）だが、`dY` は下方向なので負（縮小方向）になり、意図と逆になる。

### 設計方針

**各コーナーから「外側」へ引っ張ると拡大、「内側」へ引っ張ると縮小**になるよう、コーナーごとに符号を定義する。

| ハンドル | 外側方向 | dX の符号 | dY の符号 |
|---|---|---|---|
| TL（左上） | ← ↑ | 負 → 拡大: `-dX` | 負（上）→ 拡大: `-dY` |
| TR（右上） | → ↑ | 正 → 拡大: `+dX` | 負（上）→ 拡大: `-dY` |
| BL（左下） | ← ↓ | 負 → 拡大: `-dX` | 正（下）→ 拡大: `+dY` |
| BR（右下） | → ↓ | 正 → 拡大: `+dX` | 正（下）→ 拡大: `+dY` |

ここで `dY` はブラウザ座標系（下が正）を使う（現在は符号が反転していたのを修正）。

### 仕様

#### R01: ドラッグ状態にコーナー情報を追加

```ts
// Tree.tsx の dragStateRef 型を拡張
type ResizeCorner = 'TL' | 'TR' | 'BL' | 'BR';

dragStateRef.current = {
  mode: 'resizing',
  corner: getCornerFromTarget(target), // 'TL' | 'TR' | 'BL' | 'BR'
  // ...
};
```

`getCornerFromTarget` はターゲット要素の `data-resize-handle` 属性値からコーナーを識別する。

#### R02: コーナーごとのデルタ計算

```ts
// onPointerMove の resizing ブランチ
const dX = e.clientX - startPX;   // 右が正
const dY = e.clientY - startPY;   // 下が正（修正後）

let signX = 1;
let signY = 1;
if (corner === 'TL' || corner === 'BL') signX = -1;  // 左に引くと拡大
if (corner === 'TL' || corner === 'TR') signY = -1;  // 上に引くと拡大

const delta = (signX * dX + signY * dY) / 2;
const newScale = Math.max(0.5, Math.min(2.5, startScale + delta * 0.006));
```

#### R03: data-resize-handle 属性の変更

各ハンドル要素にコーナー識別子を付与する。

```tsx
// 現在
<div className={`${styles.resizeHandle} ${styles.resizeHandleTL}`} data-resize-handle />

// 変更後
<div className={`${styles.resizeHandle} ${styles.resizeHandleTL}`} data-resize-handle="TL" />
<div className={`${styles.resizeHandle} ${styles.resizeHandleTR}`} data-resize-handle="TR" />
<div className={`${styles.resizeHandle} ${styles.resizeHandleBL}`} data-resize-handle="BL" />
<div className={`${styles.resizeHandle} ${styles.resizeHandleBR}`} data-resize-handle="BR" />
```

`getCornerFromTarget` の実装:

```ts
function getCornerFromTarget(target: HTMLElement): ResizeCorner {
  const handle = target.closest('[data-resize-handle]') as HTMLElement | null;
  return (handle?.dataset.resizeHandle as ResizeCorner) ?? 'BR';
}
```
