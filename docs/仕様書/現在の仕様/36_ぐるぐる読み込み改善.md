# スペース URL 直アクセス時のぐるぐる読み込み改善

## 問題の概要

スペースの URL に直接アクセスしたとき、または設定画面を開いたときに
画面が白いまま長時間ローディング状態になる現象が発生していた。
リロードすると表示される場合もあった。

## 原因

Vercel ではなく、Supabase へのリクエストが直列に複数回走るウォーターフォールが原因。

### ゲストユーザーの場合

```
ページ読み込み
  ↓
fetchSpaces で全スペースを一括取得（Supabase）
  ↓
spacesLoadedFromSupabase = true になってから URL スラッグと照合
  ↓
GuestEntryScreen に遷移
```

全スペース取得が完了するまで白い画面を表示していた。

### 管理者ユーザーの場合

```
ページ読み込み
  ↓
① auth.onAuthStateChange でセッション確認（Supabase）
  ↓
② getAdminCommunity で communityId を取得（Supabase DB）
  ↓
③ communityId = undefined で fetchSpaces が1回目（無駄なリクエスト）
  ↓
④ communityId が確定して fetchSpaces が2回目
  ↓
スペース一覧が揃ってから URL スラッグと照合
```

Supabase への DB クエリが計3回、しかも直列で実行されていた。

## 修正内容

### 1. fetchSpaces の発火タイミングを修正（useHossiiStore.tsx）

`isResolvingAuth` が `false` になるまで `fetchSpaces` を発火しないよう変更。

- ゲストユーザー：auth 解決は即時なので遅延はほぼなし
- 管理者ユーザー：`communityId` が確定してから1回だけ `fetchSpaces` が実行される

これにより管理者の二重リクエストを排除。

### 2. URL スラッグ ファストパスを追加（App.tsx）

URL にスペーススラッグが含まれている場合、`fetchSpaces`（全件取得）の完了を
待たずに `fetchSpaceByUrl` で対象スペース1件だけをピンポイント取得し、
取得したスペースをローカル state に注入する処理を追加。

```
ページ読み込み（マウント時）
  ↓
fetchSpaceByUrl でスラッグに対応する1件だけ取得（並列実行）
  ↓
取得完了次第 addSpaceLocal でローカル state に注入
  ↓
URL 解決エフェクトがスペースを検出して即座に遷移
```

`fetchSpaces`（全件取得）は引き続きバックグラウンドで実行され、
完了後に state を正式データで上書きする。

### 3. addSpaceLocal メソッドを追加（useHossiiStore.tsx）

Supabase から取得済みのスペースをローカル state にのみ追加するメソッド。
`addSpace` と異なり DB への INSERT は行わない。
URL ファストパス専用のメソッド。

## 改善効果

| シナリオ | 改善前 | 改善後 |
|---|---|---|
| ゲスト・初回アクセス | 全件 fetch 完了まで待機 | 1件 fetch 完了次第遷移 |
| ゲスト・2回目以降 | localStorage キャッシュで高速（変化なし） | 変化なし |
| 管理者・初回アクセス | Supabase 3往復 | fetchSpaceByUrl + fetchSpaces 並列の1往復相当 |
| 管理者・2回目以降 | localStorage キャッシュで高速（変化なし） | 変化なし |

## SpaceSettingsScreen の読み込み改善（追加修正）

スペース設定画面を開いたとき、Supabase から設定を取得するまで
「読み込み中...」が表示されていた問題も同様に修正した。

### 原因

`settings` の初期値が `null` で、`fetchSpaceSettings` の完了を待つ間
「読み込み中...」を表示していた。
リロードで速かった理由は、前回 `saveSpaceSettings` で localStorage に
キャッシュされていたため。

### 修正

`useState` の初期化関数で `loadSpaceSettings` を同期的に呼び出し、
localStorage のキャッシュがあれば即座に表示するよう変更した。
`fetchSpaceSettings` は引き続きバックグラウンドで実行され、
Supabase の最新値で上書きする。

## Vercel 環境での管理者タブ非表示バグ（関連）

読み込み問題とは直接関係ないが、同じ調査で発見した問題。

### 症状

ローカルでは「タグ設定」「モデレーション」「Feature Flags」タブが表示されるが、
Vercel 上では表示されない。

### 原因

`.gitignore` に `.env*` が含まれているため、`.env.local` の Supabase キーが
GitHub に push されず、Vercel のビルドに含まれない。

Vercel 上では `VITE_SUPABASE_URL` と `VITE_SUPABASE_ANON_KEY` が未設定になり、
`isSupabaseConfigured = false` となってモック認証モードで動作する。

モック認証では `isAdmin` の判定がメールアドレスに `admin` が含まれるかどうかで行われる。
メールに `admin` が含まれていないアカウントでログインした場合、
`isAdmin = false` となり管理者専用タブが非表示になる。

### 解決方法

Vercel ダッシュボードの Settings → Environment Variables で以下を手動設定し、
再デプロイする。

| 変数名 | 値 |
|---|---|
| VITE_SUPABASE_URL | Supabase プロジェクトの URL |
| VITE_SUPABASE_ANON_KEY | Supabase の anon key |

## Vercel ビルドエラーの修正（関連）

上記の修正と同じタイミングで、Vercel のデプロイを失敗させていた
TypeScript エラーを2件修正した。
これらのエラーにより Vercel は古いビルドを配信し続けており、
タグ設定タブが表示されなかった真の原因でもあった。

### エラー1：PostScreen.tsx — isComposing

React の合成イベント型 KeyboardEvent に isComposing プロパティが
存在しないとして TypeScript エラーが発生していた。

修正: e.isComposing → e.nativeEvent.isComposing に変更。
nativeEvent 経由で DOM ネイティブの KeyboardEvent にアクセスする。

### エラー2：ModerationTab.tsx — 宣言前使用

allTagCandidates の useMemo が spaceHossiis を参照していたが、
spaceHossiis の useMemo 宣言がその後に書かれていた。
const はホイスティングされないため Temporal Dead Zone エラーが発生。

修正: dbHossiis と spaceHossiis の宣言ブロックを
allTagCandidates より前に移動。

## 未対応の課題

- ローディングスピナーなど視覚的なフィードバックがなく、
  白い画面のままになる点は変わらない（別途対応検討）
- fetchSpaceByUrl は community_id フィルタなしでクエリするため、
  異なるコミュニティ間でスラッグが重複した場合に意図しないスペースが
  ヒットする可能性がある（スラッグはランダム生成のため実害はほぼなし）
