## 概要

**隣人システム**は、登録・連携しているスペース同士が「隣人」としてゆるくつながる仕組み。

どうぶつの森の「友達の島」に相当するイメージで、以下の 3 つの体験を提供する。

| 機能 | 概要 |
|---|---|
| **漂着メッセージ（メッセージボトル）** | 隣スペースから設定した頻度で、誰かの投稿がボトルに入って流れ着く |
| **ワープ（訪問モード）** | LeftControlBar のアイコンを押すと、ランダムな隣スペースを閲覧専用で訪問できる |
| **隣の島 一覧（NeighborsScreen）** | 隣スペースを一覧で確認し、特定のスペースへ訪問できる |
| **管理者設定（隣人スペース登録・頻度設定）** | スペース管理画面から隣人スペースの登録・削除と漂着メッセージの頻度を設定できる |

---

## 設計方針

| 方針 | 内容 |
|---|---|
| **一方向登録** | 管理者が自分のスペースに隣人スペースを登録する（相手の承認は不要）。将来的に双方向承認フローへ拡張可 |
| **ゆるい頻度** | 漂着メッセージのデフォルトは 3 日〜7 日にランダムで 1 件。管理者が頻度を変更できる |
| **受動的な発見** | 参加者は届いた時に気づく体験。定期チェックを促さず、スペースに滞在中に自然に浮かぶ |
| **訪問は閲覧専用** | 隣スペースへのワープ中は投稿 UI を非表示にする。投稿したい場合は別タブで開く |
| **ワープはランダム** | 隣スペースが複数あっても、アイコン押下でランダムに 1 件を選ぶ（選択画面なし） |
| **読み取り専用（ボトル）** | 漂着したメッセージに対して返信・いいね等の操作はしない（将来拡張候補） |
| **Supabase 管理** | 隣人関係・漂着履歴ともに Supabase で管理し、端末をまたいで同期 |

## 1. DB スキーマ（新規テーブル）

### 1-1. `space_neighbors` テーブル

スペース間の隣人関係を管理する。

```sql
CREATE TABLE space_neighbors (
  id          UUID        DEFAULT gen_random_uuid() PRIMARY KEY,
  space_id    UUID        NOT NULL REFERENCES spaces(id) ON DELETE CASCADE,
  neighbor_space_id UUID NOT NULL REFERENCES spaces(id) ON DELETE CASCADE,
  created_at  TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE (space_id, neighbor_space_id)
);

-- RLS
ALTER TABLE space_neighbors ENABLE ROW LEVEL SECURITY;

-- 読み取りは誰でも可（スペース URL からアクセスするため）
CREATE POLICY "public read space_neighbors"
  ON space_neighbors FOR SELECT USING (true);

-- 書き込み・削除は管理者のみ（本番設定時）
CREATE POLICY "admin write space_neighbors"
  ON space_neighbors FOR INSERT USING (true); -- 本番化時に admin ロール限定に変更

CREATE POLICY "admin delete space_neighbors"
  ON space_neighbors FOR DELETE USING (true); -- 本番化時に admin ロール限定に変更
```

| カラム | 型 | 説明 |
|---|---|---|
| `id` | UUID | 主キー |
| `space_id` | UUID | 隣人を持つスペースの ID |
| `neighbor_space_id` | UUID | 隣人として登録されたスペースの ID |
| `created_at` | TIMESTAMPTZ | 登録日時 |

> `space_id` と `neighbor_space_id` の組み合わせはユニーク。
> A→B、B→A を別行として扱う（相互に登録すれば双方向になる）。

### 1-2. `bottle_deliveries` テーブル

各スペースへの漂着メッセージの履歴を管理する。

```sql
CREATE TABLE bottle_deliveries (
  id             UUID        DEFAULT gen_random_uuid() PRIMARY KEY,
  space_id       UUID        NOT NULL REFERENCES spaces(id) ON DELETE CASCADE,
  hossii_id      UUID        NOT NULL REFERENCES hossiis(id) ON DELETE CASCADE,
  from_space_id  UUID        NOT NULL REFERENCES spaces(id) ON DELETE CASCADE,
  delivered_at   TIMESTAMPTZ DEFAULT NOW()
);

ALTER TABLE bottle_deliveries ENABLE ROW LEVEL SECURITY;
CREATE POLICY "public read bottle_deliveries"
  ON bottle_deliveries FOR SELECT USING (true);
CREATE POLICY "public insert bottle_deliveries"
  ON bottle_deliveries FOR INSERT WITH CHECK (true);
```

| カラム | 型 | 説明 |
|---|---|---|
| `id` | UUID | 主キー |
| `space_id` | UUID | 届けられたスペースの ID |
| `hossii_id` | UUID | 漂着した投稿の ID |
| `from_space_id` | UUID | 投稿の元スペースの ID |
| `delivered_at` | TIMESTAMPTZ | 届いた日時 |

> このテーブルで「前回漂着日時」を管理し、配信間隔の判定に使う。

### 1-3. `space_settings` テーブルへのカラム追加

漂着メッセージの配信頻度を既存の `space_settings` テーブルに追加する。

```sql
ALTER TABLE space_settings
  ADD COLUMN IF NOT EXISTS bottle_frequency TEXT NOT NULL DEFAULT '3d-7d';
-- 取りうる値: '1d' | '3d-7d' | '2w' | '1m' | 'off'
```

| 値 | 表示ラベル | 配信間隔の計算 |
|---|---|---|
| `'1d'` | 毎日 | 1 日固定 |
| `'3d-7d'` | 3 日〜7 日（デフォルト） | `Math.random() * 4 + 3` 日 |
| `'2w'` | 2 週間に 1 回 | 14 日固定 |
| `'1m'` | 1 ヶ月に 1 回 | 30 日固定 |
| `'off'` | 無効 | ボトルを届けない |

> 型は `src/core/types/settings.ts` に `BottleFrequency` として追加する。

## 2. API 層

### `src/core/utils/neighborsApi.ts`（新規）

```ts
// 隣人スペース一覧を取得（スペース情報ごと返す）
fetchNeighbors(spaceId: string): Promise<Space[]>

// 隣人を追加（管理者のみ）
addNeighbor(spaceId: string, neighborSpaceId: string): Promise<void>

// 隣人を削除（管理者のみ）
removeNeighbor(spaceId: string, neighborSpaceId: string): Promise<void>

// 漂着メッセージを取得する（次の配信が可能かチェックして取得）
// 条件: 直近の delivered_at から frequency に応じた日数が経過しているかを確認する
fetchNextBottle(
  spaceId: string,
  neighborSpaceIds: string[],
  frequency: BottleFrequency
): Promise<BottlePayload | null>

// 漂着記録を保存する
recordBottleDelivery(spaceId: string, hossiiId: string, fromSpaceId: string): Promise<void>

// 直近の漂着日時を取得（クライアント側インターバル計算に使う）
fetchLastDeliveredAt(spaceId: string): Promise<Date | null>

// 隣スペースからランダムに 1 件を選んで返す（ワープ先決定に使う）
pickRandomNeighbor(neighbors: Space[]): Space
```

**型定義:**

```ts
// 漂着メッセージのペイロード
type BottlePayload = {
  hossii: Hossii;
  fromSpace: { id: string; name: string; spaceURL?: string };
};

// 配信頻度（src/core/types/settings.ts に追加）
type BottleFrequency = '1d' | '3d-7d' | '2w' | '1m' | 'off';

// 配信間隔の計算（ミリ秒）
function calcBottleIntervalMs(frequency: BottleFrequency): number {
  switch (frequency) {
    case '1d':    return 1 * 24 * 60 * 60 * 1000;
    case '3d-7d': return (Math.random() * 4 + 3) * 24 * 60 * 60 * 1000;
    case '2w':    return 14 * 24 * 60 * 60 * 1000;
    case '1m':    return 30 * 24 * 60 * 60 * 1000;
    case 'off':   return Infinity;
  }
}
```

## 3. 漂着メッセージ（メッセージボトル）

### 3-1. 概要

隣スペースから投稿が 1 件「ボトルに入って」スペース画面に流れ着く演出。
スペースに参加中、画面端からボトルアイコンが登場する。

### 3-2. 配信ロジック

| 項目 | 仕様 |
|---|---|
| **配信間隔** | 管理者が設定した `bottle_frequency` に従う（デフォルト: 3 日〜7 日のランダム） |
| **間隔の決定方法** | `calcBottleIntervalMs(frequency)` で計算した値を前回 `delivered_at` に加算し、現在時刻を超えたら配信可能 |
| **配信タイミング** | スペース画面（`#screen`）を開いたタイミングでチェック（ポーリングなし） |
| **`'off'` の場合** | チェックをスキップ。ボトルは一切登場しない |
| **選出ロジック** | 隣スペースの全投稿からランダムに 1 件選出（`bottle_deliveries` で既配信済みの投稿を除外） |
| **隣人が複数いる場合** | スペースをランダムに 1 件選択 → そのスペースの未配信投稿からランダムに 1 件選択 |
| **隣人が 0 件の場合** | ボトルは登場しない |
| **メッセージなし投稿** | `message` が空で `emotion` のみの投稿も選出対象（絵文字として表示） |

```
スペース画面を開く
  │
  ├─ bottle_frequency === 'off' → 終了（何も出さない）
  │
  ├─ fetchLastDeliveredAt(spaceId)
  │     └─ null（初回）または前回の日時を取得
  │
  ├─ 現在時刻 - delivered_at > calcBottleIntervalMs(frequency) ?
  │     │
  │     ├─ No → ボトルを出さない（終了）
  │     │
  │     └─ Yes → fetchNextBottle() で隣スペースからランダムに 1 件取得
  │                  └─ 取得成功 → ボトルアニメーション開始
  │                              → 20 秒後に自動消滅
  │                  └─ 取得失敗（隣人なし・投稿なし）→ 何も出さない
```

### 3-3. UI（MessageBottle コンポーネント）

**ボトル出現**

```
[スペース画面]

             🍾 ← 画面右下から浮かんでくる（CSS animation: slide-in）
                   クリック可能エリア（タップ／クリックで開封）
                   20 秒間表示 → 自動でフェードアウト
```

| 状態 | 動作 |
|---|---|
| 出現 | 画面右下から `translateY` アニメーションでゆっくり浮かぶ（0.8s） |
| 待機 | ゆっくり上下に揺れる（`@keyframes float` 2s infinite） |
| タップ/クリック | 開封モーダルを表示 + ボトルアイコンは消える |
| 20 秒放置 | フェードアウトして消える（未開封のまま消えると次回同じ投稿が再選出される可能性がある）→ 消えた時点で `recordBottleDelivery` は呼ばない |

**開封モーダル**

```
┌──────────────────────────────────┐
│ 🏝 となりの島から届きました        │
│ ──────────────────────────────  │

│                                  │
│                      ─ Aさん    │
│                 〇〇スペースより  │
│                                  │
│      [そのスペースに遊びに行く →] │
│                      [閉じる]   │
└──────────────────────────────────┘
```

| 要素 | 内容 |
|---|---|
| ヘッダー | 「🏝 となりの島から届きました」固定文言 |

| 本文 | `message`（最大 200 文字そのまま表示） |

| 元スペース名 | 「〇〇スペースより」 |

## 4. ワープ（訪問モード）

### 4-1. 概要

LeftControlBar の「隣の島」アイコンを押すと、登録済みの隣スペースから **ランダムに 1 件** を選んで訪問モードでワープする。
訪問モードは **閲覧専用**。投稿 UI は非表示になる。

### 4-2. ナビゲーション（LeftControlBar）

| 項目 | 内容 |
|---|---|
| アイコン | 🏝（または地図アイコン） |
| ラベル | 「隣の島」 |
| 表示条件 | 隣人が 1 件以上登録されているスペースのみ表示（0 件の場合は非表示） |
| 押下時の動作 | `pickRandomNeighbor(neighbors)` でランダム選択 → `visitingSpaceId` を store にセット → SpaceScreen を訪問モードで再レンダリング |

### 4-3. 訪問モードの状態管理

```ts
// useHossiiStore の state に追加
type HossiiStoreState = {
  // ... 既存フィールド
  visitingSpaceId: string | null; // null = 自スペース表示中
};
// アクション
{ type: 'SET_VISITING_SPACE'; payload: string | null }
```

- `visitingSpaceId` が `null` でない場合、SpaceScreen はその spaceId のデータを表示する
- `null` に戻すと自スペースの表示に戻る

### 4-4. 訪問モード中の UI 変更

| コンポーネント | 変更内容 |
|---|---|
| 投稿ボタン（LeftControlBar） | 非表示 |
| 投稿モーダル | 非表示 |
| LeftControlBar の「投稿」ボタン | 非表示 |
| VisitBanner（新規） | 画面上部に固定表示（詳細は 4-5 参照） |

### 4-5. VisitBanner コンポーネント

訪問モード中、画面上部にスティッキーで表示するバナー。

```
┌─────────────────────────────────────────────┐
│ 🏝 〇〇チームを訪問中          [このスペースを開く ↗]  │
│                         [← 下のスペースに戻る]       │
└─────────────────────────────────────────────┘
```

| 要素 | 内容 |
|---|---|
| 左: スペース名 | 「🏝 〇〇チームを訪問中」 |
| 右上: 「このスペースを開く ↗」 | クリックで `window.open(spaceURL, '_blank')` → 別タブで隣スペースをフル機能で開く |
| 右下: 「← 下のスペースに戻る」 | クリックで `visitingSpaceId = null` → 自スペースに戻る |

> 「このスペースを開く」は、隣スペースに投稿したい場合の手段として提供する。
> 別タブで開くと通常の参加フロー（ゲスト参加 or ログイン）が適用される。

### 4-6. ワープのフロー

```
LeftControlBar「隣の島」アイコン押下
  │
  ├─ fetchNeighbors(activeSpaceId) で隣スペース一覧を取得
  │     └─ 0 件 → アイコン自体を表示していないのでここには来ない
  │
  ├─ pickRandomNeighbor(neighbors) でランダムに 1 件選択
  │
  └─ dispatch({ type: 'SET_VISITING_SPACE', payload: neighbor.id })
       → SpaceScreen が neighbor.id のデータをフェッチして表示
       → VisitBanner を表示
       → 投稿 UI を非表示
```

| UI | トグルボタン（単一選択）。現在選択中を紫でハイライト |
| 選択肢 | 「毎日」「3日〜7日」「2週間」「1ヶ月」「無効」 |
| デフォルト | `'3d-7d'`（3日〜7日） |
| 保存先 | `space_settings.bottle_frequency` |
| 変更の反映 | 保存ボタン不要。選択した瞬間に `spaceSettingsApi` 経由で即時保存 |

### 6-4. 隣の島 追加フロー

```
管理者が spaceURL を入力
  │
  ├─ spaces テーブルで spaceURL が一致するスペースを検索
  │     │
  │     ├─ 見つからない → 「このスペースは見つかりません」エラートースト
  │     │
  │     ├─ 自分自身 → 「自分のスペースは登録できません」エラートースト
  │     │
  │     └─ 見つかった → addNeighbor(spaceId, neighborSpaceId) を呼ぶ
  │                      → 成功: 一覧に追加・「追加しました」トースト
  │                      → 重複: 「すでに登録されています」トースト
```

### 6-5. 削除フロー

- 一覧の「削除」ボタン押下 → 確認なしで即時削除（管理者操作のため）
- `removeNeighbor(spaceId, neighborSpaceId)` を呼ぶ
- 削除成功: 一覧から消える・「削除しました」トースト

### ワープボタンの表示タイミングについて

🏝 ワープボタンは **SpaceScreen（`#screen`）を開いたタイミング**で `fetchNeighbors` が実行され、1件以上登録されている場合のみ表示される。

```
隣人を NeighborsTab で登録
  │
  └─ SpaceSettingsScreen から SpaceScreen に戻る（コンポーネントが再マウント）
       → useEffect が発火 → fetchNeighbors → neighbors.length > 0
       → LeftControlBar に 🏝 ボタンが表示される
```

> 訪問モード中（`isVisiting === true`）はワープボタンは非表示になる。
> 自スペースに戻ることで再表示される。
