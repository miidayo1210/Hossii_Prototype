# 48 コードベース全体レビューと改善実施

## 概要

コードベース全体の健康診断を行い、セキュリティリスク・パフォーマンス問題・構造的負債を特定して修正した。

---

## レビュー結果サマリー

### 健康状態: 3/5（実施後: 4/5）

**良かった点（レビュー時点）**
- `any` の使用がほぼゼロ
- API 層が `core/utils/*Api.ts` に分離されており、Supabase 直接呼び出しが集中管理されている
- Realtime・BroadcastChannel・Web Speech など複雑なブラウザ API の扱いが比較的きれい
- Supabase anon key のみ使用しており、`service_role` の直接漏洩リスクがない

**危険だった点（改善対象）**
- `featureFlagsApi` が localStorage でサーバーフラグを上書き可能（ユーザーが DevTools で管理者フラグを操作できた）
- `imageStorageApi` が圧縮失敗時に型チェックなしで任意ファイルをアップロードするフォールバックを持っていた
- `spacesApi.ts` に本番コードとしてデバッグ用 `console.log`（ユーザーIDを常時出力）が残存
- `AuthContext` で `getSession` と `onAuthStateChange` が競合し、初回ロード時に認証状態が null フラッシュする可能性
- `HossiiStoreProvider.tsx` が 1208 行の単一 Context に全状態を集中させ、どの dispatch でもツリー全体が再描画
- `SpaceScreen.tsx` が 815 行・10+ useEffect で責務が肥大化
- `AuthContext` の value オブジェクトが毎 render で再生成され Consumer 全体が再描画されやすい構造だった

## 実施した改善

### Phase 1: Quick Wins（セキュリティ・パフォーマンス即修正）

#### B-3: spacesApi.ts — デバッグ console.log の削除
- `updateSpaceInDb` および `deleteSpaceFromDb` 内のデバッグ用 `console.log`（ユーザーIDを出力）を削除
- 対象: `src/core/utils/spacesApi.ts`

#### B-2: imageStorageApi.ts — MIME バリデーション追加
- アップロード前に `file.type` を検証する `validateImageFile()` 関数を追加
- 許可形式: `image/jpeg`, `image/png`, `image/webp`, `image/gif`
- `uploadBackgroundImage` / `uploadHossiiImage` の両方でアップロード前にバリデーションを実行
- 非対応形式は例外を投げてユーザーにエラーを返す
- 対象: `src/core/utils/imageStorageApi.ts`

#### B-1 + B-9: featureFlagsApi.ts — サーバー優先 + 並列取得
- サーバー取得成功時は localStorage override を無視し、サーバー値を最優先に変更
- localStorage はあくまで Supabase 取得失敗時のフォールバックのみに限定
- `feature_flags` と `space_feature_flags` の 2 クエリを `Promise.all` で並列実行
  - RTT 50ms 仮定時、直列 → 並列化で **約 50ms の短縮が期待できる**（未実測・推定値）
- 対象: `src/core/utils/featureFlagsApi.ts`

### Phase 1-d + Phase 2: AuthContext 安定化

#### B-7: value のメモ化
- 全認証関数（`signUp`, `login`, `adminLogin`, `adminSignUp`, `logout`, `loginWithGoogle`, `loginWithFacebook`, `refreshCommunitySlug`）を `useCallback` でメモ化
- context value オブジェクトを `useMemo` でメモ化
- Consumer の不要な再描画を**削減できる構成**に変更（コード解析上）
- 対象: `src/core/contexts/AuthContext.tsx`

#### B-4: getSession / onAuthStateChange 競合の低減
- `getSession()` と `onAuthStateChange` の二重発火による認証フラッシュの**原因となる競合を解消し、発生可能性を低減**
- `INITIAL_SESSION` イベントで初回セッションを処理する単一パスに統一（`getSession()` 呼び出しを廃止）
- 対象: `src/core/contexts/AuthContext.tsx`

- **`DisplayPrefsContext` と `HossiiContext` は役割が異なる。** 表示設定は `useDisplayPrefs()`、Hossii/Space データは `useHossiiStore()` から取得する

- **`HossiiStoreProvider` への機能追加時は、表示設定系は `DisplayPrefsContext` に追加すること**

### 変更前後の再描画範囲（コード解析ベース・推定）

> **注意**: 以下の表は React DevTools Profiler による実測ではなく、Context 伝播ルール（参照変化のない Consumer は再描画されない）に基づくコード解析上の推定。実測による確認は未完了。

#### 投稿送信時（`addHossii` dispatch）

`hossiis` 配列が更新されると HossiiContext が更新され、旧来は全 Consumer が再描画される構造だった。

| コンポーネント | 変更前（推定） | 変更後（推定） | 理由 |
|--------------|--------|--------|------|
| `HossiiToggle` | 再描画あり | **再描画の削減が期待できる** | `useHossiiStore` → `useDisplayPrefs` に移行。`hossiis` 更新の影響を受けない構成 |
| `DisplayScaleToggle` | 再描画あり | **再描画の削減が期待できる** | 同上 |
| `StarLayer` | 再描画あり | **再描画の削減が期待できる** | 同上 |
| `PostScreen`（`showHossii` 参照部分） | 再描画あり | **再描画の削減が期待できる** | 同上 |
| `SpaceScreen` | 再描画あり | 再描画あり | `hossiis` 自体を必要とするため変わらず（正常） |

#### 表示設定変更時（例: `setDisplayScale`）

旧来は HossiiContext 全体を更新していたため、Hossii データを購読するコンポーネントも巻き込んで再描画される構造だった。

| コンポーネント | 変更前（推定） | 変更後（推定） | 理由 |
|--------------|--------|--------|------|
| `HossiiLive`（各インスタンス） | 再描画あり | **再描画の削減が期待できる** | DisplayPrefsContext の更新が HossiiContext に伝播しない構成 |
| `TopBar` | 再描画あり | **再描画の削減が期待できる** | 同上 |

> `SpaceScreen` 自体は `useDisplayPrefs()` で `displayScale` を取得しているため、`DisplayScaleToggle` 操作時に再描画される（レイアウト変更のために必要な再描画）。

## React Profiler による再描画検証

### 計測状況

`React.Profiler` API を `AppContent`・`SpaceScreen`・`HossiiToggle`・`DisplayScaleToggle` の 4 箇所に一時的に追加し、`console.log` 経由でコンポーネント名・フェーズ（mount / update）・`actualDuration`（ms）を出力する計測コードを用意した。計測後は本コードから削除済み。

**自動計測は未実施**。ブラウザ自動操作の環境制約により数値の取得には至っていない。実測する場合は以下の手順で確認できる。

### Profiler コードの再現方法

各対象コンポーネントの return 直前に以下を追加してブラウザで確認できる:

```tsx
import { Profiler } from 'react';
import type { ProfilerOnRenderCallback } from 'react';

const onRender: ProfilerOnRenderCallback = (id, phase, actual, base) =>
  console.log(`[Profiler] ${id} | ${phase} | actual=${actual.toFixed(2)}ms | base=${base.toFixed(2)}ms`);

// return 内で:
<Profiler id="SpaceScreen" onRender={onRender}>
  {/* 既存 JSX */}
</Profiler>
```

### 確認手順

1. `http://localhost:5173` を開いてブラウザの DevTools → Console を開く
2. コンソールフィルタに `[Profiler]` と入力して絞り込む
3. 以下の操作を順番に行い、各操作後の行を記録する

| 操作 | 確認ポイント |
|------|-------------|
| ページ初期ロード（mount） | `AppContent`・`SpaceScreen` が出力されること |
| `DisplayScaleToggle` 100→125 をクリック | `DisplayScaleToggle` のみ出力。`SpaceScreen` が出力されないことが望ましい |
| `HossiiToggle` OFF→ON をクリック | `HossiiToggle` のみ出力。`SpaceScreen` が出力されないことが望ましい |
| 投稿送信（`addHossii`） | `SpaceScreen` は出力される。`HossiiToggle`・`DisplayScaleToggle` が出力されないことが望ましい |
