# 52 音声認識・入力機能

## 背景・課題

現状、音声入力をオンにするとほとんどの音が「笑い声」「😂」の気持ちとして自動投稿されてしまう。
感情検知による自動投稿は後回しにし、まず「音声による文字起こしと手動投稿」を優先的な機能として整備する。

---

## 設計方針

**現状のコードを最大限活かす。** 既存の `useSpeechRecognition` をベースに、「パネル表示」と「候補リスト」を追加する形にする。

### 変更の方向性

```
変更前: 音声認識 → バッファ → 自動で addHossii（感情ログ）
変更後: 音声認識 → バッファ → パネルに全文表示 → ユーザーが候補を選んで addHossii
```

**感情ログ（マイク音量からの笑い声・環境音の自動感情投稿）は停止中。** `SpaceScreen` では `useAudioListener` をマウントしない。ことばログの自動投稿（`speechLogEnabled` ON 時のみ）とは独立。

パネルは「手動確認して投稿する」UIとして、上記と並存させる。

**レイアウト**: 音声パネルは `FloatingPanelShell` でラップされ、**ドラッグ移動・エッジ/角リサイズ・位置とサイズの localStorage 永続化**が可能（[56_クリックフィードバックとフローティングパネル](./56_クリックフィードバックとフローティングパネル.md)）。初期位置・既定サイズは `floatingPanelStorage` のスピーチ用デフォルトを使用する。

---

## 既存リソースの整理（変更不要なもの）

| 既存リソース | 内容 | 活用方法 |
|---|---|---|
| `useSpeechRecognition.ts` | 音声認識・ノイズ除去・言語検出（`interimResults: true` 設定済み） | ほぼそのまま活用。`interimText` state を追加するだけ |
| `SpeechLevel` 型（`word` / `short` / `long`） | 粒度の3段階分類 | 候補抽出の粒度設定にそのまま再利用 |
| `classifySpeechLevel` 関数 | 文字数ベースで粒度判定（日本語: 10字以下=word, 30字以下=short） | 候補切り出し判定に流用 |
| `SpeechLevelSettings` / `listenStorage.ts` | 粒度設定の localStorage 保存 | 粒度設定の永続化にそのまま利用 |
| `listenMode` フラグ / `DisplayPrefsContext` | ON/OFF の状態管理 | パネル表示条件に利用 |
| 🎙 Listening インジケーター（`SpaceScreen.tsx`） | `listenMode` ON 時に画面下部に表示される要素 | タップでパネルを開閉するトリガーに転用 |

## 機能仕様

### 1. 音声パネルの表示トリガー

`listenMode` が ON になると、音声パネルが表示される（**フローティング**。既定は下部寄りの矩形で、ユーザーが移動・リサイズ可能。旧仕様では画面下部ドック固定だった）。

- **マイク ON と同時にパネルを開く**（追加タップ不要）
- 「🎙 Listening」インジケーターをタップすると、パネルの表示／折りたたみを切り替えられる（マイクは ON のまま）
- ダブルタップは操作が複雑になるため採用しない

> **原案との差異**: 仕様書原案ではダブルタップとしていたが、モバイルでのダブルタップ検出は複雑なため、既存の Listening インジケーターを活用するシングルタップ方式に変更する。

---

### 2. 右パネルのUI構成

```
┌──────────────────────────────┐
│ 🎙 音声テキスト         ✕   │
│                              │
│ （ここに話した内容が          │
│  リアルタイムで流れていく）   │
│  認識中のテキストはグレー表示 │
│  確定したテキストは黒で表示   │
├──────────────────────────────┤
│ 候補                         │
│                              │
│ [ 今日もいい感じ ]           │
│ [ ありがとう ]               │
│ [ 疲れたけど楽しかった ]     │
│                              │
│ 粒度: ● 短め  ○ 長め        │
└──────────────────────────────┘
```

---

### 3. リアルタイム全文表示エリア

`useSpeechRecognition` に **`interimText` state（3〜5行の追加）** を加えるだけで実現できる。
`interimResults: true` はすでに設定済みのため、`isFinal` でないテキストも `interimText` として state に流すだけでよい。

- 認識中のテキスト（未確定）はグレーで表示
- 確定したテキストは黒で表示
- テキストはどんどん追記されていく（最大 300 文字でスクロールまたは古いものを削除）
- パネルを閉じると全文テキストはリセットされる

---

### 4. 候補リスト

既存の `classifySpeechLevel`（文字数ベースの粒度判定）と `SpeechLevel` 型をそのまま活用する。

全文テキストから、設定された粒度に合わせてセグメントを切り出してリスト表示する。

**切り出しルール（日本語）:**

| 粒度 | 切り出し方 | 目安の文字数 |
|---|---|---|
| `word`（短め） | 「、」「。」「！」「？」「 」で分割した断片 | 10字以下 |
| `short`（中） | 「、」「。」「！」「？」で分割した節 | 10〜30字 |
| `long`（長め） | 「。」「！」「？」で分割した文単位 | 30字以上 |

既存の `classifySpeechLevel` の閾値（word: ≤10字, short: ≤30字）をそのまま流用する。

候補をタップすると `addHossii` を呼び出して投稿する（既存の `handleSpeechEvent` のロジックを流用）。

---

### 5. 粒度設定

`SpeechLevel` / `SpeechLevelSettings` / `listenStorage.ts` がすでに存在するため、**UIを追加するだけでよい**。

パネル下部に「短め / 長め」のトグルを追加し、`setSpeechLevels`（`DisplayPrefsContext` にすでに存在）を呼ぶ。

## 投稿フロー（変更後）

1. ユーザーが `listenMode` を ON にする（同意フロー後、または既に同意済みの場合）
2. **音声パネルが表示される**（フローティング。初回は既定矩形、以降は前回の位置・サイズを復元）
3. 話すと全文エリアにリアルタイムで文字起こしが流れる
4. パネル下部に候補リストが自動更新される
5. 気に入った候補をタップすると `addHossii` が呼ばれ投稿される
6. ✕ ボタンでパネルを閉じる（全文テキストはリセット）

## バグ修正記録

### Bug-1: パネルに文字が流れない（`speechLogEnabled` がデフォルト OFF）

**症状**: listenMode を ON にしてインジケーターをタップしてもパネルに文字が現れない。

**原因**: `useSpeechRecognition` を `enabled: listenMode && speechLogEnabled` で有効化していた。
`speechLogEnabled` はデフォルト `false` のため、マイクの音声認識自体が起動しなかった。

**修正**: `enabled: listenMode` に変更し、auto-post 制御は `handleSpeechEvent` 内で行う。

```ts
// 修正前
enabled: listenMode && speechLogEnabled

// 修正後
enabled: listenMode
// handleSpeechEvent 内で auto-post をガード
const handleSpeechEvent = useCallback((event) => {
  if (!speechLogEnabled) return;  // ← auto-post は speechLogEnabled ON 時のみ
  addHossii({ ... });
}, [addHossii, speechLogEnabled]);
```

---

### Bug-2: パネルが表示されない（z-index・親の overflow・レイアウト）

**症状**: パネルが表示されない、または一部が隠れる。

**原因**:
- `SpeechPanel` の `z-index` が SpaceScreen 内の要素より低かった。
- スペースのルート `.container` に `overflow: hidden` があり、`position: fixed` の子が**ビューポート外でクリップ**される場合がある。
- 旧実装の PC 右寄せレイアウトでは、ユーザーが「下から出るターミナル」を期待しづらかった。

**修正**:
- `z-index: 300` 程度に引き上げ（通常のスペース UI より手前）。
- `createPortal(<SpeechPanel />, document.body)` で **DOM を `body` 直下に**出し、クリップを回避。
- レイアウトを画面下部ドック（`slideInUp`）に統一していたが、その後 **フローティングパネル** に変更し、クリップ回避は引き続き portal で行う。

---

### Bug-3: 泣き笑い（😂）が自動投稿される（音量ベースの感情ログ）

**症状**: Listen をオンにすると、笑い声として 😂（`emotion: laugh` / `autoType: laughter`）が自動投稿される。

**原因**: `useAudioListener` がマイク RMS と変動パターンから「笑い」と判定し、`handleAudioEvent` 経由で `addHossii` していた。環境音・会話でも誤検知しやすい。

**対応経緯**:

1. 当初: `loadEmotionLogEnabled()` のデフォルトを `false` にし、左バーの「感情ログ」を明示 ON にしたときだけ `useAudioListener` を有効化。
2. **現在**: 当機能を**停止**。`SpaceScreen` から `useAudioListener` の利用を削除（マウントしない）。左バーの「感情ログ」チェックも非表示。
3. `listenStorage.loadEmotionLogEnabled()` は常に `false`。`saveEmotionLogEnabled` はストレージを `'false'` に揃え、旧 `true` を残さない。

`HossiiLive` の「聞いている顔」は、音声認識の `isRecognizing` を参照する。

---

### Bug-4: 音声入力 ON でもパネルが出ない（別タップが必要だった）

**症状**: マイクをオンにしても文字起こしパネルが現れない。Listening インジケーターをもう一度タップしないと開かない。

**原因**: `speechPanelOpen` が `false` のままで、パネル表示が `listenMode && speechPanelOpen` に依存していた。

**修正**: `listenMode` が `true` になったとき `useEffect` で `setSpeechPanelOpen(true)`。マイク OFF で `false` と蓄積テキストをクリア。✕ で閉じた場合は `listenMode` のまま `speechPanelOpen` のみ `false` にできる。
