# 67 コミュニティスペース一覧（スーパー管理者の表示）

## この文書の目的

**スーパー管理者**が [21_コミュニティごとのスペース管理画面](./21_コミュニティごとのスペース管理画面.md) の導線から **特定コミュニティのスペース一覧**を開いたとき、**他コミュニティのスペースが一瞬でも一覧に現れない**こと、およびそのために必要な実装方針を仕様としてまとめる。

一般管理者（自コミュニティのみ）の挙動は従来どおりを前提とする。

---

## 用語

| 用語 | 意味 |
|------|------|
| スーパー管理者 | `AppUser.isSuperAdmin === true`。全コミュニティを横断して管理する。 |
| override | `AdminNavigationContext` の `overrideCommunityId` / `overrideCommunityName` / `overrideCommunitySlug`。スーパー管理者が「どのコミュニティの管理画面を見ているか」を表す。`sessionStorage` に `hossii_adminNav` として永続化される。 |
| スペース一覧画面 | 実装上は `SpacesScreen`（ハッシュ `#spaces` または `#spaces/{communitySlug}`）。 |

---

## 現状の挙動と課題

### 期待する UX

- コミュニティ管理一覧から **あるコミュニティの「スペースを管理」** を押した直後から、**そのコミュニティに属するスペースだけ**が一覧に表示される。
- 全コミュニティ分のスペースが一度に並び、その後に絞り込まれるような **フラッシュは許容しない**。

### 技術的な原因（現行コード）

1. **`fetchSpaces` の仕様**  
   `src/core/utils/spacesApi.ts` の `fetchSpaces(communityId?: string)` は、`communityId` が **渡されない**とき `community_id` 列で **フィルタせず** `spaces` テーブルを **全件**取得する。

2. **`HossiiStoreProvider` の `communityId` の決め方**  
   `communityId = overrideCommunityId ?? currentUser?.communityId`  
   スーパー管理者は **`currentUser.communityId` を持たない**ことが多く、`overrideCommunityId` が **まだ無い**フレームでは `communityId` が **undefined** になり、`fetchSpaces(undefined)` が **全件取得**につながる。

3. **状態の持ち越し**  
   一度全件が `SET_SPACES` されると、`override` が確定してから **別の `communityId` で再 fetch** するまで、画面上は **全件のまま**残る。非同期の完了タイミングによっては **一瞬すべて表示**に見える。

4. **直接 URL・復元の遅延**  
   `SpacesScreen` は `overrideCommunityId` が無いとき `fetchCommunityBySlug(screenParam)` で override を **非同期**で復元する。この間も上記 2 により **communityId なしの fetch** が走り得る。

5. **一覧に戻ったあとの ref リセット（案 B 単体での抜け）**  
   案 B で `lastScopedCommunityFetchKeyRef` を「別コミュニティへ切り替わったか」判定に使うとき、コミュニティ一覧 `#communities` にいる間の `skipAllTenantFetch` 分岐で **ref を `undefined` に戻す**と、次に別コミュニティ B を開いた直後は `prevKey` が未定義のままになり **`shouldClearBeforeFetch` が false** になり得る。その結果、**直前に開いていたコミュニティ A の `state.spaces` が B の fetch 完了まで画面上に残る**（ブラウザキャッシュではなく React state の持ち越し）。

---

## 仕様上のゴール

| # | 要件 |
|---|------|
| G1 | スーパー管理者で **`overrideCommunityId` が未確定**のあいだ、`fetchSpaces` によって **テナント横断の全スペース一覧を state に載せない**。 |
| G2 | `overrideCommunityId` が **確定した時点**で、その **1 コミュニティ分**だけを取得して一覧に反映する。 |
| G3 | コミュニティ A から B に切り替えるとき、**A の一覧が B の読み込み完了まで残らない**（空・スケルトン・ローディングのいずれかでよいが、**他コミュニティのスペースを誤って表示しない**）。 |
| G4 | 一般管理者（`overrideCommunityId` が null で `currentUser.communityId` がある）は **現状と同様**、自コミュニティのスペースのみ取得する。 |

---

## 実装案（具体的）

以下は **いずれかまたは組み合わせ**で [G1]〜[G3] を満たす想定である。実装時は `npm run lint` / `npm run build` と、スーパー管理者での画面遷移を確認する。

### 案 A — ストア層で「スーパー管理者かつ community 未確定」は fetch しない

**場所**: `src/core/hooks/HossiiStoreProvider.tsx`

- `useAuth` から `currentUser?.isSuperAdmin` を参照する。
- 条件例: `isSupabaseConfigured && currentUser?.isSuperAdmin && !overrideCommunityId && !currentUser?.communityId` のときは **`fetchSpaces` を呼ばない**。
- この間は `spacesLoadedFromSupabase` の扱いを決める必要がある。例: **未ロード扱い**のままにする／**空配列を dispatch** して一覧を空にする。

**注意**: スーパー管理者が **コミュニティ一覧 `#communities` にいるだけ**のときも override は無いので、**誤って全件 fetch しない**ことと整合させる。

### 案 B — `communityId` 変更直前に一覧をクリア

**場所**: `HossiiStoreProvider` の `useEffect`（`communityId` 依存）の先頭、または reducer に `CLEAR_SPACES` のようなアクションを追加。

- `overrideCommunityId` が **null → 非 null** になったタイミング、または **UUID から別 UUID** に変わったタイミングで、**先に** `dispatch({ type: 'SET_SPACES', payload: [], ... })` 等し、**古い全件一覧を消す**。
- その後 `fetchSpaces(communityId)` で **正しい一覧だけ**を入れる。

**メリット**: fetch の有無と独立に **見た目のフラッシュ**を抑えやすい。

#### 案 B の補足 — **案1**: コミュニティ一覧では `lastScopedCommunityFetchKeyRef` を消さない

**場所**: `HossiiStoreProvider` の `skipAllTenantFetch` 分岐（および必要ならログアウト等でのリセット）

- **方針**: スーパー管理者が **コミュニティ一覧**にいて `fetchSpaces` をスキップしている間も、**直近スコープ付き fetch に成功したコミュニティ ID** は ref に **保持**する（`lastScopedCommunityFetchKeyRef.current = undefined` にしない）。
- **狙い**: 一覧からコミュニティ B の「スペースを管理」へ入ったとき、`prevKey` が直前の A のまま残り、`effectiveId === B` となり、**案 B の「別コミュニティへ切り替わる → fetch 前に一覧を空にする」**が確実に効くようにする（上記「技術的な原因 5」の対策）。
- **注意**: **ログアウト**・**別ユーザーでログイン**・文脈を本当にリセットすべきタイミングでは、ref を **別途クリア**し、他画面との整合を取る。

### 案 C — `fetchSpaces` の契約を厳格化する

**場所**: `src/core/utils/spacesApi.ts`

- 呼び出し側で **常に `communityId` を渡す**設計に寄せ、`undefined` のときは **空配列を返す**または **fetch 自体を行わない**（破壊的変更になり得るため、呼び出し箇所の洗い出しが必要）。

### 案 D — `SpacesScreen` の表示ガード

**場所**: `src/components/SpacesScreen/SpacesScreen.tsx`

- スーパー管理者かつ `overrideCommunityId` がまだ無い（`fetchCommunityBySlug` 待ち）の間は、**一覧 UI を出さず**ローディングのみ表示する。
- ストアに全件が入っていても **フィルタして表示しない**のは二重対策として有効だが、**データがストアに載らない**案 A・B の方が安全。

| 優先 | 内容 |
|------|------|
| 1 | **案 A** でスーパー管理者の **community 未確定時に全件 fetch を発生させない**。 |

| 3 | 必要に応じ **案 D** で UI を補強。 |

## 実装内容（反映済み）

| 区分 | 内容 |
|------|------|
| 案 A | `HossiiStoreProvider` の `fetchSpaces` 用 `useEffect` で、スーパー管理者かつ `overrideCommunityId` も `currentUser.communityId` も無いとき **`fetchSpaces` を呼ばない**。`communityId` が空のときも全件取得を避ける。スキップ時は `spacesLoadedFromSupabase` を true にし、`App.tsx` の `/s/` 解決と整合。 |
| 案 B | `lastScopedCommunityFetchKeyRef` で直近成功したスコープ付き fetch のコミュニティ ID を保持。**別コミュニティへ切り替わる**ときだけ、fetch 前に `SET_SPACES` で空配列と `SYNC_HOSSIIS` で hossii をクリアしてから `fetchSpaces`。**取得後**に `activeSpaceId` が一覧に無ければ先頭スペースへ `SET_ACTIVE_SPACE`。 |
| 案 B 補足（案1） | `skipAllTenantFetch`（コミュニティ一覧など）の分岐では **`lastScopedCommunityFetchKeyRef` を `undefined` に戻さない**（実装済み）。一覧から別コミュニティを開いたときも `prevKey` と `effectiveId` の比較で fetch 前クリアが効く。ログアウト時は `effectiveId == null` 分岐で ref をリセット。 |
| 補助 | アクティブスペースの hossii 同期 `useEffect` で、`activeSpaceId` が現在の `spaces` に存在しないときは **fetchHossiis を呼ばず** hossii を空にする（一覧クリア直後の不整合防止）。依存に `state.spaces` を追加。 |
| 案 D（最小） | `SpacesScreen` で、スーパー管理者かつ `overrideCommunityId` があり、一覧が空で **`spacesLoadedFromSupabase` が false** のとき「スペース一覧を読み込み中です…」を表示。 |

## 受け入れ条件（実装後の確認）

1. スーパー管理者でコミュニティ一覧から任意コミュニティのスペース管理へ遷移したとき、**最初のフレームから**当該コミュニティのスペースのみが表示される、または **意図したローディング**のみが見える（**全コミュニティ分が一瞬並ばない**）。
2. コミュニティ A のスペース管理を開いたあと **コミュニティ一覧に戻り**、別コミュニティ B のスペース管理を開いたとき、**A のスペース一覧が一瞬も表示されない**（空・ローディング・B の一覧のいずれか）。
3. ブラウザ更新で `#spaces/{slug}` を直打ちした場合も、復元後は **そのコミュニティのスペースのみ**（全件フラッシュなし）。
4. 一般管理者のスペース管理は **従来どおり**動作する。
5. `npm run build` が通る。
