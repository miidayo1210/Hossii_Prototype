# 69 投稿の UI 設定（吹き出し投稿／フリー投稿）

## この文書の目的

現状の **「一投稿＝一吹き出し」** を維持しつつ、**ストーリー編集に近い自由なレイアウト**で 1 画面を作れる投稿方式（**フリー投稿**）を **段階的に** 導入するための要件と、**本書の推奨案（差し替え可）** を整理する。実装の前に **スコープ境界** と **プロダクト定義** を固定し、工数と既存機能（書き出し・一覧など）との衝突を抑える。

---

## プロダクト決定（2026-04 整理・実装は追随）

以下はプロダクト上 **確定した方針**。実装との差分は下記 **「実装状況」** および **「実装と仕様の差分メモ」** を正とする。[68](./68_スペース書き出し機能%20.md) の合成・透過は別途 68 と揃える。

| 区分 | 内容 |
|------|------|
| **ユーザー向け名称** | モード切替タブは **「フリー」**。投稿種の呼び方は **「フリー投稿」**。実装上の `post_kind: canvas` はコードの正とし、文書では **フリー投稿（`post_kind === canvas`）** と併記してよい。 |
| **位置づけ** | 吹き出しより **レイアウトにこだわりたい**とき、**短文ではなくまとまった文章**を載せたいときに使う。**音声入力で得た文章**もフリー投稿に載せられるようにする（フリー編集内のテキスト入力へ接続する想定）。 |
| **表示の正** | スペース上は **完成ラスタ 1 枚**をそのまま表示。**縦横比・ピクセル感はエディタ上のキャンバス枠と一致**させる（エディタで変えた枠＝出力 PNG＝ツリー上の見え方）。 |
| **本文テキストの正** | ユーザー向けの「説明・代替テキスト」**専用フィールドは置かない**。エディタ内の **テキストブロックに入力した文章**を、投稿確定時に **`message`（既存のテキスト列）に記録**する（複数ブロックは改行等で連結するなど、ルールは実装で定義）。**ログ一覧でも本文として閲覧**できること。画像にのみ描いた内容は自動ではテキスト化しない。 |
| **背景** | **単色に加え透明を選択可**。透明時はスペースの **壁紙が透けて見える**。書き出しは **アルファチャンネル付き PNG** を前提とし、[68](./68_スペース書き出し機能%20.md) 側の合成との整合を取る。**背景写真**も引き続き利用可能（現行機能の延伸）。**実装（フリー編集）**: 既定の単色は **薄い黄色（クリーム系、`#fef3c7` 相当）**。（横長 16:9 のステージ全体がこの色で塗られる。） |
| **キャンバス枠** | PowerPoint の図形のように **ハンドルで枠のサイズ・縦横比を可変**。既定は **横長（16:9）** から開始してよい（吹き出しの画像プレビュー比率に寄せる）。**実装**: **右下 1 つのリサイズハンドル**（四隅すべてではない）。 |
| **テキスト UX** | **「＋テキストを追加」**でカード上にオブジェクトを置き、**そのテキストをタップして編集**（インスタのストーリーに近い）。**ドラッグで移動**、**角ハンドルでリサイズ**。**フォント・色の選択**は現行方針を維持。**実装**: **短いタップ**（移動が小さい場合は編集）、**ダブルタップ**でも編集、選択中は **右下つまみで文字サイズ**、ツールバーに **補助の 1 行入力・スライダ**。 |
| **画像の呼び方** | 「前景の写真」など開発用語をやめ、**イラスト・ボランティア文脈でも直感できるラベル**に差し替える。**実装**: セクション **「重ねる画像」**＋絵文字タイル（**📎 画像を選ぶ** 等）。吹き出し投稿の写真ボタンと同系の **破線タイル UI**。 |
| **誤操作防止** | **誤タップで投稿しない**工夫を設計要件とする。**実装（2026-04）**: 吹き出しに寄せて **確認ダイアログは出さない**。必要ならトースト・ボタン配置・二段押し等で後から足す。 |
| **未保存の離脱** | 理想系では **確認ダイアログ**もありうる。**実装（2026-04）**: フリーは **confirm なし**でタブ切替・パネル閉じる（手軽さ優先。未保存は破棄に注意）。 |
| **スペース書き出し** | フリー投稿ノードを **PNG キャプチャに含める**（**表示中の一覧**＝`displayHossiis` と整合）。[68](./68_スペース書き出し機能%20.md) と `spaceCanvasExport` を更新する。**実装**: `exportNodeFilter` から canvas 専用除外は **解除済み**（リサイズハンドル等の除外のみ）。 |
| **編集再開** | シーン JSON による **レイアウトの完全再編集**は必須としない。テキストはログ用に DB に残すため、保存は **画像＋`message` の併存**とする。 |
| **荒れ対策・権限** | フリー投稿だけを止める等の細則は **未決**。従来どおり **投稿機能全体のゲート**や別ガードで後から足す余地を残す。 |

## 実装状況（現行コードとの対応）

**2026-04 時点**。詳細は `PostScreen` / `CanvasPostEditor` / `Tree` / `SlideshowView` / `spaceCanvasExport` / `hossiis.post_kind` を正とする。

| 項目 | 実装 |
|------|------|
| **データ** | `hossiis.post_kind`: `bubble`（既定）\| `canvas`。フリー投稿は完成 **ラスタ 1 枚** を `image_url` に保存。 |
| **本文** | テキストブロックを **空行 `\n\n` 区切り**で連結し、投稿確定時に **`message` に保存**（`CanvasPostEditor` の `joinTextBlocksForMessage`）。ログ一覧は既存どおり `message` を表示。吹き出しのメッセージ欄とは独立。 |
| **投稿 UI** | **全投稿画面・パネル**で **吹き出し／フリー**タブを表示。ラベル **「フリー」**。デフォルトは **吹き出し**。 |
| **タブを出さない場合** | **音声候補の編集モード**（`speechEditMode`）では **吹き出し専用**、モードタブ **非表示**。 |
| **Feature Flag** | フリータブを **Feature Flag で止めない**（常時利用可能）。 |
| **エディタ（`CanvasPostEditor`）** | **カード枠**: 既定 **16:9 横長**（例: 幅 320px × 高さ 180px 相当）、**右下ハンドル**で幅・高さを可変（最小・最大あり）。**背景**: 既定色 **#fef3c7**、**壁紙透過**トグル、**色**（カラーピッカー）、**写真**（背景 1 枚・**`contain` でまるごと表示**、透過オフ時は **下層に同一画像の cover＋ぼかし**で余白の色味をつなぐ。[70](./70_背景画像サイズ.md) 参照）。**説明文**: キャンバス上の **ヒント段落は表示しない**（操作は UI のラベル・`title` に寄せる）。**ツール UI**: `PostScreen`（吹き出し）と **同色・同系のラベル・送信ボタン（3D グラデ）・破線タイル**で統一。**テキスト**: 複数ブロック、**8px 超のドラッグで移動**、未満のリリースは **タップ＝編集**、**ダブルタップ**でも編集可。選択時 **右下つまみでフォントサイズ**、ツールバーに色・フォント・スライダ・1 行入力。**重ねる画像**: 1 枚・ドラッグ。**投稿**: `html-to-image` の `toPng`（透明時 `backgroundColor: transparent`）→ Storage または data URL。プライマリ **「気持ちを置く」**／**送信中...**。**確定・離脱**: **`window.confirm` なし**。 |
| **音声 → フリー** | クイック投稿表示中、`SpeechPanel` の **「フリーに入れる」** → `speechToFreePosterRef` 経由でテキストブロック追加。吹き出しタブのときは **自動でフリータブに切替**後に追加。 |
| **スペース表示** | `Tree` 上のフリー画像は **固定アスペクト比なし**で、`width: 100%`・`height: auto`・`object-fit: contain`・`max-height` により **自然縦横比**。`alt` フォールバック **「フリー投稿」**。**拡大**: 画像の **シングルタップ／クリックではライトボックスにしない**（リサイズ誤爆防止）。**ダブルクリック**、または **選択中の 🔍** で拡大。**選択時**はバブル **`overflow: visible`** で四隅ハンドルが欠けない。フリーは **やや大きめのリサイズハンドル**。 |
| **スライドショー** | フリーは **画像主表示**。`.slideImageCanvas` も **自然比率**（固定 aspect-ratio 撤廃）。 |
| **モデレーション** | 一覧の **画像サムネ**＋**本文 `message`**（フリーはテキストブロック連結文）。 |
| **スペース書き出し** | **フリー投稿をキャプチャに含める**（`exportNodeFilter` は UI 部品の除外のみ。canvas ノードは除外しない）。透明 PNG の **スペース書き出し上の見え方**は [68](./68_スペース書き出し機能%20.md) の合成方針に従う。 |
| **永続化（Supabase あり）** | 画像は Storage（`canvas/{spaceId}/{hossiiId}.png|webp` 等）へアップロードし、`hossiis` 行の `image_url`・`post_kind` で参照。初期表示は DB 取得が正（`localStorage` の投稿キャッシュは読み込まない）。INSERT 失敗時はリロード後に当該投稿が載らない（コンソールの `[hossiisApi] insertHossii error` 等を確認）。 |
| **永続化（Supabase 未設定・デモ）** | アップロード無しのため完成 PNG を **data URL** で `imageUrl` に載せ、`localStorage` キー **`hossii.hossiis`** に保存する。`saveHossiis`（`src/core/utils/storage.ts`）は **`imageUrl`・`postKind`（`post_kind` 相当）・座標・スケール・色・タグ・画像付き吹き出し用フィールド等**を `normalizeHossii` と揃えてシリアライズする（欠落するとリロード後にフリーが「消えた」ように見える／画像表示モードでは `imageUrl` 欠落で一覧から除外される）。**data URL は肥大化しうる**ため、端末の **localStorage 容量上限**を超えると保存に失敗し得る（その場合もリロードで失われる）。詳細は [04](./04_データ保存機能.md)。 |

**メモ**: Phase 1 のコアは実装済み。**Phase 2**（再編集 JSON、複数画像、回転等）は未着手。細かな UX（単タップ編集への統一、四隅リサイズ、初回チュートリアル等）は下記参照。

## 課題整理：フリー投稿が「消える」（リロード後・操作中）

利用者から **フリー投稿が消えた** と見える事象は、次の **複合的な課題** として整理する（いずれもプロダクト／実装上の論点）。詳細な保存先の整理は [04](./04_データ保存機能.md) と突き合わせる。

| 課題 | 内容 | 実装での対応（2026-04 時点） | 残課題・注意 |
|------|------|------------------------------|-------------|
| **A. localStorage の保存不足** | Supabase 未設定時、`saveHossiis` が `imageUrl`・`postKind` 等を保存していなかった。リロード後にフリーが復元できず、**画像表示モード**では `imageUrl` 欠落で **一覧から除外**され「消えた」ように見えた。 | `storage.ts` の `saveHossiis` を `normalizeHossii` が読むフィールドと揃えて保存。 | **data URL が大きい**と `QuotaExceededError` で保存失敗し得る。コンソールに `[storage] saveHossiis: localStorage quota exceeded` を出す。デモ向けに **出力解像度の抑制**や **IndexedDB 退避**は未検討。 |
| **B. 投稿一覧の再同期タイミング** | Supabase 有効時、`state.spaces` の **参照が変わるたび**（スペース名・背景・タグ等のメタ更新）に `hossiis` を **いったん空にしてから** `fetchHossiis` する実装だった。不要なクリアが多発し、下記 C と組み合わさると不具合の温床になる。 | `HossiiStoreProvider` で **スペース ID の集合**（`spaceIdsSignature`）が変わったときだけ再フェッチするよう変更。 | スペース集合の定義を変えたときは **再取得が必要か**を設計レビューに含める。 |
| **C. 楽観更新と Realtime の競合** | 自分の INSERT を Realtime で **「重複なので無視」**していたが、直前の **SYNC で楽観投稿が消えている**と、DB には行があるのに **画面に戻らない**状態になり得た。リロード後は DB 次第だが、操作中の「消えた」と誤認しやすい。 | 自己 INSERT の Realtime でも、**state に該当 ID が無ければ** `ADD_HOSSII_FULL` で **DB 行から復元**。 | 極端な遅延・オフライン時の UX は別途（オフラインキュー等は未スコープ）。 |
| **D. INSERT 失敗が分かりにくい** | `insertHossii` が失敗しても楽観投稿だけ残り、リロードで消える。コンソールログのみで **利用者に理由が伝わらない**。 | `insertHossii` が成否を返し、**失敗・例外時は楽観投稿を削除**し、`insertedHossiiIdsRef` を解放（幽霊 ID で Realtime を捨て続けない）。 | **トーストやエラー文言**で INSERT／RLS／マイグレーション不足を伝えるのは **未実装**（要プロダクト判断）。 |

**本番（Supabase）運用でのチェックリスト（課題の再発防止）**

- `hossiis` に **`post_kind`** 列と制約が入っているか（マイグレ未適用だと INSERT 失敗しうる）。
- **RLS**・**Storage ポリシー**で canvas 用パスのアップロードが許可されているか（[04](./04_データ保存機能.md) の Storage 節）。
- ブラウザコンソールの **`[hossiisApi] insertHossii error`** の有無。

## 背景・コンセプト

- **現状**: 投稿は「一投稿」が「一つの吹き出し」にパッケージ化され、スペースに配置される。
- **維持したいこと**: **吹き出し投稿**は既存どおりの体験として残す（デフォルト／主軸の投稿形態の想定）。
- **追加したいこと**: インスタグラムのストーリー編集に近い体験として、**文字のフォント・色・配置**、**画像の配置**、将来的に **手書き・移動・回転** などを含め、**それ自体が 1 つの画面（1 投稿）** になる投稿方法（**フリー投稿**）。
- **UI のイメージ**: **投稿パネル／投稿画面**に **モード切り替え**（タブ）を置き、**吹き出し**と **フリー** を切り替える。

## 実現可否（方針）

**可能**。投稿の種類を増やし、同一スペースに **吹き出し投稿とフリー投稿を混在**させる UI/UX は取りうる。ただし **スペース上での見え方・保存形式・既存機能との整合** を先に決めないと実装が分岐・肥大化しやすい。

## 段階的スコープ（提案）

**各フェーズで「やること／やらないこと」を仕様として明文化する**（「やらない」は後回しでよいが **意図的に除外** と書く）。

### Phase 1（軽め・MVP・現行実装の土台）

**実装済み（2026-04 時点の目安）**

- 編集キャンバス: **既定 16:9 横長**＋**右下ハンドルで可変枠**（`CanvasPostEditor`）。見た目は **吹き出し投稿画面と同系の色・フォント・送信ボタン**に揃える。
- **背景**: 単色、**透明**、**背景写真** 各対応。
- **テキスト**: 複数ブロック、フォント・色・位置、**タップで編集**（微ドラッグ閾値あり）・ダブルタップ可、**角つまみでフォントサイズ**、ツールバー補助。
- **前景画像**: **1 枚**・ドラッグ、UI は **「重ねる画像」** セクションのタイル（📎 等）。
- 投稿フロー: **吹き出し／フリー**切替、**Flag によらず常設**、フリーは **confirm なし**で投稿・タブ切替・パネル閉じる。
- **保存**: **ラスタ 1 枚** ＋ **`message` にテキストブロック連結**（`\n\n` 区切り）。**代替テキスト専用欄なし**。
- **音声→フリー**、**スペース書き出しにフリー含む**、**ツリー／スライドの自然縦横比**。

**やらない（Phase 1 では明示的に除外してよい例）**

- 手書きストローク、**回転**、複数画像、動画、複雑なレイヤー順 UI、アニメーション。
- **レイアウトの完全な再編集**（シーン JSON の恒久保存）は必須としない。

### Phase 1.1（書き出し・整合）

- **実装済み**: スペース PNG 書き出しに **フリー投稿を含める**（`spaceCanvasExport`）。透明合成の細部は 68 と継続整合。

### Phase 2 以降（将来）

## 詰めるべきポイント（本書の推奨案と代替）

実装に入る前に決める論点に対し、**まず採用しやすいデフォルト案**を示す。プロダクト判断で差し替えてよい。**代替・後続**は短く併記する。一部は **プロダクト決定**で埋まった。

### 1. プロダクト定義（最優先）

| 項目 | 本書の推奨案 |
|------|----------------|
| **スペース上の見え方** | **1 投稿＝スペース上の 1 オブジェクト**とし、**完成ラスタをそのまま表示**。**吹き出しと同じ座標系・同じ「木に載る」モデル**。縦横比は **エディタのキャンバス枠と一致**。**フリー投稿の拡大**は **ダブルクリック**または **選択中の 🔍**（シングルタップでは開かない。編集・リサイズ優先）。 |
| **「1 投稿」の意味** | **ビジュアルの正は画像 1 枚**。あわせて **テキストブロック由来の本文を `message` に保存**し、ログ・運用で読めるようにする。 |
| **混在ポリシー** | **同一スペースで吹き出しと混在**を基本。**スペース設定での ON/OFF** は運用で必要になったら追加。 |
| **利用権限** | **全員が投稿可能**を基本。**フリーだけ制限**等の細則は **未決**。**フリータブを Flag で止める**方針は採用しない。必要なら **投稿機能全体**や **別のガード**で検討。 |

**代替・後続**: 「サムネ＋タップでだけ全画面」中心にし、スペース上はアイコンのみ、など **別パターンは Phase 2 以降**で検討してよい。

### 2. 保存形式

| 項目 | 本書の推奨案 |
|------|----------------|
| **ラスタ** | **WebP または PNG**（**透明背景時はアルファ付き**）。 |
| **本文** | テキストブロックを **1 本のプレーンテキスト**として `message` に保存（連結ルールは実装で定義。例: 改行区切り）。**ユーザー向け「代替テキスト」専用フィールドは置かない**。 |
| **編集再開** | **レイアウトの完全再開**は Phase 2 以降で検討。Phase 1 では **画像確定後はレイアウト編集不可**を前提にしてよい。 |

**比較（参照）**

| 選択肢 | メリット | デメリット |
|--------|----------|------------|
| **ラスタ＋message** | 表示が単純。ログに本文を残せる。 | 画像のみの情報はテキスト検索に載らない。 |
| **シーン JSON ＋ クライアント描画** | 再編集・拡張に強い。 | スキーマ・バージョン管理・互換が必要。工数が増えやすい。 |

### 3. 端末・操作

| 項目 | 本書の推奨案 |
|------|----------------|
| **主戦場** | **スマホ優先**（**誤タップで投稿しない**・片手操作を設計要件に明記）。 |
| **アスペクト** | **可変**（ハンドル）。既定比率は実装で **16:9 横長**から開始してよい。 |
| **解像度** | 端末差は **エクスポート解像度の下限／上限**（実装で調整）。編集プレビューは **CSS スケール**で吸収する想定。 |

### 4. 制限・品質（初期値・実装で調整可）

| 項目 | 本書の推奨案 |
|------|----------------|
| **画像** | アップロード **1 枚・5MB 上限**（Phase 1）。背景／前景の分離をする場合は仕様で **どちらにカウントするか** を固定。 |
| **テキスト** | **ブロック数上限**、**ブロックあたり文字数**でレイアウト崩れと負荷を抑える（具体値は実装値）。 |
| **フォント** | **ライセンス明確なプリセット 3〜5 種**（**日本語必須**）。 |
| **アクセシビリティ** | 専用の「代替テキスト」入力は **置かない**。**スクリーンリーダー**は `message` や「画像」扱いで読み上げる想定（詳細は A11y 設計で調整）。 |

### 5. 既存機能との整合

| 機能 | 本書の推奨案 |
|------|----------------|
| **スペース書き出し**（[68](./68_スペース書き出し機能%20.md)） | フリー投稿を **キャプチャに含める**。**表示中の一覧**（`displayHossiis`）に整合。**透明背景**のときの合成方法は 68 と実装で揃える。 |
| **スライドショー・一覧** | **吹き出しと同じ並び・同じフィルタ軸**に載せ、フリーは **サムネ＝保存画像** で統一。 |
| **モデレーション** | **吹き出しと同等**（非表示・通報の対象）。**画像＋`message`** を確認できる UI。 |
| **ログ一覧** | フリー投稿も **本文（`message`）を表示**できること。 |
| **同期・ストレージ** | 画像は **オブジェクトストレージ**（本番）。**デモ・オフライン**は **data URL ＋ `localStorage`（`hossii.hossiis`）** で、`post_kind` 相当・`image_url` 相当を欠かさず保存する（実装: `saveHossiis`／`normalizeHossii`）。投稿行は **既存 `hossiis` に `post_kind` ＋ 画像 URL 等を拡張**する案が扱いやすい。 |

## UI 要件（確定度の高い部分）

- **投稿パネル／投稿画面**（`PostScreen` が使われる **すべての経路**）に、**吹き出し**と **フリー** を切り替える **モード UI** を **常に表示**する（**Feature Flag では制御しない**）。タブラベルは **「フリー」**。
- **例外**: **音声候補の編集など、吹き出し専用のフロー**ではタブを出さず、従来の吹き出し UI のみとする。
- 吹き出し投稿は **既存フローを壊さない**こと。**デフォルトは吹き出し**。
- **誤タップで投稿**しにくい操作設計とする（現行フリーは confirm なし。要望があれば後から追加）。
- フリー編集中の **未保存離脱**は、現行は **確認なし**（手軽さ優先）。理想系・後続でダイアログを戻す余地あり。

## プロダクト要望メモ（UX・文言）

以下は **意図のメモ**（優先度はプロダクトで調整）。多くは Phase 1 で一部反映済み。残りは **「実装と仕様の差分メモ」** と **Phase 2** で扱う。

- **カードサイズ**: 既定は **横長（16:9）**。PPT の四角のように **枠のサイズ・比率を変えたい** → **右下ハンドルで可変**済み。**四隅すべて**は未対応。
- **背景**: **色**と同時に **透明**、さらに **背景写真** → いずれも対応。透明時はスペース壁紙が透ける（チェッカーはエディタ内の見やすさ用）。
- **テキスト**: 「＋テキストを追加」で配置し、**タップで編集（インスタ風）** → **短いタップで編集**（わずかに動かしただけでは移動しない閾値あり）＋ダブルタップ可＋ツールバー。
- **移動・文字サイズ**: ブロックをドラッグで移動。**角のつまみ**でサイズ（現行は **フォントサイズ**）変更。
- **色・フォント**: 現行どおりツールバーから選択。
- **前景画像の呼び方**: 「前景の写真」等をやめ、**重ねる画像**セクション＋アイコンタイル。**title** で補足可能。
- **「説明・代替テキスト」専用欄**: **置かない**（プロダクト決定どおり・実装済み）。
- **総じて**: 操作と文言を **より直感的**に → 初回ガイド、ラベル棚卸し、確定前サマリ等はバックログ。


フリー投稿画面で、『テキストを追加』ボタンを押して、追加する”テキスト”の枠が出て、そこにテキストを入力した後に、『気持ちを置く』ボタンを押すと、テキストを編集する枠自体が消えないまま（存在したまま）画像となって、投稿されてしまう。テキストを編集する枠は、投稿されないようにしてほしい。
