# 70 背景画像サイズ（スペース壁紙・フリー編集の背景写真）

## この文書の目的

**画像を背景として敷いたとき**、ビューポートやコンテナの縦横比と写真の縦横比が一致しない場合の **切り抜き・余白** の扱いを整理する。実装の正は CSS（`background-size` 等）および `CanvasPostEditor` のステージ背景。

---

## 現状の挙動

| 箇所 | 実装の目安 | 結果 |
|------|------------|------|
| **スペース全体の背景画像** | [`spaceBackgrounds.module.css`](../../../src/styles/spaceBackgrounds.module.css) の `.bgImage` が **`background-size: cover`** | 画面（コンテナ）を **常に塗りつくす**。アスペクト比が違うと **上下または左右がクロップ** される。 |
| **フリー編集ステージの背景写真** | [`CanvasPostEditor.module.css`](../../../src/components/PostScreen/CanvasPostEditor.module.css) の `.stageBg` が **`background-size: cover`** | ステージ矩形（既定は **16:9 横長**）に合わせて **クロップ**。例: **4:3 の写真**を敷くと **縦方向が切れやすい**。 |

「16:9 だから切れる」より、**`cover` であることが切れの直接原因**と捉えるとよい。

---

## 課題（プロダクト）

- 利用者が **「写真まるごと見せたい」** とき、`cover` だけでは不満が出る。
- 逆に **`contain` にすると余白** が出る。余白の色・ぼかし・グラデをどうするかは見た目要件になる。

---

## 方針の選択肢（推奨の整理）

| 方針 | 内容 | メリット | デメリット |
|------|------|----------|------------|
| **A. cover のまま** | 現状維持 | 画面が常に「埋まる」。実装が単純。 | 比率が合わないと **必ず切れる**。 |
| **B. contain（＋余白処理）** | 画像全体を枠内に収める | **切れない**。 | 余白が出る。余白色・ぼかし等のデザインが必要。 |
| **C. ユーザー選択** | 「全体表示／画面いっぱい」をトグル | 用途に合わせられる。 | UI・保存項目（将来）の増加。 |
| **D. スペースサイズを画像に合わせる** | レイアウトの基準を画像比率に依存 | 理論上は切れにくい | **スペースのキャンバス比率が投稿ごとに変わる**と、吹き出し配置・書き出し・既存 UI と **衝突しやすい**。基本は **非推奨**（特化ワールド等の別プロダクトなら再検討）。 |

### 本書の推奨（段階的）

1. **短期**: スペース背景・フリー背景のどちらか、または両方で **`contain` 案を試す**場合は、**余白を単色（設定色 or 暗幕）で埋める**のが実装コスト対効果が高い。  
2. **中期**: 需要があれば **C（トグル）** をスペース設定やフリー編集ツールに追加。  
3. **スペース全体の縦横比を画像に追従**（D）は、**既存の「スペース＝固定ビューポート」モデル**と両立しにくいので、**デフォルト案にはしない**。

## 実装メモ（後続）

- スペース: `.bgImage` の `cover` → `contain` 変更時、**親要素の背景色**を必ず指定（余白が透ける事故防止）。
- フリー: ステージは **書き出し PNG にそのまま焼く**ため、`contain` にすると **余白も画像に含まれる**。意図と一致するか確認。
- 詳細は [69 投稿の UI 設定](./69_投稿のUI設定.md)（フリー背景）、[68 スペース書き出し機能](./68_スペース書き出し機能%20.md)（合成時の見え方）と突き合わせる。
