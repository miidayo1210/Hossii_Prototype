# 71 スマホマイク機能（音声入力のしやすさ・画面まわり）

> **分類:** `[UX / モバイル]` スペース画面・Listen・音声パネル  
> **関連:** [52_音声認識・入力機能.md](./52_音声認識・入力機能.md)（Listen・パネル） / [56_クリックフィードバックとフローティングパネル.md](./56_クリックフィードバックとフローティングパネル.md)（`FloatingPanelShell`） / [62_スマホのスペース画面.md](./62_スマホのスペース画面.md)（左バー・大画面）

---

## 1. この文書の目的

スマートフォンで Hossii の **音声入力（Listen）** を使うとき、**「端末をマイクのように持って話す」**体験を損なわないよう、レイアウト・没入表示・向きの案内を **プロダクト方針として固定**する。

元メモの論点:

- 縦長画面のため **PC 前提のレイアウトとずれる**。
- **大画面ボタン**操作後に **真っ白に見える**ことがある。
- スマホで **無理に横長（ランドスケープ）に固定**するのは難しい可能性がある。
- **横持ち・横画面を促す案内**はありうる。

---

## 2. プロダクト上のゴール

| ゴール | 説明 |
|--------|------|
| **認知負荷を下げる** | 「マイクはオンか」「文字はどこに出るか」「投稿までの道」が一目で分かる。 |
| **片手・縦持ちでも最低限使える** | 理想は横持ちだが、**縦持ち禁止にはしない**。縦でもパネルとスペースが両立しやすいレイアウトを目指す。 |
| **没入（大画面）の意味が端末でブレない** | ボタンを押したあと **何も変わらない／真っ白**に見えない。変化が視覚的に説明できる。 |
| **マイク権限・Listen 同意** | 既存の同意フロー（52）を崩さず、拒否時の案内は短く分かりやすく。 |

---

## 3. 課題の整理（なにが起きているか）

### 3.1 縦長レイアウトと「PC 想定」のずれ

- スペースは **横広のキャンバス**、音声パネルは **フローティング**（56）。縦では **可視領域の縦が短く**、パネルを開くとスペースが **極端に狭く見える**。
- 左バー・下ナビ・安全域（ノッチ）が重なると、**「マイクに集中」モードがない**と感じやすい。

### 3.2 「大画面」が真っ白／変化が分からない

**実装メモ（現行）:** `SpaceScreen` では **アプリ内没入**（`immersiveLayout`・CSS の `containerImmersive` 等）に加え、没入 ON 時に **`document.documentElement.requestFullscreen()`** を **試行**している。失敗時はレイアウトのみ継続（コメントあり）。

**起こりうる原因（仕様上の仮説）:**

| 仮説 | 内容 |
|------|------|
| **Fullscreen API と描画のずれ** | 全画面になった要素ツリーと **背景（星空・壁紙）の描画先**が一致せず、一瞬〜継続で **無地に見える**。 |
| **モバイルブラウザの制限** | iOS Safari 等では Fullscreen の挙動・対象要素が限定的で、**期待と異なるリサイズ**になる。 |
| **没入 OFF の同期** | `fullscreenchange` で没入を落とす処理と、ユーザー操作の競合で **状態だけずれる**。 |

→ **対策の基本方針**は [62](./62_スマホのスペース画面.md) と同じく、**「見た目の正」は CSS 没入**。Fullscreen API は **補助**とし、**真っ白はバグとして潰す**（実装タスク化）。

### 3.3 横持ちを強制しにくい

- **`screen.orientation.lock`** は **HTTPS・ユーザー操作・端末**依存が大きく、**必須機能にしない**（62 と整合）。
- よって **「促す（ソフト）」** と **「レイアウトで吸収（ハード）」** の二段構えが現実的。

## 4. 方針（どうするか）

### 4.1 優先度の原則

1. **まず CSS 没入**でスペース領域を最大化し、**常に意味のある見た目の変化**を出す（62）。
2. **Fullscreen API** は **成功してもしなくても**、ユーザーが **同じ体験**と解釈できる文言・見た目にする。
3. 音声利用が活発なときは **パネル優先のレイアウト**（高さ・初期位置）を検討する（後述フェーズ）。

### 4.2 横持ちの案内（推奨だが強制しない）

**採用してよい案内（例）:**

- Listen ON または **音声パネルを初めて開いた**狭い縦画面で、**一度だけ／控えめに**  
  **「横持ちにするとスペースと文字が見やすいです」** 程度の **バナー・ツールチップ・パネル内1行**。
- **閉じる・再表示しない**トグル付き（うるさくしない）。

**避けること:**

- 全画面モーダルで **回転を必須**にする。
- 回転しないと **Listen をブロック**する（アクセシビリティ・固定端末の都合）。

### 4.3 「スマホ＝マイク」向けのレイアウト（ロードマップ）

| フェーズ | 内容 | 備考 |
|----------|------|------|
| **P0（仕様確定）** | 62 の「大画面＝CSS 優先」と本書の **白画面＝不具合** を共有。コンソール／QA で Fullscreen 失敗を許容挙動として文書化。 | 既存挙動の説明責任 |
| **P1（軽量 UX）** | 上記 **横持ちソフト案内**、大画面トグルに **「スマホでは画面端まで広がります（全画面にならない場合があります）」** 等の短い `title` / 初回のみの説明。 | 実装コスト小 |
| **P1.5（品質）** | **真っ白**再現手順を固定し、`requestFullscreen` の対象・親要素・背景レイヤーを見直し（`#root` vs `documentElement`、背景の付与先）。 | バグフィックス |
| **P2（音声フォーカス）** | モバイルかつ `speechPanelOpen` 時に、**パネルの既定高さ・幅・下寄せ**を「話し手向け」プリセットに（56 のストレージキーと両立）。 | 任意で従来位置に戻せるとよい |
| **P3（検討）** | PWA 表示モード・ステータスバー色など、**長時間 Listen** の快適さ。 | 優先度はプロダクト判断 |

## 5. 受け入れ条件（チェックリスト）

- **縦持ち**で Listen ON → パネルが開き、**文字が読め、候補操作に届く**（52）。
- **大画面トグル**後、**少なくともスペース表示領域が明確に変化**する（変化ゼロは NG）。真っ白のみ表示は **不具合**。
- Fullscreen API が **失敗しても**、アプリが **利用不能にならない**。
- 横持ち案内は **恒久的に邪魔にならない**（閉じられる／頻繁に出ない）。
