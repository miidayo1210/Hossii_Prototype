# 72 Hossii 要約・コメント機能

## この文書の目的

**1 つのスペースに集まった投稿**を踏まえて、Hossii が **短いコメント**や **要約**を出す機能の要件を定義する。コメントは天気予報のような軽いトーンを想定し、要約は「どんなことが書かれているか」を短く伝えることを目指す。

**書き出し画像への載せ方・フッタレイアウト・権限の土台**は [68_スペース書き出し機能 .md](./68_スペース書き出し機能%20.md)（フッタ左「Hossii からの一言」等）と **一体で設計**する。本書は **文言の生成方法**（固定ローテーション／AI）と **プロダクト上の振る舞い**に重点を置く。

**関連:** [68_スペース書き出し機能 .md](./68_スペース書き出し機能%20.md) / [10_ログインユーザー管理設計.md](./10_ログインユーザー管理設計.md)

---

## 概要

| 要素 | 内容 |
|------|------|
| **Hossii コメント** | 「今日の皆さんは〜のようです」のような **天気予報風・前向きな短文**（ハッピーパワー、ラッキーアイテム等も可）。**当面は固定文案を数十案ほど用意しローテーション**する。**将来、AI で生成**する拡張を妨げない設計にする。 |
| **要約** | スペース内の投稿内容を集約し、「どんなことが書かれているか」を **短い段落**で返す。**AI API 連携**を想定するが、**API キーはクライアントに置かず**、Edge Functions 等の **サーバ側**経由とする。 |
| **表示・出力** | アプリ内表示の有無は **別途 UI 仕様で確定**（本書では「書き出しに載せる」を必須要件とする）。**書き出し PNG** では、**メインキャプチャの下・フッタ帯**に配置し、**QR の左**にテキストブロックを置く（68 のレイアウト方針に従う）。 |

---

## 実装の重さ（前提）

| 領域 | 見積もりの目安 |
|------|----------------|
| **固定コメントのローテーション** | 軽い。文案配列＋決定的または疑似乱数の選択（スペース ID・日付・書き出し時刻などで日替わりにしてもよい）。初期は **バックエンド不要**でよい。 |
| **書き出しへの合成** | 中程度。既存の canvas 合成（`exportSpaceCanvasWithFrame` 相当）に **左ブロックの高さ・折り返し・文字数上限**を足す。日本語の **折り返し・フォント**は調整の余地あり。 |
| **AI 要約** | 重めになりやすい。**認証・レート制限・コスト・失敗時・個人情報／投稿本文の取り扱い**が絡む。事前生成とキャッシュ、または書き出し前の **明示的な待ち時間**のどちらか（または併用）を仕様で決める。 |

## フェーズ分け（推奨）

### フェーズ 1（MVP）

- **コメント**: 文案 **約 50 案**（プロダクト側で用意。必要なら後から差し替え）を **ローテーション**。
- **要約**: **未実装またはプレースホルダ**（例: 「みんなの気持ちが集まっています」程度の **固定短文**、または **非表示**）。AI は接続しない。
- **書き出し**: 68 の **フッタ左ブロック**に、**コメント（＋任意で要約プレースホルダ）**を載せられる状態にする（68 の「一言」データモデルと **矛盾しないよう**、同一フィールドを流用するか、**コメント用／要約用を分けるか**は実装設計で決定。下記「データモデル」参照）。

### フェーズ 2

- **要約**: **サーバ側**で AI API を呼び、投稿本文を **トークン上限内**に切り詰めて要約。**空スペース・短文のみ**のときは **固定フォールバック**。
- **キャッシュ**: スペース単位・投稿更新をトリガに **再生成**するか、**書き出し直前のみ生成**するかを決める（コストと鮮度のトレードオフ）。
- **UX**: 生成中の **ローディング**、**キャンセル**、**失敗時メッセージ**（書き出し 68 の `AbortSignal` と整合）。

### フェーズ 3（任意）

- **コメントも AI 化**（トーン指定: 天気予報風・前向き）。**フェーズ 1 の文案**をフォールバックに残すと安全。
- 管理者向け **下書き提案 → 承認・編集**（68 にある「管理者手入力優先」との統合）。

## コメント（天気予報風）の要件

| 項目 | 方針 |
|------|------|
| トーン | **軽く前向き**。断定しすぎず、「〜のようです」系の **比喩・雰囲気**でよい。 |
| 内容例 | 「今日の皆さんは〜」「ハッピーパワーいっぱい」「ラッキーアイテムは〜」等。 |
| 文案 | **約 50 案**を初期セットとする。追加・差し替えは **定数または CMS／設定**で可能にしてよい。 |
| 選択ロジック | **同一スペースで毎回同じ**にするか、**日付で変える**かはプロダクト判断（書き出しの再現性が必要なら日付＋スペース ID のハッシュ等が扱いやすい）。 |

---

## 要約（AI）の要件

| 項目 | 方針 |
|------|------|
| 入力 | 書き出し対象と **同じ集合**（表示中・権限内の投稿テキスト等）を **上限件数・文字数**で切り詰めて送る。 |
| 出力 | **1〜3 文程度**の短文。個人名の **過度な推測・特定**を避けるプロンプト方針とする（詳細はプロンプト設計書または運用で補足）。 |
| API | **ブラウザから直接**外部 LLM を呼ばない。**Edge Function 等**にキーを置き、**認可済みユーザのみ**呼べるようにする。 |
| 失敗時 | **固定フォールバック文**、または **要約ブロック非表示**（コメントのみ表示）を仕様で決める。 |
| コスト | **キャッシュ**、**再生成条件**、**レート制限**を検討する。 |

---

## 書き出しとの関係

- **位置**: **出力キャンバス（メイン領域）の下**、**フッタ帯**。**QR ブロックの左**にテキスト（68 と同じレイアウト思想）。
- **構成案**: **見出し＋コメント行＋（任意）要約段落**のように **縦に積む**。幅が狭いときは **改行・行数上限・文字数上限**（68 の目安と揃えてよい）。
- **68 との役割分担**: 68 が **枠・権限・「一言」フィールド**を定義する。本機能は **その本文をどう生成するか**（固定／AI／管理者上書き）を追加定義する。**管理者が手入力した値を最優先**するかどうかは 68 の方針に **従う**（従う場合、AI は **未入力時のみ**または **提案として**に留める）。

## データモデル（案）

68 の `export_footer_blurb` 等と **二重管理にならないよう**、次のいずれかを採用する（実装時に 1 つに決定する）。

| 案 | 内容 |
|----|------|
| **A. 単一フィールド** | 表示用は **1 つの本文**にし、ソースは「管理者手動 ＞ キャッシュ済み AI 要約 ＞ 固定コメント」の **優先順位**で決定。 |
| **B. 分割** | `footer_comment`（ローテーション／AI コメント）と `footer_summary`（AI 要約）を分け、書き出しで **上下に並べる**。管理者用は 68 の手入力と **マージルール**を定義。 |

## 非機能・コンプライアンス

- 投稿本文を外部 API に送るため、**利用規約・プライバシー**での説明範囲を確認する。
- **エラーログ**に本文を **残さない**、または **マスク**する方針を推奨。

## 未決定・要確認

- アプリ内（書き出し以外）に **同じコメント／要約を出すか**。
- コメントと要約を **常に両方**載せるか、**トグル**か。
- ローテーション文案の **初回セット**の担当（プロダクト文案として本リポジトリに置くか、別管理か）。
