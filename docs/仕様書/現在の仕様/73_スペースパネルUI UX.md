# 73 スペースパネル UI/UX（投稿・ログ・音声）

スペース画面（`SpaceScreen`）で、**投稿**・**ログ一覧**・**音声入力**のために重ね表示する UI の **目標仕様** と、現行実装の **参考メモ** を集約する。

> **関連:** [56_クリックフィードバックとフローティングパネル.md](./56_クリックフィードバックとフローティングパネル.md)（`FloatingPanelShell`） / [62_スマホのスペース画面.md](./62_スマホのスペース画面.md)（左バー・下ナビ） / [71_スマホマイク機能.md](./71_スマホマイク機能.md)（Listen・音声・横持ち案内） / [69_投稿のUI設定.md](./69_投稿のUI設定.md)（`PostScreen`）  
> **注意:** 下ナビ「ログ」の遷移先は従来 [62](./62_スマホのスペース画面.md) どおり **`#comments` ページ**とする。**本仕様はその動線を残したうえで**、スペース上に **クイックログ** を追加する（併存）。

---

## 1. 用語・前提

| 用語 | 意味 |
|------|------|
| **三パネル** | クイック投稿・クイックログ・音声パネルの総称 |
| **クイック投稿** | スペース上で開く投稿 UI（`PostScreen`） |
| **クイックログ** | スペース上で開くログ一覧（`LogListBody`。`#comments` の全画面ログとは別） |
| **音声パネル** | `SpeechPanel`（認識結果・候補・Listen 等）。`ScaledContent` 外へ portal 可 |
| **キューブドック** | 画面右の「投稿 / ログ / 音声」3 ボタン（`SpacePanelCubeDock`）。**主に広いビューポート向け** |
| **下シート** | 画面下辺から **スライドイン**（みょーんと伸びる）して重なる UI。角丸・スクイームはデザイン任せ |
| **スマホ狭** | ビューポート幅 **`max-width: 768px`** の状態（現行 `isMobile` と一致しやすい） |
| **スマホ横** | 狭スマホを **ランドスケープ**にした状態で、論理幅が広がったケース等（**PC 列をベースにしつつスマホ横向けに最適化**する対象） |
| **タブレット帯** | **768px を超え、かつデスクトップの広幅ではない**レンジ。数値閾値は **実装時に定義**（例: `769px`〜`1024px` を第一候補とし、`pointer` / `hover` と組み合わせて **スマホ横** と誤判定しないよう調整） |
| **固定シート／固定パネル** | **ドラッグ・リサイズ不要**（`FloatingPanelShell` に依存しない実装を許容）。スマホ狭における三パネルの既定 |

---

## 2. 目標: 三パネル共通の原則

| 原則 | 内容 |
|------|------|
| **レイヤー** | スペース本体より手前。モーダル（同意・書き出し中等）との **z-index 順序は別表で固定**する（実装タスク） |
| **スマホ狭の操作感** | 三パネルは **固定シート／固定パネルでよい**（動かせなくてよい）。必要なら閉じる・スワイプで下に退けるのみ |
| **キーボード（ソフトウェア）** | 入力フォーカス時は、下シートごと **上に押し上げる**（`visualViewport` 等で可視領域に収める）。入力欄がキーボードに隠れないことを優先 |
| **訪問モード** | 他人スペース閲覧中は **三パネルはいずれも出さない**（開かない・トグルも出さない）。既存の閲覧 UI のみ |
| **没入（大画面）モード** | **三パネルは表示してよい**（禁止しない） |
| **アニメーション** | 下からの表示は **スライドイン**（みょーんと伸びる）を基本トーンとする |

---

## 3. 目標: ブレークポイントとレイアウト一覧

**確定:** 狭さの第一閾界は **768px**。**タブレット帯の上限幅**および **スマホ横とタブレットの区別**は、実装フェーズで `matchMedia` の組み合わせ（幅・向き・`pointer` 等）により定義し、本表の意図と齟齬が出ないように検証する。

| 区分 | 向き | クイック投稿 | クイックログ | 音声（目安） |
|------|------|--------------|--------------|--------------|
| **スマホ狭** | **縦（ポートレート）** | **ダブルタップ**で **下からスライドイン**の下シート | **下ナビから `#comments` へ飛ぶ動線に加え**、スペース上から **下にスライドイン**。占有は **画面の下半分**を目安 | 下シート系で統一しやすい（詳細は 71 と整合） |
| **スマホ狭** | **横（ランドスケープ）かつ幅が狭い**（≤768 のまま等） | **下からスライドイン**の下シート。可視高に合わせて高さ調整 | **画面右 約 1/3** を目安（縦の「下半分」と対になる横レイアウト） | 下シートまたは右寄せパネルで統一しやすい |
| **スマホ横（広がった状態）** | ランドスケープで幅が広い（典型: **幅 >768px**） | **PC 列（サイド等）をベース**にするが、**PC と完全同一にはしない**。タッチターゲット・余白・密度を **スマホ横向けに最適化** | **画面右 約 1/3** の領域（ドロワ／パネル）を目安 | PC 列ベース＋最適化 |
| **タブレット帯** | **縦** | **下シート** | **下シート** | 下シート |
| **タブレット帯** | **横** | **サイドパネル** | **サイドパネル** | サイド（PC に近いがタブレット横向けに調整可） |
| **PC（広いディスプレイ）** | — | 現行どおり（第 5 節参照） | 現行どおり | 現行どおり |

**補足（ログの動線）:** クイックログは **一覧の素早い参照**用。フル機能のログ閲覧・フィルタは **`#comments` ページ**も継続利用できること。

## 5. PC（広いディスプレイ）— 現状を当面の正とする節

主に **幅が十分大きい**デスクトップ向け。実装の概要は以下（詳細はコード参照）。

| パネル | 開き方（主な導線） | レイアウト・操作感 |
|--------|-------------------|---------------------|
| **クイック投稿** | スペース背景の **ダブルクリック**、右 **キューブ「投稿」**、**Ctrl+N** / **⌘+⌥+N** | `panelMode="side"`。右寄せ縦長。`FloatingPanelShell`＋ドラッグ可 |
| **クイックログ** | 背景 **トリプルクリック**、右 **キューブ「ログ」** | `LogListBody`＋サイド。同上 |
| **音声** | 左バー Listen、右 **キューブ**、**Listening** インジケータ | フローティング既定矩形。同上 |

**目標との関係:** **タブレット横**では本節に近い **サイド**を正とするが、**スマホ横（広）**では見た目は近づけつつ **§3 の「最適化」** を必須とする（完全コピー禁止）。

## 6. 現行実装メモ（参考・2026-04 時点）

実装は段階的に §2〜§4 に寄せる。**現状**の要点のみ残す。

| 項目 | 現状 |
|------|------|
| ブレーク | 主に **`max-width: 768px`** の `isMobile`。横持ちで幅が広がると PC 列に切り替わる |
| スマホ狭のシェル | `FloatingPanelShell`＋矩形永続化。**目標ではスマホ狭は固定シート化してよい**（§2） |
| クイックログ（狭） | トリプルクリック無効・キューブ非表示で **オープン導線が弱い** → §3 で解消方針を定義済み |
| クイック投稿（狭） | `dblclick`＋下寄せ `panelMode="bottom"`。**目標では明示的に下シート＋スライドイン**（§3） |
| 下端余白 | 幅 ≤768px で **72px** インセット（下ナビ）。キーボード押し上げは **未整備の可能性** → §2 |
| CSS | 没入時の **landscape** 用 safe-area 調整あり（`SpaceScreen.module.css`） |

| # | 項目 | 向け先 |

## 8. 今後この文書で追記したい細目

- 三パネルと **モーダル・バナー**の **z-index 順序表**（固定）
- **スマホ横**と **タブレット帯**を誤判定しないための **媒体クエリ一覧**（確定値）
- **閉じる**操作（スワイプ下げ・バックキー・スクリムタップ）の統一
- **アクセシビリティ**（フォーカストラップ、Esc、`aria-modal`）
- **スライドショー**時の扱い（訪問モードと同様に隠すかは別途決定）
