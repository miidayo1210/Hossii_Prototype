# Hossii 外部物理ボタン入力テスト仕様書
## Arduino からの "Hello" 信号を受信して「こんにちは」を投稿する

### 1. 目的
Arduino に接続された物理ボタンの入力を、Hossii の Web アプリに接続し、外部入力によって投稿動作を実行できるかを検証する。

今回は最小構成のテストとして、以下のみを対象とする。

- Arduino 側でボタン1が押されたとき、シリアル通信で `"Hello"` を送信する
- Hossii 側で `"Hello"` を受信したとき、投稿文として `こんにちは` を送信する

---

### 2. テストの範囲
今回の実装対象は以下に限定する。

- 物理ボタン1つのみ
- 受信文字列は `"Hello"` のみ
- 投稿文は `こんにちは` のみ
- Hossii の既存UIを大きく変更しない
- テスト用の接続ボタン追加は可

今回の範囲外：
- 複数ボタン対応
- `"わーい"` や `"ウフフ"` など他文言対応
- 常設向けの本実装
- セキュリティや接続再試行の高度な実装
- スマホ対応
- Arduino からの双方向通信

---

### 3. 想定動作
#### 3.1 Arduino 側
- 物理ボタン1が押される
- Arduino が USB シリアル通信で `"Hello"` を送信する

#### 3.2 Hossii 側
- ブラウザ上で Web Serial API により Arduino と接続する
- `"Hello"` を受信したら、Hossii 上で以下を実行する
  1. 投稿内容を `こんにちは` にする
  2. 「気持ちをおく」ボタンと同等の既存投稿処理を実行する

---

### 4. 実装方針
#### 4.1 接続方式
- Web Serial API を使用する
- 対応ブラウザは Chrome 系を前提とする
- 初回接続はユーザー操作（接続ボタン押下）で行う

#### 4.2 実装の考え方
その場しのぎで DOM を直接いじるのではなく、可能であれば既存の投稿処理を再利用できる形に整理する。

理想形（当初案）：
- `submitFeeling(text: string)` のような関数を用意する
- 通常UIからの投稿も、Arduino経由の投稿も、この関数を通す

実装では同等の役割として `PostScreen` 内の **`submitBubbleHossii(input: AddHossiiInput)`** に集約した（下記 §12 参照）。

#### 4.3 最小要件
最小構成として、以下が満たされればよい。

- 「接続」ボタンがある
- 接続後、Arduino のシリアル通信を読み続ける
- `"Hello"` を受信したら `こんにちは` を投稿する

---

### 5. 必須仕様
#### 5.1 シリアル接続
- ユーザーが「接続」ボタンを押すと、シリアルポート選択ダイアログを開く
- Arduino のポートを選択して接続する
- baudRate は `9600` とする

#### 5.2 受信処理
- シリアル受信データを文字列として読む
- 改行コードが含まれる可能性があるため、必要に応じて trim する
- `"Hello"` を受信した場合のみ処理する

#### 5.3 投稿処理
- `"Hello"` 受信時に `こんにちは` を投稿する
- 可能なら既存の投稿処理関数を呼び出す
- 既存の「気持ちをおく」ボタン押下時と同じ結果になること

---

### 6. UI要件
- テスト用に「接続」ボタンを追加してよい
- 既存デザインを大きく壊さないこと
- 接続状態がわかる簡単な表示があると望ましい
  - 例：未接続 / 接続中 / 接続済み

---

### 7. 例外・エラー時の扱い
最低限、以下を考慮すること。

- ポート未選択時は何もしない
- 接続失敗時は console にエラーを出す
- 受信データが `"Hello"` 以外なら無視する
- 読み取りが終了した場合はループを抜ける

---

### 8. 動作確認項目
以下が確認できれば成功とする。

1. Arduino の Serial Monitor では `"Hello"` が送信されている
2. Hossii の「接続」ボタンから Arduino に接続できる
3. 物理ボタン押下時にブラウザ側で `"Hello"` を受信できる
4. `"Hello"` 受信時に `こんにちは` が投稿される
5. 通常の Hossii 投稿機能が壊れていない

---

### 9. 実装上の注意
- Web Serial API を使うため、Chrome 系ブラウザ前提
- ユーザー操作なしで自動接続しようとしない
- DOM を直接無理やり操作するより、投稿処理の共通関数化を優先する
- 今後の拡張（複数コマンド対応）を見越して、受信文字列に応じた分岐構造にしておくとよい

例：
- `"Hello"` -> `こんにちは`
- 将来的に `"Yay"` -> `わーい`
- 将来的に `"Ufufu"` -> `ウフフ`

### 12. 実装済み内容（コード・挙動）

実装日の目安: 2026年4月。次のファイルが本仕様のテスト実装の本体である。

| ファイル | 役割 |
|---------|------|
| `src/components/PostScreen/PostScreen.tsx` | Web Serial UI（接続／切断・状態表示）、受信行のハンドラ、`submitBubbleHossii` および Arduino 経路の `submitFromArduinoRef` |
| `src/core/utils/webSerialArduino.ts` | `SERIAL_LINE_TO_MESSAGE`（`Hello` → `こんにちは`）、`messageForSerialLine`、`readSerialLinesFromPort`（改行分割・終了時バッファフラッシュ・`abort` 時 `reader.cancel`）、`requestAndOpenSerialPort`（9600） |
| `src/web-serial.d.ts` | TypeScript 用の Web Serial 最小型宣言（`SerialPort` / `Navigator.serial`） |
| `src/components/PostScreen/PostScreen.module.css` | テスト用シリアル行（`.serialTestRow` など）のスタイル |

#### 12.1 投稿の共通化

- **`submitBubbleHossii`** … `savePostBubbleColorDraft` → `addHossii` →（ログイン時）スタンプ・成功トースト、の順で実行する。従来「気持ちを置く」と同じ核。
- **通常の手動投稿** … `handleSubmit` が検証・画像アップロード・位置計算の後に `submitBubbleHossii` を呼ぶ。送信後のフォームクリア・パネル閉じ・`navigate('screen')` は従来どおり `handleSubmit` 側に残す。

#### 12.2 Arduino（シリアル）経路

- `navigator.serial` が使える環境（主に Chrome 系）でのみ、投稿画面タイトル下に **「Arduino（テスト）」** 行を表示（接続／切断、未接続・接続中・接続済み）。
- 受信行は trim 後 **`Hello` のみ** マッチし、そのとき **`こんにちは`** を本文にしたバブル投稿を行う。
- 付与するメタデータ: **本文由来のハッシュタグのみ**（`parseHashtags('こんにちは')`）。感情・画像・数値・プリセットタグは付けない。吹き出し色・形状・位置は **画面上の選択**（`canvasPosition` 等）に合わせる。
- **Arduino 経路では画面遷移しない・フォームをクリアしない**（バックグラウンド投稿）。
- 連打対策: `serialPostLockRef` と `sendingRef` / `setSending` によるガード。

#### 12.3 未実装・今後の改善候補（コードレビューで挙がったもの）

- アンマウント後に読み取りループの `finally` で `setSerialUi` が走る可能性への対策（`mountedRef` など）。
- `requestPort` 成功後に `port.open()` が失敗した場合の `port.close()` を `catch` で試すこと。
- 上記は本仕様の「範囲外」には含めず、品質改善タスクとして残してよい。

### 13. Push前コードレビュー メモ

レビュー観点: 既存通常投稿への影響、Web Serial の安全性、投稿処理の妥当性、ロック・連打、コード品質・拡張余地。実コードに基づく記録。

#### 13.1 重大な問題

- **なし**（当時のレビュー範囲では、既存の手動投稿フローを破壊する明確な欠陥は見つからなかった）。

#### 13.2 Push 前に直した方がよい問題（品質・リーク対策）

1. **アンマウント後の `setState`** … 読み取りループ終了時の `setSerialUi('idle')` が、アンマウント後に実行されると React の警告や無駄更新の原因になりうる。`mountedRef` でガードするなどの対応を推奨。
2. **`open()` 失敗時のポート** … `requestPort` 成功後に `open()` が失敗した場合、`catch` で `port.close()` を試すと安全。
3. **`web-serial.d.ts` の未使用型** … `SerialPortInfo` が未使用なら削除して整理可能（動作への影響は小さい）。

#### 13.3 そのままでもよい改善点

- `isWebSerialSupported` を `navigator.serial?.requestPort` まで厳密化する（エッジケース対策）。
- `onLine` で既にコマンド判定済みのため、Arduino 投稿本文はマップの戻り値に寄せて重複を減らせる（将来 BTN 別コマンド時に有効）。
- `serialPostLockRef` と `sendingRef` の二段は、再入と手動投稿との競合防止として意図的で矛盾はない、との評価。

#### 13.4 結論（当時）

- **デモ／テスト用途としては Push 可**。
- **本番相当の堅牢性**を求める場合は、§13.2 の **アンマウント時の状態更新ガード** と **`open` 失敗時の `port.close()`** を入れてから Push する方が望ましい、とした。
