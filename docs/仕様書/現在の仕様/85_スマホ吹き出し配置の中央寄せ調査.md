# 85 吹き出し配置の中央寄せ修正

> **分類:** `[Bug / UX]` スマホ・PC で投稿・星の座標が画面中央に偏る問題の修正
> **関連:**
> - [78_スペース投稿配置と16比9領域](./78_スペース投稿配置と16比9領域.md)
> - [45_投稿の位置を定める機能](./45_投稿の位置を定める機能.md)
> - [84_スペース表示モード拡張](./84_スペース表示モード拡張.md)

> **ステータス:** ✅ 実装済み

---

## 1. 目的

スマホ（および PC ランダム配置）で、投稿・星が **X/Y とも画面中央付近に偏って見える** 問題を修正する。

| 症状 | 例 |
|------|-----|
| ランダム配置 | 10 件投稿しても星が **中央帯に帯状** に集まる |
| ダブルタップ投稿 | タップ位置と表示位置が **ずれる**（特に Y） |
| 上下の余白 | シャープ矩形 **内** でも上下が空いて中央に密集 |

[78](./78_スペース投稿配置と16比9領域.md) は **レターボックス外** へのはみ出しを防いだ。本仕様は **矩形内の分布の偏り** と **座標系の不整合** を直す。

---

## 2. 原因（調査結果）

| ID | 原因 | 深刻度 | 説明 |
|----|------|--------|------|
| **A** | 乱数の平均化 | **高** | `createBubblePosition` / `createBubblePositionInSharp` が `(r1+r2)/2` で X/Y を決めており、分布が **中央に尖る**（三角分布） |
| **B** | ダブルタップ座標系 | **高** | タップは **コンテナ %** で保存されるが、表示時に **論理 %** として `mapLogicalToContainerPercent` され **二重解釈** される |
| **C** | Y レンジが狭い | **中** | PC 論理 Y が 12〜78%（66% 幅）で X（8〜92%）より狭く、縦方向も中央帯に偏りやすい |
| **D** | ナビ経路の中央固定 | **高** | 下ナビ・TopRightMenu からの `openQuickPost()` が **`{ x: 50, y: 50 }` + `isPositionFixed: true`** を常に付与。モバイルの大半の投稿がこの経路のため、**X 軸中央に帯状集中**（A〜C 修正後も残存） |

**採用した修正:** A・B を必須、C を Y レンジ統一で対応（§4.3）。**D は §4.6**（`quickPostOpen` / `quickPostPos` 分離）。

---

## 3. 座標系の契約（修正後）

[78](./78_スペース投稿配置と16比9領域.md) §4.3 を **明確化** する。

| 条件 | DB の `positionX/Y` の意味 | 表示時 |
|------|---------------------------|--------|
| モバイル **かつ** 画像壁紙あり | **シャープ矩形内の論理 %**（0〜100） | `mapLogicalToContainerPercent` → コンテナ % |
| 上記以外（PC・単色背景等） | **コンテナ全面の %**（0〜100） | 変換なし |

**クイック投稿（ダブルタップ）** も上表に従い、モバイル壁紙時は **保存前に論理 % へ正規化** する。

```
タップ（コンテナ px）
  → [モバイル壁紙] mapContainerPercentToLogical
  → DB 保存（論理 lx, ly）
  → 表示 mapLogicalToContainerPercent
  → StarView / Bubble（コンテナ %）
```

## 4. 修正仕様

### 4.1 乱数分布の一様化（原因 A）

**対象:** [`bubblePosition.ts`](../../../src/core/utils/bubblePosition.ts) の `createBubblePosition` / `createBubblePositionInSharp`

**変更前:**

```typescript
const x = MIN + ((r1 + r2) / 2) * SPAN;
const y = MIN + ((r2 + (1 - r1)) / 2) * SPAN;
```

**変更後（実装済み）:**

```typescript
const x = 8 + r1 * 84;
const y = 8 + r2 * 84;
```

| 関数 | X レンジ（論理 %） | Y レンジ（論理 %） |
|------|-------------------|-------------------|
| `createBubblePosition` | 8〜92 | **8〜92**（§4.3 で C も解消） |
| `createBubblePositionInSharp` | 5〜95 | 5〜95 |

| 項目 | 仕様 |
|------|------|
| 影響範囲 | 新規投稿の `addHossii` 初期座標、ランダム並びの未固定投稿、インデックスからのフォールバック配置 |
| 既存 DB | **マイグレーションしない**。既存 `positionX/Y` はそのまま |
| 投稿順・byAuthor | **変更なし**（格子配置は均等分割のため） |
| 決定論 | `index` 由来の seed は **維持**（同じ index なら同じ位置。分布形状だけが一様になる） |

### 4.2 逆変換 API の追加（原因 B の前提）

**実装:** [`sharpContentRect.ts`](../../../src/core/utils/sharpContentRect.ts)

```typescript
/**
 * コンテナ基準 % → シャープ矩形内論理 %（78 §4.3 の逆変換）
 * シャープ外タップは 0〜100 に clamp
 */
export function mapContainerPercentToLogical(
  dx: number,
  dy: number,
  containerW: number,
  containerH: number,
  aspectRatio = 16 / 9,
): { x: number; y: number };
```

単体テスト: [`sharpContentRect.test.ts`](../../../src/core/utils/sharpContentRect.test.ts) — シャープ中心タップ → (50, 50)、四隅付近 → 0 または 100 付近、往復誤差。

### 4.3 Y レンジの X との統一（原因 C）

`createBubblePosition` の Y を **12〜78% から 8〜92%** に変更（X と同一バンド）。

| 項目 | 変更前 | 変更後 |
|------|--------|--------|
| `createBubblePosition` Y | `12 + avg * 66` | `8 + r2 * 84` |

`createBubblePositionInSharp` はもともと 5〜95 で X/Y 対称のため **レンジ変更のみ不要**（4.1 の一様化のみ）。

> **注:** `createOrderedBubblePosition`（投稿順格子）は従来どおり Y 12〜78% 帯内の等分割。ランダム配置のみ本修正の対象。

### 4.4 ダブルタップ → クイック投稿の正規化（原因 B）

**対象:** [`SpaceScreen.tsx`](../../../src/components/SpaceScreen/SpaceScreen.tsx) `handleDoubleClick`

| ステップ | 処理 |
|---------|------|
| 1 | `bubbleArea` の `getBoundingClientRect` から **コンテナ %** `(cx, cy)` を算出 |
| 2 | `shouldMapToSharp && rect.width/height > 0` のとき `mapContainerPercentToLogical(cx, cy, …)` → `(lx, ly)` |
| 3 | `setQuickPostOpen(true)` および `setQuickPostPos({ x: lx, y: ly })`（論理 %）または PC 時は `{ x: cx, y: cy }` |
| 4 | [`PostScreen`](../../../src/components/PostScreen/PostScreen.tsx) `panelMode` 投稿確定時、`positionX/Y = initialPosition`、`isPositionFixed: true` |
| 5 | 表示時は `mapDisplayPos` → 論理 % として正しくコンテナ % に戻る |

### 4.5 ドラッグ保存座標

**実装:** `SpaceScreen.tsx` の `toStoragePos` + `handlePositionSave`

| 条件 | 保存値 |
|------|--------|
| モバイル壁紙 | ドラッグ終了時の表示 % を `mapContainerPercentToLogical` で **論理 % に逆変換してから** DB 保存 |
| PC・単色背景 | 表示 % をそのまま保存 |

`Bubble` の `onPositionSave` → `handlePositionSave` → `updateHossiiPositionAction` 経路で統一。

### 4.6 クイック投稿の開閉と座標の分離（原因 D）

**問題:** [51](./51_簡単投稿.md) / [76](./76_スマホ投稿ナビとクイック投稿.md) ではナビ経路の既定座標を `{ x: 50, y: 50 }` としていた。パネル表示の有無と座標指定が **`quickPostPos` 1 本** に束ねられていたため、ナビから開くたびに **中央固定投稿** になっていた。

**修正（実装済み）:** `SpaceScreen.tsx` で state を分離する。

| state | 役割 |
|-------|------|
| `quickPostOpen` | パネル表示の ON/OFF。下ナビ・TopRightMenu・imperative API は **こちらのみ** 操作 |
| `quickPostPos` | **ダブルタップ時のみ** 設定。`PostScreen` の `initialPosition` に渡す |

| 経路 | `quickPostOpen` | `quickPostPos` | 投稿時 |
|------|-----------------|----------------|--------|
| 下ナビ / TopRightMenu / 音声候補編集 | `true` | **設定しない**（`null` のまま） | `initialPosition` なし → `isPositionFixed: false` → ID シードのランダム配置（[87](./87_スペース表示パフォーマンス最適化.md) §6） |
| 背景ダブルタップ | `true` | タップ座標（§4.4 で論理 % 正規化） | `isPositionFixed: true`、指定座標に表示 |
| パネル閉じる | `false` | `null` | — |

**レイアウト判定:** `mobilePostSplit` / `mobilePostLandscapeSplit` は **`quickPostOpen`** を参照（`quickPostPos` ではない）。

**既存 DB:** 原因 D 修正前にナビ経路で投稿した **`isPositionFixed: true` かつ (50, 50)** のレコードは **マイグレーションしない**（§7 と同様）。

## 10. 調査メモ（参考）

### 10.1 コード根拠（原因 A）

[`HossiiStoreProvider.tsx`](../../../src/core/hooks/HossiiStoreProvider.tsx): `addHossii` デフォルト座標に `createBubblePosition` を使用。

### 10.2 78 実装後も中央に見えた理由

78 はシャープ **外** を防いだのみ。`(r1+r2)/2` による **内側の中央尖り** は未修正だった。

### 10.3 A〜C 修正後も中央に見えた理由（原因 D）

乱数一様化・座標正規化（§4.1〜4.5）後も、**ナビ経路のクイック投稿** が `{ x: 50, y: 50 }` 固定のままだった。モバイルでは [76](./76_スマホ投稿ナビとクイック投稿.md) 経路が主であるため、実機では **X=50 の垂直帯** に投稿が積み上がって見えた。§4.6 で解消。
