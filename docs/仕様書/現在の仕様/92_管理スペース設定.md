## 0. 本指示書に対する評価（所感）

再構成指示の方向性は **妥当** です。現状コードを横断すると、指示書が挙げる問題はすべて実在します。

特に有効な判断:

| 判断 | 理由（コード上の裏付け） |
|------|------------------------|
| 基本設定タブの解体 | `GeneralTab` に 9 セクション・3 系統のデータ源が混在 |
| 投稿 ON/OFF と投稿フォームの統合 | `postFieldSettings.ts` が既に `features.*` からのフォールバックを持ち、二重管理の芽がある |
| Feature Flags の役割分離 | 8 キーのうち 2 つは未接続、1 つは UI 非掲載、いいねは別設定と競合 |
| 保存 UX の統一 | グローバル保存 / タブ内即時 / 偽保存ボタン（タグ）が混在 |
| カードタイプ廃止 | `Space.cardType` は表示切替に未使用。`viewMode` は参加者 UI 側の `displayPrefs` |

**確定済み（2026-06-26）:**

1. **権限モデル [Q1]**: 今回の再構成は **コミュニティ管理者（`isAdmin`）のみ** が管理画面にアクセス。スペース単位 RBAC は **独立タスク**（§2.2）。
2. **メッセージ vs 投稿へのコメント [Q2]**: 用語を分離。`commentPost` は UI 上「メッセージ」。投稿へのコメント機能は **未実装・将来機能**（§2.3, §5.4.5）。
3. **スペースモード配置**: **案 A 採用** — サイドバー最上部の独立画面（§4.2）。

**実装前に設計が必要な点（本書内で構造案あり）:**

- 中心キャラクター第 1 段階: `characterName` フィールド新規追加
- スペースモード保存: `SpaceSettings.mode`（§7.4）

## 1. 背景と目的

### 1.1 現在の問題

| # | 問題 | 現状 |
|---|------|------|
| P1 | 基本設定タブの混在 | 名前・機能 ON/OFF・権限・非公開・見た目・文言・カードタイプが 1 タブ |
| P2 | 投稿設定の重複 | `features.emotionPost/photoPost/numberPost` と `postFields.*` が並立。`commentPost` は `message` のフォールバック |
| P3 | 保存方式の混在 | ヘッダー一括保存 / タブ内即時（PostForm, FF, 隣人）/ タグの偽保存 |
| P4 | データ分散 | `Space` / `SpaceSettings` / `space_feature_flags` / localStorage |
| P5 | 名称の二重管理 | `Space.name` と `SpaceSettings.spaceName`、`Space.cardType` と `SpaceSettings.cardType` |
| P6 | Feature Flags の役割混乱 | 表示品質・未完成機能・管理者設定・利用目的プリセットが同居 |
| P7 | 開発用語の露出 | `Feature Flags` タブ名、未実装トグル（`public_board_mode`, `zine_export_enabled`） |
| P8 | 権限表示の曖昧さ | 一部タブだけ `isAdmin` ガード。設定画面はコミュニティ管理者のみ到達（§2.2） |
| P9 | UI パターンのバラつき | タブごとに `.container` / `.section` / トグル / テーブルのスタイルが独立 CSS。見た目は似るが共通化なし |
| P10 | ナビのフラットさ | 10 タブが同列でグループ感がなく、目的別の見通しが悪い |
| P11 | ヘッダー保存の分かりにくさ | どのタブの変更が保存されるか不明。PostForm だけ即時保存でルールが混在 |
| P12 | 偽 UI 要素 | タグタブの「保存」ボタン（実保存なし）、Feature Flags タブ名など |

### 1.2 再構成の目的

- スペース管理者が **「何のための設定か」** を理解できる情報設計にする
- **個別設定（Space Settings）を正本** とし、スペースモードは一括適用のプリセットに限定する
- Feature Flags を **Hossii 運営者向け Internal Feature Flag** に再定義する
- 保存 UX を **明示的保存** と **即時保存** の 2 種に統一する
- **UI パターンを共通化** し、既存の宇宙テーマを活かしたまま見やすさ・操作の予測可能性を上げる（§15）

### 1.3 設計原則

1. **正本は一つ**: 各動作を決める設定キーは 1 箇所に集約する
2. **モードは上書きプリセット**: スペースモードは永続的な強制ではなく、適用後は個別変更可能
3. **未完成は Internal FF**: 一般管理 UI から隠し、`正式設定 AND Internal FF` で段階公開
4. **用語は参加者向け**: 管理 UI に `Feature Flag` 等の開発用語を出さない
5. **段階移行**: 既存データ・旧フィールドは後方互換を保ちながら廃止
6. **UI は小さく統一**: 全面リデザインしない。既存 CSS を **1 つの共通モジュールに寄せる** 程度に留め、コンポーネント追加は最小限（§15.1）

## 2. 用語定義

| 用語 | 定義 |
|------|------|
| **スペース管理者** | 当該スペースの設定を変更できるユーザー（**将来の目標用語**。§2.2 参照） |
| **コミュニティ管理者** | コミュニティが承認済み（`communities.status === 'approved'`）のユーザー。`currentUser.isAdmin === true`。**現行・今回の再構成で管理画面を操作する主体** |
| **Hossii 運営者** | プロダクト全体を管理する内部運営者。Internal Feature Flag・Kill Switch を操作 |
| **一般参加者** | スペースを閲覧・投稿する通常ユーザー。スペース管理画面にはアクセス不可 |
| **メッセージ投稿** | 投稿パネルで本文テキストを入力する機能。UI 表記は「メッセージ」（§2.3） |
| **投稿へのコメント** | 既存投稿に対する返信・コメント機能。**現状未実装**（§2.3, §5.4.5） |
| **スペースモード** | 利用目的に応じた設定の一括プリセット。適用後も個別設定で上書き可能 |
| **個別設定** | スペースモード適用後も保持・変更できる各項目。正本は `SpaceSettings` + `Space` |
| **Internal Feature Flag** | 開発・段階公開用。`space_feature_flags` または環境変数。一般管理 UI 非表示 |

### 2.2 権限モデル [Q1 確定]

#### 2.2.1 現行権限（コード上の事実）

| 主体 | 判定 | 管理画面（`#settings`） | 備考 |
|------|------|-------------------------|------|
| コミュニティ管理者 | `currentUser.isAdmin === true` | ✅ アクセス可（`SpacesScreen` 歯車 → `#settings`） | コミュニティ内の全スペースを操作 |
| 一般参加者 | 上記以外 | ❌ 入口なし | `#settings` への直接ルートガードも未実装 |
| Hossii 運営者 | 同上（現状は管理者と同一ロール） | ✅ | Internal FF は将来分離 |

- スペース単位の **所有者・招待管理者** を表す DB 列・API・UI は **存在しない**
- 設定画面内の一部タブ（タグ・モデレーション・Feature Flags・隣人）は `SpaceSettingsScreen` 内で `isAdmin` ガード。到達者はいずれにせよコミュニティ管理者

#### 2.2.2 今回の管理画面再構成で使用する権限

| 方針 | 内容 |
|------|------|
| **アクセス制御** | **現行維持** — コミュニティ管理者（`isAdmin`）のみ |
| **変更しないもの** | `AuthContext` の判定、`SpacesScreen` からの遷移、タブ内 `isAdmin` ガード |
| **用語** | 仕様書・UI 文案では「コミュニティ管理者」または文脈上「管理者」と表記。「スペース管理者」は将来目標用語として §2.2.3 に予約 |
| **スコープ外** | スペース単位 RBAC の新設は **本再構成に含めない** |

#### 2.2.3 将来的な目標権限

```
スペース管理者（将来）
├── スペース所有者      … スペース作成者。全設定の変更可
├── 招待管理者          … 所有者が招待。設定変更可（範囲は将来定義）
└── （対比）一般参加者  … 閲覧・投稿のみ。管理画面不可
```

#### 2.2.4 将来必要になる変更（独立タスク）

| レイヤ | 変更内容 |
|--------|----------|
| **DB** | `spaces.owner_id`、`space_admins`（または同等）テーブル、RLS 更新 |
| **API** | スペース設定 CRUD の権限チェックをコミュニティ admin → スペース admin へ |
| **UI** | 管理画面入口をスペース画面/プロフィール等にも（スペース管理者向け）、招待 UI |
| **仕様書** | [10_ログインユーザー管理設計.md](./10_ログインユーザー管理設計.md) へのフェーズ追加 |

> **独立タスクとして扱う。** 管理画面 IA 再構成（Phase 2〜4）とは **並行しない**。RBAC 導入後に権限チェックを差し替える前提で、今回の画面構造は `isAdmin` 前提のまま実装する。

### 2.3 メッセージ投稿と投稿へのコメント [Q2 確定]

#### 2.3.1 メッセージ投稿 — 内部名称の改名方針

**影響範囲（調査結果）:**

| 箇所 | ファイル |
|------|----------|
| 型 | `src/core/types/settings.ts`（`commentPost`） |
| DB マッピング | `spaceSettingsApi.ts`（`feature_comment_post`） |
| UI（旧） | `GeneralTab.tsx`（ラベル「コメント投稿」→ 再構成で削除） |
| フォールバック | `postFieldSettings.ts`（`commentPost` → `message.enabled`） |
| テスト | `postFieldSettings.test.ts` |

**推奨案: 今回の再構成（Phase 3）では `commentPost` / `feature_comment_post` を維持し、Phase 5 データクリーンアップで改名**

| 理由 | 説明 |
|------|------|
| リスク低 | Phase 3 は UI/IA 変更が主。DB 列・型・API 改名を同時に行うとマイグレーションとテスト範囲が膨らむ |
| 正本移行で十分 | UI は `postFields.message` のみ編集。`commentPost` は読み取りフォールバックとして残す |
| 改名タイミング | Phase 5 で `messagePost` または `feature_message_post` へ。旧キー読み取り 1 リリース維持 |

**Phase 3 で必須の変更:**

- 管理 UI から「コメント投稿」表記を **完全排除**
- `PostScreen` 等の参加者向け UI は既に「メッセージ」表記（変更不要）

## 7. スペースモード仕様

### 7.1 基本原則

1. モードは **プリセット**。適用後も個別設定を自由に変更できる
2. 各画面は **モードを直接参照しない**。適用時に `SpaceSettings` / `Space` を更新する
3. 適用時は **確認ダイアログ** で上書き項目を明示
4. 適用後は「`{モード名}` をもとにカスタマイズされています」と表示

### 7.2 モード一覧（初期案）

| モード ID | 表示名 | 用途 |
|-----------|--------|------|
| `plaza` | 自由な広場 | オープンな交流 |
| `reflection` | 内省・振り返り | 個人の記録・振り返り |
| `workshop` | ワークショップ | セッション型 |
| `event` | イベント | 短期イベント |
| `custom` | カスタム | プリセット未適用 / 手動構成 |

### 7.3 モード別初期値 [Q5 確定: v1 採用]

既存 [docs/spaces/*.md](../../spaces/) および [20_feat_内省スペース.md](./20_feat_内省スペース.md) と整合。以下を **v1 初期値として採用** する。

| 運用方針 | 内容 |
|----------|------|
| 適用 UI | 変更内容プレビュー（diff）を必ず表示し、管理者確認後に一括適用 |
| 微調整 | 実装後フィードバックでモード定義 JSON を更新可能（設定ファイル化推奨） |
| 投稿へのコメント | 未実装のためモード定義に **含めない** |

#### 自由な広場 (`plaza`)

| 設定 | 値 |
|------|-----|
| 公開範囲 | URL を知っている人 |
| いいね | ON |
| 投稿位置 | 選択可能 |
| タグ/写真 | ON |
| 過去の投稿との出会い | OFF |
| 投稿の編集権限 | 全員 |

#### 内省・振り返り (`reflection`)

| 設定 | 値 |
|------|-----|
| 公開範囲 | 非公開 |
| いいね | OFF |
| 投稿位置 | 自動 |
| メッセージ/気持ち/タグ | ON |
| 過去の投稿との出会い | ON |
| 投稿の編集権限 | 投稿者本人と管理者のみ |

#### ワークショップ (`workshop`)

| 設定 | 値 |
|------|-----|
| メッセージ | 表示 + **必須** |
| タグ | ON |
| 写真/数値 | ON（任意） |
| いいね | ON |
| 投稿位置 | 選択可能 |
| 画像書き出し | 利用可 |

#### イベント (`event`)

| 設定 | 値 |
|------|-----|
| 気持ち/タグ | ON |
| メッセージ | 任意 |
| いいね | ON |
| 投稿位置 | 自動 |
| 公開範囲 | URL を知っている人 |
| 画像書き出し | 利用可 |

### 7.4 保存形式

```ts
// SpaceSettings に追加（案）
type SpaceModeId = 'plaza' | 'reflection' | 'workshop' | 'event' | 'custom';

type SpaceModeState = {
  appliedMode: SpaceModeId;      // 最後に適用したモード。個別変更後も保持
  isCustomized: boolean;         // 適用後に手動変更があったか
  appliedAt?: string;            // ISO8601
};
```

`isCustomized` 判定: モード適用時のスナップショットと現設定を比較（Phase 4 で実装）。

### 7.5 UI フロー

```
[モードカード一覧]
  → 選択
  → [変更内容プレビュー] モーダル
  → 「この設定を適用」
  → 一括 updateSpace + upsertSpaceSettings
  → トースト「内省・振り返り の設定を適用しました」
```

**保存方式**: 即時保存（モード適用）

## 10. 保存 UX

> 保存バーの見た目・配置は §15.6。トースト・Undo のスタイルは §15.7。

### 10.1 明示的保存（画面下部バー）

**対象**: 基本情報, 公開・共有, 投稿フォーム, 投稿・交流ルール, 背景, 投稿の見た目, 中心キャラクター

```
┌─────────────────────────────────────────────┐
│ 変更があります                               │
│              [変更を破棄]  [保存]            │
└─────────────────────────────────────────────┘
```

| 状態 | 動作 |
|------|------|
| 未変更 | バー非表示 |
| 変更あり | バー表示。保存ボタン enabled |
| 保存中 | ボタン disabled + 「保存中...」 |
| 成功 | トースト「保存しました」。バー非表示 |
| 失敗 | トースト（エラー）。編集内容保持 |

**廃止**: ヘッダーのグローバル「保存」

**保存対象**: **現在開いている画面** の変更のみ（`Space` / `SpaceSettings` の該当フィールド）

### 10.2 即時保存

**対象**: タグ, 装飾, 投稿管理, 隣のスペース, 漂着頻度, スペースモード適用

| 操作 | フィードバック |
|------|----------------|
| 成功 | トースト（例: 「タグを追加しました」） |
| 失敗 | エラートースト + ロールバック |
| 非表示 | 「投稿を非表示にしました [元に戻す]」 |

### 10.3 未保存変更とタブ移動

**優先方式**: 確認ダイアログ

```
変更が保存されていません。
このまま移動すると変更が失われます。
[キャンセル] [変更を破棄して移動]
```

- タブごとの下書き保持は Phase 5 以降の改善候補
- 自動保存は採用しない（明示的保存を原則）

### 10.4 競合 [Q6 確定]

| 項目 | 方針 |
|------|------|
| Phase 2〜4 | **Last-Write-Wins**（現行維持）。同一コミュニティ admin の同時編集は稀なため |
| 楽観的ロック | **Phase 5 以降の改善候補**。`updated_at` 比較 + 競合トースト。本再構成の必須要件ではない |
| 根拠 | 管理画面は単一 admin 操作が主。複数 admin 競合が問題化してから対応で十分 |
