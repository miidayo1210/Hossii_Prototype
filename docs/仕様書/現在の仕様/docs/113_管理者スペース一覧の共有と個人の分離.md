# 113 管理者スペース一覧の共有と個人の分離

> **ステータス:** 実装済み（`feat/admin-spaces-list-ui`）。  
> **最終更新:** 2026-07-15  
> **想定 branch:** `feat/admin-spaces-list-ui`  
> **上位仕様:** [109_コミュニティ所属個人アカウント・個人スペース・招待制スペース](../109_コミュニティ所属個人アカウント・個人スペース・招待制スペース.md)（§11 個人スペーステンプレート）  
> **関連:** [09_スペース管理画面UI仕様書](../09_スペース管理画面UI仕様書.md)（A04）、[21_コミュニティごとのスペース管理画面](../21_コミュニティごとのスペース管理画面.md)、[111_個人スペースをタブとして表示](./111_個人スペースをタブとして表示.md)

## 目的

コミュニティ管理者のスペース一覧（`#spaces` / `SpacesScreen`）で、**共有スペース（チーム向け）** と **メンバー個人スペース** を明確に分離し、運用しやすくする。

メンバー数が増えても（例: 20 アカウント → 最大 20 個人スペース）、共有スペースの作成・設定・アーカイブ運用が埋もれないようにする。

## 背景（現状の問題）

### 現行 UI

`SpacesScreen` は `fetchSpaces(communityId)` で取得した **全スペース** を、種別に関係なく **同一カードグリッド** に並べている。

| 現状 | 問題 |
|------|------|
| `space_type = shared` と `personal` が同列 | 管理者が「コミュニティのスペース」と「個人のスペース」を混同しやすい |
| 個人スペースの表示名がほぼ「個人スペース」 | **誰のスペースか** が一覧から分からない |
| `PersonalSpaceTemplateEditor` が一覧上部に常時表示 | 共有スペース管理と同じ視線の高さにあり、テンプレート設定の文脈が分かりにくい |
| 「新しいスペースを作成」が共有専用であることが弱い | 個人スペースはメンバー作成であり、管理者が一覧から増やす対象ではない |

### データ上の事実（109 整合）

- **共有スペース**（`spaces.space_type = 'shared'`）: コミュニティのイベント・チーム活動用。管理者が作成・運用する。
- **個人スペース**（`spaces.space_type = 'personal'`）: メンバーごとに 1 つ（初回入室時などに作成）。**owner 本人とコミュニティ管理者** が閲覧可能（RLS）。
- 管理者一覧に個人スペースが出るのは **仕様どおり**。ただし **UI 上の扱い** を分離する。

## 用語（UI 表示）

| 内部 | UI セクション名 | 説明 |
|------|------------------|------|
| `shared` | **共有スペース** | ユーザー表現の「チームスペース」に相当。コミュニティ全体で共有する活動用スペース |
| `personal` | **個人スペース** | メンバー本人専用。管理者は閲覧・設定・（自分名義の）投稿が可能。owner 本人の投稿の編集・削除は不可（109 §10.3） |

本書では DB 型は `shared` / `personal`、画面ラベルは **共有スペース** / **個人スペース** を正とする。

## 画面構成（確定案）

`SpacesScreen`（`#spaces`）を **上下 2 セクション** に分ける。

```text
┌─────────────────────────────────────────────┐
│ ヘッダー（既存）: コミュニティ名のスペース管理 │
│  [+ 新しい共有スペースを作成]  ← 文言を明確化    │
├─────────────────────────────────────────────┤
│ ■ 共有スペース                               │
│   説明: チーム・イベントなどコミュニティ全体のスペース │
│   [カードグリッド] Dev Public Space …         │
│   （既存: 名前/slug/背景/招待URL/設定/アーカイブバッジ）│
├─────────────────────────────────────────────┤
│ ▼ 個人スペース（3件）  ← 折りたたみヘッダー      │
│   ┌─ 展開時 ─────────────────────────────┐  │
│   │ [個人スペーステンプレート]（既存 UI）    │  │
│   │ ・テンプレートを有効にする              │  │
│   │ ・名前の初期値 / 背景パターン           │  │
│   ├──────────────────────────────────────┤  │
│   │ [メンバー別リスト]                      │  │
│   │  ほっしー太郎（dev-user-a@…） [設定][開く]│  │
│   │  Community Admin（自分）      [設定][開く]│  │
│   └──────────────────────────────────────┘  │
└─────────────────────────────────────────────┘
```

### 1. 共有スペースセクション（メイン）

- **常に展開**（折りたたみなし）。
- 対象: `spaceType !== 'personal'`（= `shared`）。
- 既存カード UI を **そのまま移す**（名前編集、slug、背景、招待 URL、⚙️ 設定、アーカイブバッジ）。
- 空状態: 「まだ共有スペースがありません」＋「新しい共有スペースを作成」案内。
- ヘッダーの作成ボタンは **共有スペース専用** であることをラベルで明示する（例: `+ 新しい共有スペースを作成`）。

### 2. 個人スペースセクション（サブ・折りたたみ）

- **デフォルト: 折りたたみ（閉じた状態）**。
- ヘッダー行に件数バッジ（例: `個人スペース（3件）`）。クリックで展開／折りたたみ。
- 展開時の **先頭ブロック** に `PersonalSpaceTemplateEditor` を配置する（現状の一覧上部から **移動**）。
  - 文言は現行どおり:
    - 「メンバーがこのコミュニティ内で個人スペースを**新規作成するとき**の初期設定です」
    - 「変更しても、すでに作成済みの個人スペースには反映されません」
    - 「テンプレートを有効にする」トグル
- その下に **メンバー個人スペースのリスト**（カードグリッドではなく **コンパクトな行リスト** を推奨）。
- 各行に最低限表示する情報:

| 表示項目 | 優先順位（フォールバック） |
|----------|---------------------------|
| 所有者ラベル | ① コミュニティ表示名（`community_memberships.community_nickname`）→ ② `user_profiles.default_nickname` → ③ メールのローカル部 → ④ `owner_user_id` 短縮 |
| 補足 | メールアドレス（管理者のみ閲覧可・小さく） |
| スペース名 | `spaces.name`（多くは「個人スペース」） |
| 状態 | アーカイブバッジ（112 連携・既存 `SpaceArchiveBadge`） |

- 行アクション:
  - **設定**（⚙️）: 既存 `handleOpenSettings` → `#settings`（個人スペースでも管理者が開ける範囲は 109 / 92 に従う）
  - **開く**: `/c/{slug}/s/{space_url}#screen`（既存導線）
- 個人スペース行では **名前・slug のインライン編集はしない**（owner の領域。管理者は設定画面経由）。
- 空状態（0 件）: 「まだ個人スペースはありません（メンバーが作成するとここに表示されます）」。テンプレート編集は **表示する**（作成前でも設定可能）。

### 3. スーパー管理者（override 時）

[21](./21_コミュニティごとのスペース管理画面.md) / [67](../67_コミュニティスペース一覧%20.md) と同様、選択コミュニティ単位で上記 2 セクションを表示する。挙動はコミュニティ管理者と同型。

## データ取得（実装方針）

### クライアント分割（初版）

既存 `fetchSpaces(communityId)` の結果をクライアントで分割する。

```ts
const sharedSpaces = spaces.filter((s) => s.spaceType !== 'personal');
const personalSpaces = spaces.filter((s) => s.spaceType === 'personal');
```

DB / RPC 変更は **初版では不要**。

### 所有者表示用のルックアップ（初版で追加）

`personal` 行のラベル用に、管理者が読める範囲で次を結合する。

| ソース | 列 |
|--------|-----|
| `spaces` | `owner_user_id`（既に `Space.ownerUserId`） |
| `community_memberships` | `community_nickname`（同一 `community_id` + `auth_user_id`） |
| `user_profiles` | `default_nickname`（任意） |

**PII 方針:** 一覧は管理者画面のためメール表示可。ただし **ログ・エラーに UID を垂れ流さない**。将来、表示名のみに絞る設定は検討可。

### 将来（件数が多い場合）

- 個人スペースリストに **検索・ページング**（20 件超を想定）
- サーバー側 `list_community_personal_spaces` RPC（owner 表示名付き）— **本仕様の必須ではない**

## 個人スペーステンプレートの位置づけ（109 §11 との関係）

| 項目 | 本仕様での扱い |
|------|----------------|
| 保存先 | 変更なし（`communities.personal_space_template` 等・109 案 A） |
| 適用タイミング | 変更なし（**新規作成時のみ**コピー） |
| UI 配置 | **個人スペースセクション展開時のみ**表示（一覧上部から移動） |
| 権限 | 変更なし（コミュニティ管理者・`super_admin`） |

`CommunityHomeScreen` の管理メニュー内テンプレート編集がある場合、**二重入口は許容**する（HOME = 運用導線、一覧 = スペース文脈）。文言・設定内容は同一 API を正本とする。

## やらないこと（初版）

- 個人スペースの **新規作成ボタン** を管理者一覧に追加（作成はメンバー操作 / 既存 `ensureMyPersonalSpace` 導線のまま）
- 個人スペースの **削除** を一覧から実行
- 共有と個人の **並び順統合**（共有を常に上、個人は別セクション）
- 個人スペースの **一括アーカイブ**
- DB migration / RLS 変更
- `pair` / `team` 型の追加（109 将来）

## 受入条件

| # | 条件 |
|---|------|
| 1 | 共有スペースのみがメインのカードグリッドに表示される |
| 2 | 個人スペースは別セクションにまとまり、**デフォルト折りたたみ** |
| 3 | 個人スペースセクションを展開すると **テンプレート編集** が表示される |
| 4 | 個人スペース各行に **所有者が識別できるラベル** がある |
| 5 | 個人スペースからも **設定・開く** ができる |
| 6 | アーカイブバッジ（112）が共有・個人の両方で正しく出る |
| 7 | 「新しいスペースを作成」が **共有専用** であることが UI 上明確 |
| 8 | スーパー管理者のコミュニティ override 時も同じ 2 セクション構成 |
