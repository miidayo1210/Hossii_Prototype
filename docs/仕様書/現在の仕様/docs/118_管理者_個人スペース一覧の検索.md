# 118 管理者：個人スペース一覧の検索

> **ステータス:** 調査・仕様ドラフト（**未実装**）  
> **最終更新:** 2026-07-16  
> **想定 branch:** `feat/admin-personal-space-search`（仮）  
> **上位仕様:** [109_コミュニティ所属個人アカウント・個人スペース・招待制スペース](../109_コミュニティ所属個人アカウント・個人スペース・招待制スペース.md) / 管理者スペース一覧の共有・個人分離（PR #15 実装済み・main `e8a7da8`）  
> **関連画面:** `#spaces` → `SpacesScreen` 個人スペース折りたたみセクション

## 背景

管理者の `#spaces` には、共有スペース一覧の下に **個人スペース** 折りたたみセクションがある（PR #15 / main 反映済み）。

現状は全件を `created_at` 昇順で縦リスト表示するのみで、コミュニティ規模が大きくなると目的の所有者を探しにくい。

## 現状調査（main `e8a7da8`）

### データ取得

| 層 | 実装 | 備考 |
|----|------|------|
| スペース本体 | `fetchSpaces(communityId)` → `state.spaces` | `select('*')` / `created_at ASC` |
| 分区 | `partitionAdminCommunitySpaces` | `communityId` 一致 + `spaceType === 'personal'` |
| 所有者表示 | `fetchPersonalSpaceOwnerLabels` | 非同期で別取得 |

### 一覧に使えるフィールド（クライアント既所持）

| 項目 | ソース | 検索候補 |
|------|--------|----------|
| 所有者表示名 | `resolvePersonalSpaceOwnerDisplay` | ○（主対象） |
| 補助 email | `supplementaryEmail`（accepted invitation） | ○ |
| スペース名 | `space.name` | ○ |
| スペース slug | `space.spaceURL` / `space.id` | △（管理者向け補助） |
| owner user id | `space.ownerUserId` | ×（UI に出さない） |
| アーカイブ | `space.isArchived` | フィルタ用（検索文字列ではない） |

**email の網羅性:** `auth.users.email` は直接取得していない。表示 email は **accepted な community invitation の invitee_email** のみ。未招待・別経路参加ユーザーは email 行が空のことがある。

### 件数とクライアント検索の妥当性

| 指標 | 値 |
|------|-----|
| DB 制約 | 1 ユーザー × 1 コミュニティ = 個人スペース最大 1 件 |
| `fetchSpaces` 上限 | PostgREST 既定 ~1000 行（community 内の shared + personal 合算） |
| 実運用想定 | 数十〜数百メンバー規模なら **クライアント側全文検索で十分** |
| API 追加が必要な目安 | 単一 community で personal が **500 件超** かつ初回表示性能が問題化したとき |

**v1 結論:** 追加 API・migration **不要**。既存 store + ownerLabels のクライアント filter で足りる。

### community 混在防止

- `partitionAdminCommunitySpaces` が `space.communityId === communityId` で厳密に分区
- super_admin は `overrideCommunityId`（`#spaces/{slug}`）で community を固定してから `fetchSpaces`
- 他 community の personal はリストに出ない（テスト済み）

### archived personal space

- **除外しない**（現行どおり一覧に含める）
- `space.isArchived === true` のとき `SpaceArchiveBadge` を表示
- 検索・フィルタで「アーカイブのみ」切替は v1 対象外（将来）

### mobile UI

- 個人セクションは縦リスト（`@768px` で行が column 積み）
- 検索欄は折りたたみヘッダー直下、全幅 input が自然
- PC 前提レイアウトだが、検索自体はモバイルでも単一 input で問題なし

## v1 仕様（推奨）

### 検索欄の表示場所

- **個人スペース折りたたみセクション内**、ヘッダー（件数バッジ + 展開トグル）の **直下**
- セクションが **展開されているときだけ** 表示（折りたたみ時は非表示でよい）

### 初期状態

- 検索文字列: **空**
- フィルタ: 全件表示（アーカイブ含む）
- セクション自体: **折りたたみ**（現行どおり）

### 検索対象（v1）

次の文字列を **部分一致** で検索する（いずれかにマッチすれば表示）:

1. 所有者表示名（`displayName`、「さん」含む）
2. 補助 email（`supplementaryEmail`）
3. スペース名（`space.name`）

**対象外（v1）:** slug / user id / community 名

### 大文字小文字・空白の扱い

- **大文字小文字:** 無視（`toLowerCase()` 相当）
- **前後空白:** trim してから検索
- **連続空白:** 単一スペースとして扱わなくてよい（部分文字列なのでそのまま）
- **全角半角:** v1 では正規化しない（必要なら v2）

### 0 件時表示

- 検索文字列が空で 0 件: 現行どおり（「個人スペースがまだありません」相当。現状は空リストのみなら文言追加を検討）
- 検索後 0 件: `「{query}」に一致する個人スペースはありません`（やさしい短文）
- 折りたたみヘッダーの件数バッジは **フィルタ後件数** を表示（`3件` → 検索で 1 件なら `1件`）

### 折りたたみとの関係

- 折りたたみ **閉** → 検索欄非表示、検索 state は **保持**（再展開で復元）
- 折りたたみ **開** → 検索欄表示、入力即時フィルタ（debounce 不要・件数少）

### archived 表示

- 検索で **除外しない**（現行どおりアーカイブもヒット対象）
- アーカイブバッジはフィルタ後の行にも表示

### 件数表示

- ヘッダーバッジ: **表示中件数 / 全件** は v1 では **表示中件数のみ**（`12件`）
- 検索中で件数が減ったときだけ変化が分かれば十分

### mobile

- input は `type="search"` 推奨（iOS でクリアボタン）
- プレースホルダ: `名前・メール・スペース名でさがす`
- キーボード表示時もリストはその下にスクロール

## 非対象（v1）

- サーバー側検索 RPC
- 共有スペース一覧への検索（別タスク）
- アーカイブのみ / 未アーカイブのみフィルタ
- ソート変更（→ [120](./120_管理者_スペース一覧の並び替え.md)）
- 一括操作（→ [119](./119_管理者_個人スペースの一括操作.md)）
